Key Management

Two techniques that harden keys: stretching slows down brute force on weak keys, and splitting means no single party (or breach) holds the whole key.

overview

Key management is the lifecycle of generating, distributing, storing, and handling cryptographic keys. Two techniques: key stretching strengthens weak/short keys (like passwords) with thousands of hashing iterations to make brute force expensive, and key splitting divides a key into parts stored separately so no single point of compromise exposes it.

technique 01

Key Stretching

Iterative hashing/encryption of a weak key (PBKDF2, Bcrypt) plus salting, making each brute-force guess slow and rainbow tables useless.

technique 02

Key Splitting

Divide a key into parts stored separately; only combining all parts reconstructs it. No single breach exposes the key.

key stretching

Key stretching repeatedly runs a key through thousands of rounds of hashing/encryption, turning a simple key into a longer, more complex one (often extended to at least 128 bits). It doesn't make the key inherently stronger, but it makes every brute-force attempt go through the same expensive process, dramatically slowing an attacker. Two common methods:

PBKDF2
Combines the password with a salt and runs HMAC many times. Originally ~1,000 iterations; modern guidance recommends 310,000+ with HMAC-SHA-256.
Bcrypt
Adaptive hashing + salting that can raise its iteration count over time as hardware improves. Stays resistant to brute force as computers get faster; used in OpenBSD and Linux distros to protect stored passwords.

Salting adds random data to each password before hashing, so identical passwords produce different hashes. This defeats rainbow table attacks (precomputed hash-to-plaintext lookups) and prevents a cracked password on one system from being reused elsewhere.

key splitting

Key splitting divides a single cryptographic key into separate pieces, each stored independently; only by combining all parts can the key be reconstructed and used. Even if an attacker grabs one part, they can't decrypt anything, eliminating a single point of failure. It's used where multiple parties must cooperate to access sensitive data (finance, government, sensitive corporate info). Example: a banking system requires three parts, one on the bank's servers, one in an HSM, and a one-time code sent to the user, all three must combine to grant access.

EXAM TIPS

what to know for the exam

  • Key stretching makes brute-forcing weak keys slow via thousands of iterations. PBKDF2 and Bcrypt are the named methods.
  • Salting adds random data per password so identical passwords hash differently, defeating rainbow tables.
  • Bcrypt is adaptive: it can increase iterations over time as hardware improves.
  • Key splitting divides a key into separately-stored parts; all parts are needed to reconstruct it, no single point of compromise.
  • Rainbow table = precomputed hash-to-plaintext lookups; salting is the countermeasure.
decision rules
  • Scenario slows password cracking with many hashing iterations → key stretching (PBKDF2/Bcrypt)
  • Scenario adds random data so identical passwords differ → salting
  • Scenario uses precomputed hash tables to crack passwords → rainbow table attack
  • Scenario requires multiple stored parts to rebuild one key → key splitting

Security Properties

Two properties that strengthen sessions: mutual authentication makes both sides prove who they are, and forward secrecy keeps past sessions safe even if a key leaks later.

overview

Two cryptographic security properties: mutual authentication, where both parties verify each other's identity before exchanging data (not just the client trusting the server), and forward secrecy (perfect forward secrecy), where unique per-session keys ensure that compromising one key can't decrypt past or future sessions.

property 01

Mutual Authentication

Both client and server prove their identity (via certificates/keys) before data flows. Two-way trust that blocks impostors and on-path attacks.

property 02

Forward Secrecy

Unique session keys per session, never reused. A compromised key exposes only that one session, not past or future ones.

mutual authentication

Mutual authentication has both parties verify each other before any data is exchanged: the client confirms the server, and the server confirms the client. It's like showing your ID to a security guard who must also show you their credentials, only when both are satisfied does interaction proceed. In practice it uses digital certificates or keys exchanged before sensitive data flows. Common in online banking and VPNs, it establishes two-way trust and blocks unauthorized access and on-path attacks.

forward secrecy

Forward secrecy (perfect forward secrecy) generates a unique encryption key for each session and never reuses it. So traffic captured today can't be decrypted later, and a key compromised today doesn't expose past or future sessions.

Older SSL/TLS used the server's long-term RSA key to protect sessions: the client encrypted a session key with the server's public key. The flaw, if that private key were ever stolen, an attacker could decrypt all previously captured traffic. The 2014 Heartbleed bug (an OpenSSL TLS heartbeat flaw leaking server memory, including private keys) showed exactly why this matters; forward secrecy would have prevented decryption of past sessions.

Forward secrecy avoids this by using ephemeral key agreement like Diffie-Hellman, which never exposes a long-term private key. It's standard in modern VPNs, TLS 1.3, and SSH, and is built into WPA3 via Simultaneous Authentication of Equals (SAE).

EXAM TIPS

what to know for the exam

  • Mutual authentication = BOTH parties verify each other (two-way), not just the client checking the server.
  • Forward secrecy (perfect forward secrecy) = unique per-session keys; a leaked key exposes only that session.
  • Heartbleed (2014) is the example: leaked private keys could decrypt past traffic, forward secrecy would have prevented it.
  • Forward secrecy relies on ephemeral Diffie-Hellman and is in TLS 1.3, VPNs, SSH, and WPA3 (SAE).
decision rules
  • Scenario has both client and server prove identity before data flows → mutual authentication
  • Scenario keeps past sessions safe even after a key is compromised → forward secrecy
  • Scenario references leaked private keys decrypting old captured traffic → lack of forward secrecy (Heartbleed)
  • Scenario uses unique per-session ephemeral keys → perfect forward secrecy

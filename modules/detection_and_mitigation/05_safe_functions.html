Safe Functions

Writing code that behaves reliably under pressure: operations that complete all-or-nothing, that respect memory boundaries, and that survive multiple threads at once.

overview

Safe functions operate securely and reliably. Three concepts: atomic functions (complete in a single indivisible step, all-or-nothing), memory-safe functions (prevent buffer overflows/leaks by respecting memory boundaries), and thread-safe functions (operate correctly under concurrent threads without race conditions).

concept 01

Atomic Functions

All-or-nothing: an operation fully completes or doesn't run at all. Prevents partial states (no half-finished bank transfer). Uses atomic ops / mutex locks.

concept 02

Memory-safe Functions

Respect memory boundaries, free unused memory: prevent buffer overflows, leaks, segfaults. Rust/Java (auto), smart pointers in C/C++.

concept 03

Thread-safe Functions

Operate correctly with concurrent threads, no race conditions or data corruption (shared shopping cart). Uses locks/mutexes.

atomic functions

Atomic functions complete an operation as a single indivisible action, all-or-nothing, with no in-between state, critical in concurrent environments to prevent partial completion leaving data inconsistent and to avoid race conditions. Example: a bank transfer that either moves the full amount or nothing, never debiting one account without crediting the other. Implemented with built-in atomic operations, atomic variables, or mutex locks (only one thread accesses a resource at a time).

memory-safe functions

Memory-safe functions manage memory to avoid common vulnerabilities, buffer overflows, memory leaks, segmentation faults, by ensuring allocated memory doesn't exceed what's available, respecting memory boundaries, and automatically freeing unused memory. Example: an app handling large files that keeps each chunk within available memory. Use memory-safe languages like Rust or Java (garbage collection, compile-time checks); for C/C++, use libraries and smart pointers (objects that automatically manage allocation/deallocation to prevent leaks).

thread-safe functions

Thread-safe functions operate reliably when multiple threads access shared data simultaneously, preventing race conditions and data corruption. Unlike atomic functions (single indivisible step) or memory-safe functions (memory protection), thread-safe functions specifically manage execution concurrency. Example: an online shopping cart where multiple users add items at once, thread-safety prevents one user's changes overwriting another's, keeping cart totals accurate. Achieved with locking mechanisms (mutexes), atomic functions, or thread-safe data structures.

EXAM TIPS

what to know for the exam

  • Atomic = all-or-nothing single step (bank transfer); prevents partial states and race conditions.
  • Memory-safe = respects memory boundaries and frees memory; prevents overflows/leaks (Rust, Java, smart pointers).
  • Thread-safe = correct under concurrent threads (shared cart); prevents race conditions/data corruption.
  • Distinguish: atomic (indivisibility), memory-safe (memory protection), thread-safe (concurrency).
  • Mutex locks support both atomicity and thread-safety (one thread at a time).
decision rules
  • Scenario needs an operation to fully complete or not at all → atomic function
  • Scenario prevents buffer overflows/leaks via boundary checks → memory-safe function
  • Scenario keeps concurrent threads from corrupting shared data → thread-safe function
  • Scenario uses smart pointers in C++ → memory-safe function

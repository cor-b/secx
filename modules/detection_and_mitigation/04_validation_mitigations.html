Validation Mitigations

Two checkpoints on data: screen what comes in (input validation), and neutralize what goes out (output encoding), so malicious content never executes.

overview

Validation mitigations ensure data entering or leaving a system is properly validated and encoded. Two concepts: input validation (checking and sanitizing incoming data to meet expected formats, the first line of defense) and output encoding (converting data into a safe format before display/processing so it can't be interpreted as executable code, the second line).

concept 01

Input Validation

Screen incoming data (like airport security) against expected formats. Blocks SQLi, XSS, overflows. Do both client- AND server-side (server is authoritative).

concept 02

Output Encoding

Convert special characters to safe text before display (backend). Catches what input validation missed; key XSS defense.

input validation

Input validation verifies and cleans user-provided or external data before processing, ensuring it matches expected patterns, formats, and types, like airport security screening passengers before boarding. Without it, attackers can inject harmful content causing SQL injection, XSS, and buffer overflows. Key practices:

  • Do both client-side (better UX, catches errors early) and server-side validation, server-side is authoritative because client-side checks can be bypassed.
  • Prefer allowlisting acceptable inputs over denylisting forbidden ones (more secure).
  • Use validation libraries, regular expressions, data-type constraints, length checks, and OWASP validation resources.

output encoding

Output encoding is applied to data before it's sent from a server or displayed, ensuring it can't be interpreted as executable code. Applied on the backend, it's an additional layer assuming input validation may have missed something. It converts dangerous characters (like < and >) into harmless text representations, so they render as plain text, not executable code. The classic example is a comment section: without encoding, a user could post a script that steals cookies from other viewers; encoding renders the script as visible text. Encode data as close to the point of display/use as possible, and use security libraries' built-in encoding functions. Output encoding is especially crucial for preventing XSS, and it complements (doesn't replace) input validation.

EXAM TIPS

what to know for the exam

  • Input validation = screen incoming data against expected formats; the first line of defense.
  • Output encoding = convert special characters to safe text before display; the second line (catches what input validation missed).
  • Do input validation both client- and server-side; server-side is authoritative (client checks can be bypassed).
  • Allowlisting inputs is more secure than denylisting.
  • Both together defend against XSS; output encoding is the key XSS rendering defense.
decision rules
  • Scenario rejects malformed/unexpected input before processing → input validation
  • Scenario renders user content as text instead of running it → output encoding
  • Scenario trusts only server-side checks → server-side input validation
  • Scenario allows only known-good input patterns → allowlisting

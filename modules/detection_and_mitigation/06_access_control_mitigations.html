Access Control Mitigations

Shrinking the attack surface three ways: give users only the permissions they need, leave only the features they need running, and allow only what's explicitly approved.

overview

Access control mitigations restrict access to system resources and functionality. Three concepts: least privilege (minimum permissions for a role), least functionality (only essential features/functions enabled), and allowlisting (only explicitly approved apps/IPs permitted, block everything else).

concept 01

Least Privilege

Minimum permissions for a user/process role (limits user permissions). Payroll sees salaries; customer service doesn't. Contains breaches.

concept 02

Least Functionality

Only essential features/services enabled (limits system features). Disable unused APIs, admin tools, default configs. Shrinks attack surface.

concept 03

Allowlisting

Only explicitly approved apps/IPs allowed; block all else. More secure than blocklisting (which only blocks known-bad).

least privilege

Least privilege limits a user's or process's permissions to only what's necessary for their role, so a compromised account can't exceed its limited scope. Example: payroll can view/update salaries, customer service cannot, so a compromised customer-service account can't reach payroll data. Enforce with consistent access reviews, clear role definitions, prompt permission adjustments when roles change (including offboarding), and IAM/PAM tools that flag excessive permissions.

least functionality

Least functionality restricts available system features and functions to only those essential for operations, reducing the attack surface by removing unused features that could harbor vulnerabilities. Unlike least privilege (which limits user permissions), least functionality limits system features. Example: restrict an admin panel to admin accounts while standard users get only the standard interface. Apply by evaluating each feature, disabling/removing what's extraneous, unused APIs, unnecessary admin tools, debugging interfaces, default configs, inactive network services.

allowlisting

Allowlisting (formerly whitelisting) limits access to a specific list of approved applications, IP addresses, or entities, blocking anything not explicitly permitted, so only trusted, known resources interact with the system. It's more secure than blocklisting (formerly blacklisting), which only blocks known-bad items and is only as good as that known-bad list. Example: a corporate network allowing only trusted IPs/VPNs to reach sensitive resources, blocking unauthorized IPs automatically. Some orgs allowlist applications by hash value to permit only specific trusted versions. Apply at firewalls and application servers; manage with tools like Active Directory Group Policy, AppLocker, and Defender Application Control.

EXAM TIPS

what to know for the exam

  • Least privilege limits user/process PERMISSIONS; least functionality limits system FEATURES. Don't confuse them.
  • Allowlisting permits only explicitly approved entities (block by default); more secure than blocklisting (which only blocks known-bad).
  • Least privilege contains breaches; least functionality shrinks the attack surface.
  • Allowlisting can use application hash values to permit only trusted versions.
  • Tools: IAM/PAM (least privilege), AppLocker/Defender Application Control (allowlisting).
decision rules
  • Scenario gives a user only the permissions their job needs → least privilege
  • Scenario disables unused services/features → least functionality
  • Scenario blocks everything except an approved list → allowlisting
  • Scenario only blocks known-bad apps/IPs → blocklisting (less secure)

Design Mitigations

Building security in from the start: proven secure-architecture patterns, multiple redundant layers of defense, and disciplined management of the third-party code you depend on.

overview

Design mitigations build security into the system design phase to prevent or reduce vulnerabilities. Three concepts: security design patterns (best practices like input validation, least privilege, secure defaults), defense-in-depth (layered, redundant protections), and dependency management (selecting, monitoring, updating third-party components).

concept 01

Security Design Patterns

Best practices baked into architecture: input validation, least privilege, secure defaults, error handling/logging. Proactive, not reactive.

concept 02

Defense-in-Depth

Layered, redundant protections around critical assets (not just the perimeter). If one layer fails, others hold.

concept 03

Dependency Management

Select, monitor, and update third-party libraries to avoid inherited vulnerabilities. Like car maintenance for code.

security design patterns

Security design patterns are established practices that guide secure architecture, building security directly into application structure, including input validation, least privilege, secure defaults, and error handling/logging. Applied early and throughout development, they address vulnerabilities proactively rather than reactively. Example: input validation blocks SQL injection by filtering harmful input; secure defaults configure apps to the safest settings to avoid misconfiguration. They're most effective when consistently applied across the whole system.

defense-in-depth

Defense-in-depth builds multiple redundant layers of protection around critical assets, not just the network perimeter, so if an attacker breaches one layer, others still protect the asset. Each layer addresses different attack vectors (encryption, access controls, intrusion detection, network segmentation). Example: a banking database protected by firewalls, encryption, strict access controls, and continuous monitoring, even past the firewall, encrypted data stays unreadable and access controls limit manipulation. Implement by combining different security technologies, each tailored to specific threats, and regularly testing them.

dependency management

Dependency management monitors and maintains third-party libraries/components so they don't introduce security flaws, outdated or vulnerable dependencies become risks. Like a mechanic checking for worn parts, developers monitor dependencies to avoid vulnerabilities. Be proactive: regularly audit dependencies, check for updates/patches, assess each library's risk, and use automated vulnerability scanning integrated into the development pipeline to catch issues early.

EXAM TIPS

what to know for the exam

  • Security design patterns = best practices (input validation, least privilege, secure defaults) baked into architecture; proactive.
  • Defense-in-depth = layered, redundant protections around critical assets (not just the perimeter).
  • Dependency management = vet, monitor, and update third-party libraries; use automated scanning in the pipeline.
  • All three are design-phase mitigations, security built in, not bolted on.
decision rules
  • Scenario applies input validation/secure defaults in architecture → security design patterns
  • Scenario layers firewalls + encryption + access controls + monitoring → defense-in-depth
  • Scenario audits and patches third-party libraries → dependency management
  • Scenario protects critical assets, not just the perimeter → defense-in-depth

Security in Systems

Given a scenario, implement security at every stage of a system's lifecycle — from initial hardware and requirements through testing, CI/CD pipelines, and end-of-life planning.

overview

OBJ 2.2 treats security as a property that must be designed in from the start, not added after deployment. Nine topic areas span the full system lifecycle: hardware and software assurance, supply chain risk, security requirements, pre- and post-deployment testing, CI/CD management and testing practices, and end-of-life considerations.

nine topic areas

  1. Hardware Assurance Certification verifies hardware meets security standards before deployment. Validation confirms it continues to meet those standards as updates and changes occur.
  2. Security Requirements Functional requirements define specific security behaviors a system must have (authentication, encryption). Non-functional requirements define the qualities it must exhibit (performance, reliability). Security and usability often require trade-offs.
  3. Software Assurance A Software Bill of Materials (SBOM) inventories all components. Software Composition Analysis (SCA) scans them for vulnerabilities. Formal methods use mathematical proofs to verify correctness.
  4. Supply Chain Assurance Hardware supply chain management addresses counterfeit and tampered components. Software supply chain management addresses vulnerable or non-compliant third-party libraries.
  5. Pre-deployment Testing SAST analyzes code without executing it. DAST tests the running application with simulated attacks. IAST combines both by integrating an agent into a running application for real-time feedback.
  6. Post-deployment Testing Vulnerability analysis scans a live application for known weaknesses. RASP (Runtime Application Self-Protection) monitors behavior inside the running application and blocks threats in real time.
  7. CI/CD Management Coding standards and linting enforce consistency. Branch protection requires reviews and tests before merging. Continuous improvement refines the pipeline based on feedback and metrics.
  8. CI/CD Testing Canary (small rollout), regression (no regressions), integration (component interaction), automated (pipeline-driven), and unit testing (individual function correctness) together validate code before production.
  9. End-of-Life Considerations Lifecycle management ensures unsupported components are upgraded or replaced before becoming liabilities. Data migration and secure deletion protect sensitive data during decommission.

hardware assurance

Hardware assurance ensures physical components are secure, reliable, and free from malicious alterations or defects before and during their operational life.

Certification
A formal evaluation that verifies hardware components meet specific security standards and operate as expected. Certification is conducted before deployment. It provides documented evidence that a component has been tested against a defined security baseline. Government agencies, for instance, may require hardware used in critical systems to pass a rigorous certification process before it is approved for use.
Validation
Ongoing testing and verification that hardware continues to meet its design standards, especially when updates or changes are made. Unlike certification, which is a one-time evaluation, validation is continuous. Any change to hardware — a firmware update, a component replacement — triggers a new validation cycle to confirm the security posture has not degraded.

security requirements

Security requirements define what a system must do and how it must behave to protect against threats. They are established before development begins and drive design decisions throughout the lifecycle.

Functional Security Requirements
Specific security-related behaviors and features the system must implement. Examples: the system must authenticate users with MFA, enforce role-based access controls, and encrypt data in transit using TLS. These are concrete, verifiable behaviors that can be tested directly.
Non-functional Security Requirements
The qualities and characteristics the system must exhibit while performing its security functions. Examples: the authentication system must respond within 500ms (performance), remain available 99.9% of the time (reliability), and not require more than three steps to complete login (usability). These define how well the system does something, not what it does.

A persistent challenge is the security vs usability trade-off: increasing security controls (requiring MFA, enforcing password complexity) reduces convenience and accessibility. Requirements must balance both considerations rather than maximizing one at the expense of the other.

software assurance

Software assurance ensures that software is developed and maintained in a way that consistently meets security and reliability standards throughout its lifecycle.

SBOM (Software Bill of Materials)
A comprehensive inventory of all components, libraries, and dependencies used in a software application. An SBOM provides transparency into what a piece of software is actually made of. When a new vulnerability is disclosed in a library, an organization with an up-to-date SBOM can immediately determine which of its applications are affected, rather than discovering exposure only after an incident.
SCA (Software Composition Analysis)
Automated scanning and analysis of software components and dependencies to identify known vulnerabilities or licensing issues. SCA tools check the components listed in an SBOM against known vulnerability databases (such as CVE). They identify which versions have disclosed weaknesses and flag licensing conflicts that could create legal exposure. SCA is often integrated into the CI/CD pipeline to run automatically with each build.
Formal Methods
Mathematically based techniques used to verify the correctness, safety, and security of software algorithms and systems. Formal methods use mathematical proofs rather than testing to establish that software behaves exactly as specified — including in edge cases that testing might miss. They are used in high-assurance environments such as cryptographic systems, aerospace software, and safety-critical infrastructure.

supply chain assurance

Supply chain assurance ensures that every component and process in the supply chain meets security and quality standards, from initial sourcing through delivery and integration. Vulnerabilities can be introduced at any point in the chain.

Hardware Supply Chain Risk Management
Evaluating and securing physical components against tampering, counterfeit parts, or vulnerabilities introduced during manufacturing and distribution. An attacker who compromises a hardware component during manufacturing can introduce a backdoor that persists regardless of software-level security. Risk management involves vetting suppliers, requiring provenance documentation, and physically inspecting critical components. A financial institution verifying its trading platform hardware checks each component for authenticity and evidence of tampering before installation.
Software Supply Chain Risk Management
Assessing and managing risks from third-party software components, including vulnerabilities and compliance issues that could propagate into the final system. Most modern software includes dozens or hundreds of open-source libraries. A single vulnerable dependency can expose the entire application. Software supply chain risk management involves tracking all dependencies (via SBOM), scanning them with SCA, evaluating vendor security practices, and monitoring for new disclosures in components already in use.

pre-deployment testing

Pre-deployment testing identifies vulnerabilities before software reaches the live environment. Three approaches target different phases of the development process.

TypeHow it worksWhat it finds
SAST Analyzes source code or binaries without executing the application (static) Coding flaws, insecure patterns, security weaknesses detectable in code structure
DAST Tests the running application by simulating attacks from outside (dynamic) Runtime vulnerabilities: injection flaws, authentication weaknesses, exposed API endpoints
IAST Integrates an agent into the running application to monitor behavior from inside during functional tests Real-time feedback on vulnerabilities as they are triggered; more comprehensive than SAST or DAST alone

post-deployment testing

Post-deployment testing continues security evaluation after the application is live. The environment is real, the users are real, and new vulnerabilities can emerge as the application interacts with production systems.

Vulnerability Analysis
Scanning the deployed application for known security weaknesses that could be exploited. Post-deployment vulnerability scanning checks the live system against current vulnerability databases. It catches weaknesses introduced by configuration changes, new integrations, or newly disclosed CVEs against components already in production.
RASP (Runtime Application Self-Protection)
Security technology embedded within the running application that monitors behavior and blocks threats in real time. RASP operates inside the application process itself, giving it visibility into what the application is actually doing at runtime. When it detects anomalous behavior — such as an attempted SQL injection — it can block the malicious request automatically, without waiting for a human response. Example: an online banking RASP detects an SQL injection attempt and blocks the request before it reaches the database, even if no WAF was configured to catch that specific pattern.

CI/CD management

CI/CD (Continuous Integration / Continuous Deployment) management automates code integration, testing, and deployment. Security controls must be built into the pipeline itself so they run consistently with every change.

  • Coding standards and linting: predefined guidelines for how code should be written; linting tools automatically scan code for style violations, potential bugs, and security anti-patterns before it is committed
  • Branch protection: rules that prevent code from being merged into important branches (e.g., main) without passing automated tests and receiving required code reviews — ensuring no unreviewed or untested code reaches production
  • Continuous improvement: regularly updating the CI/CD pipeline itself based on performance metrics, emerging threats, and new tooling — the pipeline is a living system, not a fixed configuration

CI/CD testing

CI/CD testing systematically evaluates code changes at multiple levels to catch security and functional issues before they reach production.

Test typeWhat it does
Canary Deploys changes to a small subset of users first; monitors for issues before rolling out to everyone
Regression Verifies that new code changes do not break existing functionality that was previously working
Integration Tests interactions between different system components to catch issues that arise from combined functionality
Automated Runs tests automatically on every code change in the pipeline, removing manual bottlenecks and ensuring consistent coverage
Unit Verifies individual functions or components in isolation for correctness

end-of-life considerations

Every system component has a finite supported life. Planning for end-of-life (EOL) prevents the security debt that accumulates when unsupported components remain in production.

Lifecycle Management
Ensuring that unsupported software or hardware is upgraded or replaced before it becomes a security liability. Once a vendor stops issuing security patches, any discovered vulnerability in that component goes unaddressed indefinitely. Lifecycle management means tracking EOL/EOS dates for every component in the environment and planning upgrades or replacements well in advance — not reactively after support has already ended.
Data Migration and Secure Deletion
Moving data from decommissioned systems to supported ones, and securely wiping the old storage so that sensitive data cannot be recovered. Simply decommissioning a server without wiping its storage leaves all its data readable to anyone who later gains physical or logical access to it. Secure deletion standards (such as DoD 5220.22-M or NIST 800-88) ensure data is rendered unrecoverable. Example: when replacing an old server, migrate live data to the new system, then perform a secure wipe of the old storage before disposal or repurposing.
exam tips

key takeaways

  • SAST, DAST, IAST: know the distinction. SAST = static, no execution, catches code-level flaws early. DAST = dynamic, running app, simulates attacks from outside. IAST = agent inside running app, real-time feedback, most comprehensive. The exam gives a scenario and asks which testing type is being described.
  • SBOM is the inventory; SCA is the scanning. SBOM lists what components are in the software. SCA checks those components for known vulnerabilities. Both are software assurance concepts; they work together but are distinct tools.
  • RASP operates inside the running application. Unlike a WAF (which is external), RASP is embedded in the application process itself. It monitors and blocks threats from inside, in real time, without human intervention. Post-deployment control.
  • Functional requirements = what; non-functional = how well. Authentication is functional. Response time under 500ms is non-functional. Both are security requirements; the distinction determines how they are tested and verified.
  • Supply chain risk has two faces: hardware (counterfeit/tampered) and software (vulnerable libraries). Know both. A scenario about a backdoored network component is hardware supply chain risk. A scenario about a vulnerable npm package is software supply chain risk.
  • Canary testing deploys to a small group first. It is a CI/CD testing technique for detecting issues before a full rollout — not a penetration testing technique. Know it by name and purpose.
  • EOL and secure deletion go together. Decommissioning without secure deletion leaves data exposed. Any scenario involving retiring a server or device should trigger the answer: migrate data, then securely wipe the storage.
quick decision rules
  • Analyzing source code without running it → SAST (pre-deployment)
  • Simulating attacks against a live application from outside → DAST (pre-deployment)
  • Agent inside a running app providing real-time vulnerability feedback → IAST (pre-deployment)
  • Scanning a deployed live application for known weaknesses → Vulnerability analysis (post-deployment)
  • Security embedded in the running app that blocks threats automatically → RASP (post-deployment)
  • Complete inventory of all software components and dependencies → SBOM
  • Scanning software components for known CVEs → SCA
  • Mathematical proof of software correctness → Formal methods
  • Counterfeit or tampered physical component in the supply chain → Hardware supply chain risk
  • Vulnerable third-party library in the software stack → Software supply chain risk
  • Deploy to a small user group before full rollout → Canary testing
  • New code must not break existing functionality → Regression testing
  • Decommissioning a server: what must happen to its storage → Secure deletion

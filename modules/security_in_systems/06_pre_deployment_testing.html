Pre-Deployment Testing

Identifying security vulnerabilities, functional issues, and performance concerns before software reaches a live environment, using static, dynamic, and interactive techniques.

overview

Three complementary approaches cover pre-deployment security testing. SAST analyzes source code without execution, catching issues early. DAST attacks a running application to find runtime vulnerabilities. IAST works as an agent inside a running application during functional testing, combining the coverage of both. Using all three together gives the most thorough pre-deployment assurance.

method 01

SAST

Static: analyzes source code or binaries without running the application. Catches flaws early in development before any deployment occurs.

method 02

DAST

Dynamic: tests a running application by simulating attacks from the outside. Finds vulnerabilities that only manifest at runtime.

method 03

IAST

Interactive hybrid: an agent inside the running application provides real-time feedback during functional testing. Analyzes code and runtime behavior simultaneously.

SAST — static application security testing

SAST analyzes source code or compiled binaries for security vulnerabilities without executing the software. The goal is to catch flaws like SQL injection, cross-site scripting, buffer overflows, and improper error handling as early as possible, ideally before code is ever merged or built.

automated tools

Automated SAST tools scan entire codebases for patterns associated with known vulnerability classes. Common tools include Checkmarx, Veracode, and Fortify. A classic example: if a developer uses the C function strcpy, which does not check input size, an automated tool flags it and suggests strncpy as a safer alternative to prevent buffer overflows.

manual code review

Manual review by a developer inspecting a colleague's code catches what automated tools miss: complex logical flaws, improper error handling in edge cases, and insecure use of cryptographic libraries. For example, a reviewer might identify the use of sprintf, which is susceptible to buffer overflows, and recommend snprintf instead, which enforces a size limit on the output.

SAST limitations

  • False positives: flagged issues that are not actual vulnerabilities, requiring investigation time
  • Cannot detect vulnerabilities that only appear when the application is running (runtime logic, environment-dependent behavior)

DAST — dynamic application security testing

DAST tests a running application from the outside by simulating real attacks. It finds vulnerabilities that cannot be seen in static code: SQL injection through live input fields, cross-site scripting in rendered output, authentication bypass via improper session management, and similar runtime-only flaws.

key tools

OWASP ZAP (Zed Attack Proxy)
Open-source DAST tool. Actively probes a running application for common vulnerabilities.
Burp Suite
Commercial DAST platform. Intercepts and manipulates HTTP/S traffic to identify injection flaws, authentication weaknesses, and session handling issues.
Peach Fuzzer
Specializes in fuzzing: sending large numbers of random, malformed, or unexpected inputs to the application to surface crashes and security flaws. Example: a file upload feature tested with malformed or unexpected file types. If the application crashes or behaves erratically in response, it signals a potentially exploitable vulnerability.

DAST limitations

  • Only sees the application from the outside, so it cannot find internal logic flaws visible only in source code
  • Can generate false positives by flagging issues that are not actually exploitable
  • Less effective against vulnerabilities that require deep understanding of the application's business logic

IAST — interactive application security testing

IAST integrates an agent directly into a running application. As developers run their normal functional tests, the IAST agent simultaneously monitors both code execution and runtime behavior, providing immediate feedback on vulnerabilities as they are triggered.

Example: developers testing an e-commerce platform's payment gateway and user registration flow. The IAST agent observes the application in real time and surfaces issues like improper input validation (user inputs not sanitized) or insecure data storage (passwords not correctly encrypted). Because the feedback is immediate, issues are fixed before the application goes live.

key tools

  • Contrast Security: integrates with the application at runtime, provides real-time vulnerability detection and remediation guidance
  • Veracode: offers both SAST and IAST capabilities in one platform

IAST limitations

  • Only covers code paths that are actually exercised during testing. Parts of the application that are not tested remain unchecked.
  • The agent adds overhead to the running application, which can slow functional tests and affect development workflow

EXAM TIPS

what to know for the exam

  • SAST = static, no execution, analyzes code. DAST = dynamic, attacks running app from outside. IAST = agent inside running app during functional testing.
  • SAST tools: Checkmarx, Veracode, Fortify. DAST tools: OWASP ZAP, Burp Suite, Peach Fuzzer. IAST tools: Contrast Security, Veracode.
  • Fuzzing (Peach Fuzzer) sends random/malformed inputs to find crashes. Classic DAST technique.
  • False positives are a weakness shared by all three methods: flagged issues that turn out not to be real vulnerabilities.
  • IAST is the most comprehensive because it sees both the code and runtime at once, but only where the application is exercised during testing.
  • Best practice: use SAST + DAST + IAST in combination. Each covers blind spots the others miss.
decision rules
  • Scenario involves analyzing source code without running it → SAST
  • Scenario involves simulating attacks against a running application from outside → DAST
  • Scenario involves an agent providing real-time feedback during functional testing → IAST
  • Scenario involves sending malformed or random inputs to find crashes → fuzzing (Peach Fuzzer)
  • Question asks which method catches only runtime vulnerabilities → DAST
  • Question asks which method is a hybrid combining static and dynamic → IAST

Security Requirements

Defining what a system must do and how well it must perform to stay secure, and navigating the constant tension between stronger security and everyday usability.

overview

Security requirements are the specific criteria a system must meet to protect against threats and vulnerabilities. They divide into two categories: functional requirements (what the system must do, such as authentication and access controls) and non-functional requirements (how well it must do it, such as performance and reliability). A persistent implementation challenge is finding the right trade-off between security and usability.

type 01

Functional

Specific security-related behaviors and features a system must have. Authentication, access controls, encryption protocols. The "what must it do" layer.

type 02

Non-functional

Qualities of the system: performance, reliability, and usability under stress. The "how well must it do it" layer. Often in tension with stronger security controls.

functional security requirements

Functional requirements define the concrete, observable defenses a system must implement. They are testable: either the feature exists and works, or it does not.

Authentication mechanisms
Ensuring only authorized users can access a system. May involve passwords, biometric data, certificates, or multi-factor authentication (MFA). MFA requires verification through two or more factors, such as a password combined with a one-time code or fingerprint scan.
Access controls
Determining who is permitted to access which resources within the system. Ensures users can only interact with the parts of the system relevant to their role. Implemented through role-based access control (RBAC), attribute-based access control (ABAC), or discretionary/mandatory models.
Encryption protocols
Securing communications and data by converting sensitive information into unreadable ciphertext. Prevents unauthorized parties from accessing or interpreting data in transit or at rest. Examples include TLS for in-transit protection and AES-256 for at-rest encryption.

non-functional security requirements

Non-functional requirements focus on the qualities that allow a system to remain secure and operational under real-world conditions. They are harder to measure but equally critical.

Performance
Security controls must not degrade the system beyond acceptable limits. Encryption algorithms must be robust enough to resist attack while also optimized to avoid significant processing delays. In environments requiring real-time data handling (financial transactions, healthcare networks), a system that is secure but too slow becomes a liability.
Reliability
The system must remain operational even under high traffic, hardware failures, or partial outages. Achieved through redundancy (backup components that activate on failure) and failover mechanisms (automatic rerouting to alternate pathways). Reliability ensures availability and security are maintained even during disruptions.

security vs. usability

A common challenge in enterprise IT is that stronger security controls often reduce convenience for legitimate users. The goal is not to eliminate security, but to find the right balance for the environment and risk profile.

common trade-off scenarios

multi-factor authentication (MFA)

MFA adds an essential security layer, preventing unauthorized access even when a password is compromised. However, employees who authenticate multiple times per day across multiple systems can find it cumbersome. Solutions like single sign-on (SSO) can reduce friction while preserving the security benefit.

encryption latency

Encrypting and decrypting data introduces processing overhead. In environments requiring real-time operations, such as financial transactions or large-scale data transfers, administrators may need to choose between more complex (slower) algorithms for maximum security or lighter encryption that preserves system responsiveness.

granular access controls

Highly specific access controls protect against insider threats by limiting what users can do. However, requiring multiple approvals to access a tool or dataset can slow workflows significantly, delaying project timelines and frustrating legitimate users performing routine tasks.

EXAM TIPS

what to know for the exam

  • Functional requirements define specific behaviors: authentication, access controls, encryption. Non-functional requirements define qualities: performance, reliability, usability.
  • The security vs. usability tension is a core theme: stronger controls (more complex encryption, granular RBAC, MFA) can reduce efficiency and user convenience
  • MFA is the classic security-usability trade-off example. SSO is the mitigation that preserves security while reducing friction.
  • Redundancy and failover are non-functional reliability measures, not security features per se, but they support availability (the A in CIA)
  • Encryption introduces latency. Real-time environments (financial, healthcare) feel this most acutely.
decision rules
  • Question asks what a system must do (specific behavior or feature) → functional security requirement
  • Question asks how well the system must perform (speed, uptime, resilience) → non-functional security requirement
  • Scenario describes a security control slowing down users or workflows → security vs. usability trade-off
  • Scenario describes a way to reduce MFA friction without removing MFA → single sign-on (SSO)

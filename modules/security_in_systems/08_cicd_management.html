CI/CD Management

Automating code integration, testing, and deployment through consistent standards, enforced quality gates, and continuous refinement of the pipeline.

overview

CI/CD management automates the process of integrating code changes, running tests, and deploying applications to ensure consistent, efficient software delivery. Four practices form its foundation: coding standards enforce consistent style across a team, linting catches bugs and security issues before code runs, branch protection prevents untested code from reaching production, and continuous improvement keeps the pipeline effective against new threats and changing requirements.

practice 01

Coding Standards

Guidelines for how code should be written. Ensures consistency across a team, improves readability, and reduces the risk of errors and maintenance debt.

practice 02

Linting

Automated checks that enforce best practices and flag potential bugs or security issues before code is executed. An early warning system for the pipeline.

practice 03

Branch Protection

Rules that prevent untested or unreviewed code from being merged into important branches like main. Code reviews and automated tests are required gates.

practice 04

Continuous Improvement

Ongoing refinement of the pipeline based on feedback, performance data, and new security requirements. Keeps the pipeline efficient and threat-aware over time.

coding standards

Coding standards define how code should be written across a team. Without them, large codebases developed by multiple contributors become inconsistent and hard to maintain. Standards address concerns like indentation (spaces vs tabs), naming conventions, and file structure.

naming convention examples

  • CamelCase: first letter of each word capitalized, no spaces (getUserProfile, parseAuthToken)
  • snake_case: all lowercase, words separated by underscores (get_user_profile, parse_auth_token)

key tools

  • Prettier: opinionated formatter that automatically enforces consistent code style (spacing, line breaks, quote style)
  • ESLint: configurable linting and style enforcement for JavaScript. Can check both formatting standards and code quality rules.

linting

Linting goes beyond style to enforce best practices and flag code patterns that are likely to cause bugs or security vulnerabilities. Where coding standards ensure the code looks consistent, linting ensures the code behaves safely.

ESLint in JavaScript flags issues like unused variables (code that was declared but never called, indicating a logic error or dead code) and unreachable code (statements following a return or break that can never execute). Linters also catch security-relevant patterns: for example, flagging places where user input is passed directly into a query or command without sanitization, a common precursor to injection attacks.

Linting acts as an early warning system: problems caught at the linting stage cost far less to fix than problems discovered in production.

branch protection

In team-based development, the main branch of a repository holds stable, production-ready code. Branch protection rules prevent anyone from pushing unreviewed or untested changes directly to it. Instead, developers work on separate feature branches, and changes are only merged into main after passing a series of required checks.

common required checks

  • Code review: at least one other developer must approve the changes before merge
  • Automated tests: all tests in the CI pipeline must pass before the merge is allowed
  • Linting: code must pass style and quality checks as part of the merge process

automation tools

  • GitHub Actions: runs automated checks (tests, linting, builds) triggered by pull request events
  • CircleCI: CI/CD platform that automates build, test, and deployment workflows integrated with version control

continuous improvement

Continuous improvement means regularly refining the CI/CD pipeline based on feedback, performance metrics, and emerging threats. This includes introducing new security tools, optimizing slow steps, and updating practices as the threat landscape evolves.

  • Dependabot: automatically scans project dependencies for known vulnerabilities and opens pull requests to update affected packages when fixes are available
  • If builds are taking too long, teams look to run tasks concurrently or eliminate unnecessary steps to restore pipeline efficiency
  • New vulnerability classes may warrant adding new SAST or SCA tools to the pipeline without waiting for a scheduled review

EXAM TIPS

what to know for the exam

  • Coding standards = style and consistency (Prettier, ESLint). Linting = best practices and bug/security pattern detection (ESLint also handles this). ESLint does both.
  • Branch protection prevents untested or unreviewed code from reaching main. Required checks typically include code review and passing automated tests.
  • GitHub Actions and CircleCI automate checks during the merge process as part of branch protection.
  • Dependabot is the key continuous improvement tool: auto-scans dependencies for CVEs and raises pull requests with fixes.
  • Unreachable code = code that follows a return/break statement and can never execute. Linters flag it.
decision rules
  • Scenario involves enforcing consistent code style across a team → coding standards (Prettier, ESLint)
  • Scenario involves flagging unused variables or unsanitized input before code runs → linting (ESLint)
  • Scenario involves preventing unreviewed code from reaching the main branch → branch protection
  • Scenario involves automating test runs on pull requests → GitHub Actions or CircleCI
  • Scenario involves automatically detecting vulnerable dependencies and suggesting updates → Dependabot

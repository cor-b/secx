Software Assurance

Ensuring software is built and maintained to meet security and reliability standards through transparency, composition analysis, and mathematical verification.

overview

Software assurance covers three interlocking practices. A Software Bill of Materials (SBOM) provides full transparency into every component in an application. Software Composition Analysis (SCA) scans those components for known vulnerabilities. Formal methods use mathematical techniques to prove software behaves correctly under all conditions, including in safety-critical systems where failures carry severe consequences.

concept 01

SBOM

A complete inventory of all components, libraries, and dependencies in a software application, including transitive dependencies.

concept 02

SCA

Scanning software to identify third-party components and check them against known vulnerability databases. Automated tools do the heavy lifting.

concept 03

Formal Methods

Mathematically-based validation techniques that prove software correctness under all possible conditions, not just tested ones.

software bill of materials (SBOM)

An SBOM is a detailed ingredient list for software. Every component, library, and dependency is catalogued with its name, version, and a cryptographic hash that verifies the component has not been altered. The SBOM covers both direct and transitive dependencies.

dependency types

Direct dependencies
Libraries and modules explicitly imported or called by the software. If a web application imports React for its frontend and Express.js for its backend, both are direct dependencies. They appear in the project's package manifest and are intentionally chosen by developers.
Transitive dependencies
Components that direct dependencies rely on, pulled in automatically. If Library A (a direct dependency) depends on Library B, then Library B is a transitive dependency of the application, even though no developer explicitly chose it. These are the dependencies that are easiest to overlook and hardest to audit without tooling.

Direct and transitive dependencies together form a dependency graph, a complete map of everything running inside the application. If a vulnerability is found in a specific version of React, an SBOM lets teams immediately identify whether that version is in use, rather than auditing the codebase manually.

software composition analysis (SCA)

SCA involves scanning software to detect all third-party components and checking them against known vulnerability databases. It automates the process of staying on top of both direct and transitive dependencies as they change over time.

key tools

OWASP Dependency-Check
Scans a project and its libraries to identify publicly disclosed vulnerabilities (CVEs) in any component. Point-in-time scan. Run it as part of a build pipeline to catch vulnerable components before code is shipped. Widely used and integrates with common CI/CD systems.
OWASP Dependency-Track
Continuous monitoring and analysis of component security across the full software portfolio. Goes beyond a one-time scan by continuously ingesting new vulnerability data and alerting teams when a newly disclosed CVE affects a component already in production. Supports SBOM ingestion (CycloneDX format).

the equifax breach: an SCA failure

In 2017, attackers exploited CVE-2017-5638, a remote code execution vulnerability in Apache Struts 2. The flaw was in the Jakarta Multipart Parser, which improperly handled file upload requests, allowing attackers to execute arbitrary commands on the server. Equifax had not patched the vulnerable version of Apache Struts in their systems despite a fix being available.

This incident illustrates the core SCA problem: organizations may trust frameworks (Apache Struts, Django, Ruby on Rails, Hibernate, Microsoft .NET) without tracking which versions they run or whether those versions have known vulnerabilities. When a CVE is published, organizations without SCA tooling have no automated way to know they are exposed.

formal methods of validation

Formal methods use mathematical proofs and rigorous models to verify that software behaves correctly under every possible condition, not just the conditions covered by test cases. They are especially valuable in aviation, healthcare, and finance, where failures can have catastrophic consequences.

Model checking
Creates a model of the system and exhaustively verifies it against predefined rules. Tool: SPIN (Simple Promela Interpreter). Used in distributed systems to explore every possible state and transition, identifying issues like deadlocks or safety violations that standard testing would miss. Example: verifying that no two traffic light directions receive a green signal simultaneously.
Theorem proving
Generates formal mathematical proofs to verify the correctness of algorithms and software components. Tools: Coq and Isabelle. Coq has been used to formally verify cryptographic algorithms, providing a mathematical guarantee that the algorithm meets its security specification under all inputs, not just tested ones.
Abstract interpretation
Analyzes software behavior without executing it, checking for runtime errors across all possible execution paths. Tool: Astree. Used by Airbus to verify flight control software, checking for runtime errors like division by zero and buffer overflows across all code paths. Finds issues that traditional testing cannot reliably surface because not every path can be exercised at runtime.

EXAM TIPS

what to know for the exam

  • SBOM = complete ingredient list of all components (direct and transitive), each identified by name, version, and cryptographic hash
  • Transitive dependencies are the components your dependencies depend on. They are the easiest to miss and the most common source of untracked vulnerabilities.
  • SCA tools scan for CVEs in third-party components. OWASP Dependency-Check = point-in-time scan; OWASP Dependency-Track = continuous monitoring
  • The Equifax breach (CVE-2017-5638) came from an unpatched Apache Struts 2 vulnerability. Classic SCA failure: known fix available, not applied.
  • Formal methods provide mathematical guarantees. SPIN = model checking, Coq/Isabelle = theorem proving, Astree = abstract interpretation (Airbus)
  • Formal methods are for critical systems (aviation, healthcare, finance) where standard testing cannot cover all possible failure states
decision rules
  • Scenario involves listing all components in a software application for transparency → SBOM
  • Scenario involves scanning code for vulnerable third-party libraries → SCA (Dependency-Check or Dependency-Track)
  • Continuous vs point-in-time SCA: ongoing monitoring → Dependency-Track; single build scan → Dependency-Check
  • Scenario involves mathematically proving software correctness for a critical system → formal methods
  • Question names SPIN → model checking; Coq/Isabelle → theorem proving; Astree → abstract interpretation

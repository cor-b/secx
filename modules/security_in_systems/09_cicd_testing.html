CI/CD Testing

Systematically evaluating code changes throughout the pipeline to catch vulnerabilities and functional issues before they reach production.

overview

Five testing types work together in a CI/CD pipeline. Canary testing limits exposure by rolling changes to a small user group first. Regression testing confirms existing functionality survives new changes. Automated testing runs all checks continuously on every integration. Unit testing verifies individual components in isolation. Integration testing checks how components interact with one another.

canary testing

Canary testing takes its name from the practice of using canaries in coal mines: miners brought caged canaries underground to detect hazardous gases before humans were affected. In software, the same principle applies. A new feature or code update is deployed to a small subset of users first, while the rest of the user base continues using the previous version.

The goal is to observe real-world behavior at limited scale: catching security vulnerabilities, memory leaks, or crashes before they affect everyone. If problems arise, developers roll back the change without impacting the full user base. If the canary group shows no issues, the deployment proceeds to everyone.

  • Detects issues that only appear under real conditions (not caught in pre-production testing)
  • Rollback is safe and fast because only a small subset is exposed
  • Adds a quality control layer between staging and full production rollout

regression testing

Regression testing reruns previously successful test cases after new code is introduced, confirming that the changes have not broken existing functionality. If a new feature enhances user authentication, regression testing verifies that existing features (profile management, password recovery, session handling) continue working as intended.

In CI/CD pipelines, regression tests typically run automatically after new code is merged into the main branch. Because regression testing involves rerunning the same suite of tests repeatedly, it is well-suited for automation, freeing developers to focus on writing new tests and resolving newly discovered issues.

automated testing and retesting

Automated testing runs testing scripts automatically every time new code is integrated into the pipeline. It is not a standalone test type but rather the mechanism that makes all other testing types continuous and reliable. Automated test and retest means that if a bug is found and fixed, the pipeline automatically reruns the relevant tests to confirm the fix works and that no new issues were introduced in the process.

  • Automated tests cover unit tests, integration tests, regression tests, and more in a single pipeline run
  • Eliminates human error from manual test execution
  • Dramatically speeds up feedback in large codebases where manual testing would be impractical

unit testing

Unit testing tests individual functions or classes in isolation, verifying that each unit of code produces the correct output for a given input. Tests are written alongside the production code and run frequently, giving developers immediate feedback on whether their changes break any individual component.

Because unit tests focus on the smallest pieces of the codebase, they identify the exact location of a bug with precision. However, they cannot confirm that the components work together correctly. Unit testing must be complemented by integration testing to validate the whole system.

integration testing

Integration testing focuses on how different units or modules interact with each other. While unit tests check individual pieces in isolation, integration tests verify that those pieces work together correctly in a composed system.

Examples: testing how an authentication service communicates with a database, or how the frontend interacts with a backend API. Integration tests surface issues that unit tests cannot find: data mismatches between services, broken API contracts, misconfigured communication channels, and inter-service authentication failures. These tests are especially critical in microservices architectures where many independent services must function together seamlessly.

EXAM TIPS

what to know for the exam

  • Canary testing = deploy to a small user subset first. Named after canaries in coal mines. Allows rollback before full blast radius.
  • Regression testing = reruns old tests after new changes. Runs after merge to main. Ensures new code does not break existing functionality.
  • Automated test and retest = the mechanism, not a test type. Enables all other tests to run continuously without human intervention.
  • Unit testing = individual function or class in isolation. Precise, fast, but cannot confirm components work together.
  • Integration testing = tests how units interact. Catches data mismatches, broken APIs, misconfigured services. Complements unit testing.
  • Unit tests alone are not enough. Integration testing is required to validate the assembled system.
decision rules
  • Scenario involves deploying to a small user group to catch issues before full rollout → canary testing
  • Scenario involves confirming existing features still work after a code change → regression testing
  • Scenario involves tests that run on every commit automatically → automated testing
  • Scenario involves verifying a single function produces the correct output → unit testing
  • Scenario involves testing how an auth service communicates with a database → integration testing
  • Question asks which test type runs in production first → canary
  • Question asks which test type catches broken API contracts between services → integration testing

Post-Deployment Testing

Identifying and addressing vulnerabilities in a live environment through continuous scanning and real-time runtime protection.

overview

Post-deployment security operates on two tracks. Vulnerability analysis continuously scans the deployed application and its infrastructure for known weaknesses, catching new threats as they emerge. Runtime Application Self-Protection (RASP) operates inside the running application, detecting and blocking attacks in real time based on observed behavior rather than static signatures.

method 01

Vulnerability Analysis

Continuously scans the live application and infrastructure for known vulnerabilities: unpatched software, misconfigurations, insecure code.

method 02

RASP

An agent embedded inside the running application. Monitors behavior in real time and blocks attacks as they occur, from inside the runtime environment.

vulnerability analysis

Vulnerability analysis scans a deployed application and its surrounding infrastructure for known security weaknesses: unpatched software, misconfigurations, and insecure code. The goal is to identify weaknesses before attackers can exploit them. Because new vulnerabilities are disclosed continuously after software ships, recurring scheduled scans are essential, not a one-time activity.

OpenVAS

OpenVAS is an open-source vulnerability scanner that detects outdated software, insecure configurations, and missing patches across systems, networks, and applications. It generates detailed reports to help prioritize remediation by severity.

configuring recurring scans in OpenVAS

  1. 01 Create a target: define the systems or network ranges to scan (e.g., a subnet like 192.168.0.0/24)
  2. 02 Create a scan task: define the scan configuration (e.g., full and fast) and assign the target
  3. 03 Schedule the scan: set a recurring schedule (e.g., every Saturday at 01:00 UTC) to automate ongoing coverage
  4. 04 Configure alerts: set up notifications for high-severity findings so teams are informed immediately when critical issues surface

Recurring scans are critical because even previously secure environments become vulnerable over time when new software updates, configuration changes, or newly disclosed CVEs introduce fresh risks.

vulnerability analysis limitations

  • Zero-day blind spot: scanners rely on databases of known vulnerabilities. A zero-day (a previously unknown threat with no published CVE or patch) will not appear in scan results.
  • False positives: flagged issues that are not actual vulnerabilities slow down remediation as security teams investigate non-threats.

runtime application self-protection (RASP)

RASP operates from inside the application, embedding directly into the runtime environment. Unlike perimeter defenses or external scanners, RASP monitors the application's actual behavior as it runs and can block an attack the moment it is detected, before any damage occurs.

Example: a RASP tool detects an attempted SQL injection attack against a live banking application. Rather than logging the event for later review, RASP immediately blocks the malicious request, preventing unauthorized database access. This real-time response is what distinguishes RASP from reactive scanning.

key tools

Contrast Security
Integrates into the application at runtime. Monitors for abnormal behavior, unexpected input patterns, and known attack signatures, blocking threats automatically.
Imperva RASP
Embeds into the application layer to detect and block attacks in real time, including injection attacks and zero-day exploits that bypass perimeter controls.

RASP limitations

  • Performance impact: operating inside the application adds overhead, which can slow performance in resource-intensive environments
  • RASP is not a replacement for pre-deployment testing. It defends against runtime attacks but does not fix underlying code vulnerabilities, which still require code review, SAST, and DAST.

EXAM TIPS

what to know for the exam

  • Post-deployment testing runs after software ships to a live environment. Pre-deployment runs before.
  • Vulnerability analysis scans for known CVEs in the live environment. OpenVAS is the key open-source tool: detects outdated software, misconfigurations, missing patches, and supports recurring scheduled scans.
  • Zero-day = a previously unknown vulnerability with no CVE or patch. Vulnerability scanners cannot detect them because they are not in the database.
  • RASP operates inside the running application and blocks attacks in real time. It is a preventive control, not just detective.
  • RASP is not a substitute for pre-deployment testing (SAST/DAST/IAST). It addresses runtime threats but does not fix code-level vulnerabilities.
  • Contrast Security appears in both IAST (pre-deployment) and RASP (post-deployment). Context matters: IAST = during functional testing; RASP = in live production.
decision rules
  • Scenario involves scanning a live deployed application for known vulnerabilities → vulnerability analysis (OpenVAS)
  • Scenario involves a threat that has no patch and is unknown to scanners → zero-day (vulnerability analysis will miss it)
  • Scenario involves blocking an attack in real time inside a running production application → RASP
  • Question asks about control type: vulnerability analysis = detective; RASP = preventive
  • Scenario states RASP is being used as the only security measure → flag this as insufficient; RASP does not replace pre-deployment testing

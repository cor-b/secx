The Data Lifecycle

Six primary stages governing how data is managed from creation to destruction, and why staging is not one of them.

overview

Every piece of data passes through six stages. Staging is a commonly tested trap. It is an intermediate phase between creation and use, not a primary lifecycle stage. The four environments (dev, test, QA, production) all live within staging.

The 6 Stages

  1. Creation Data is acquired, entered, captured, or generated. Examples: receiving an email, a device generating system logs, a form submission.
  2. Use Data is accessed, processed, or modified. Includes the audit trail: logs of who accessed what and when belong here.
  3. Sharing Data is made available to others. Example: a financial report sent to external partners via email or a shared drive.
  4. Storage Data is maintained for future use. Example: saving annual financial records in a database for trend analysis.
  5. Archival Data moved to long-term storage for later recovery, still accessible but not in active use. Example: customer records older than 2 years moved to an archive for historical reference.
  6. Destruction Data is securely removed when no longer valuable, freeing storage and eliminating exposure. Example: deleting financial records older than 7 years per data retention policy.

memory hook

Creation→Use→Sharing→Storage→Archival→Destruction "Can Using Shared Storage Allow Destruction?" (C, U, S, S, A, D)

Staging: The Exception

exam trap

Staging is NOT one of the six primary stages. If asked "which of the following is NOT a stage of the data lifecycle," staging is the answer.

Staging is an intermediate phase between Creation and Use. It is temporary storage where data is prepared, validated, and tested before moving into production.

Purpose of staging

  • Development: building and configuring the system
  • Testing: finding bugs and errors before release
  • Quality assurance: validating against standards before go-live
  • Ensures data integrity before it reaches the production environment

The Four Environments

All four environments live inside the staging phase; they represent the progression from building to deploying.

  1. Development Build phase: writing code, designing architecture, configuring software. Multiple iterations before moving on. Example: developers writing features for a new mobile app.
  2. Testing Find bugs phase: the system is evaluated against various scenarios to catch errors and inconsistencies. Example: simulating production-level workloads on a new software update.
  3. Quality Assurance (QA) Validate against standards: ensures the system meets required standards and is ready for release. Catches issues missed during testing. Example: testing a website across multiple browsers and operating systems.
  4. Production Live phase: system is fully deployed and actively supports real users and business processes. Example: an e-commerce site goes live and customers begin purchasing.

environment progression

Development→Testing→Quality Assurance→Production build  ·  find bugs  ·  validate  ·  go live

exam tips

Key Takeaways

  • Staging is NOT one of the six stages. It is the most common trap answer on this topic.
  • Know the order: Creation is always first, Destruction is always last.
  • Don't confuse the six lifecycle stages with the four environments. They describe different things.
  • Archival = long-term storage with recovery option. Destruction = permanent, secure removal. These are different.
  • Audit trails and logs belong to the Use stage.
quick decision rules
  • "Which is NOT a primary data lifecycle stage?" → Staging
  • Long-term retention with recovery option → Archival
  • Permanent, secure removal → Destruction
  • Real users interacting with the live system → Production
  • Checking across browsers and OSes for consistency → Quality Assurance
  • Audit logs and access records → Use stage

Injection Vulnerabilities

When unvalidated input becomes executed instructions. Six injection flaws, distinguished by what gets injected and whose trust gets abused.

overview

Injection vulnerabilities occur when an attacker injects malicious code through an input field or interface, leading to unintended command execution. Six types: command injection, code injection, cross-site scripting (XSS), cross-site request forgery (CSRF), server-side request forgery (SSRF), and deserialization. The common root cause and fix: missing input validation and sanitization.

command vs code injection

Command injection
Executes arbitrary commands on the host operating system via flaws in an app that passes user input into a system command. A ping field abused with ; rm -rf /path can delete files or open a reverse shell. Highest impact (full OS access).
Code injection
Inserts malicious code executed by the application itself (in the app's language), affecting application behavior, not the OS directly. E.g. user input passed to eval() running os.system('ls'). Can bypass app-layer controls but doesn't give direct OS access.

SQL injection and XSS

SQL injection inserts malicious SQL into a database query to alter/delete records or gain admin control over the database. Cross-site scripting (XSS) injects malicious scripts into web pages, three types:

  • Reflected XSS: script in a URL/form reflected back to the user without sanitization, delivered via phishing links; affects only the user who clicks.
  • Stored XSS: script saved on the server (comment, post, profile) and executed for every visitor; persistent and high-reach.
  • DOM-based XSS: script executes by modifying the client-side DOM (no server involvement); hard to detect. Avoid innerHTML/document.write with user data.

Exam tip: <script> tags or JavaScript in a URL/log = XSS; anything with document. (e.g. document.cookie) = DOM-based XSS.

CSRF vs SSRF and deserialization

Cross-site request forgery (CSRF)
Tricks an authenticated user into performing unintended actions, abusing the trust the server has for the client (e.g. an unwanted bank transfer). Defend with anti-CSRF tokens and re-authentication.
Server-side request forgery (SSRF)
Forces the server to make unauthorized requests to internal/external resources, abusing the trust between a front-end and back-end (e.g. retrieving cloud metadata). Defend by validating and limiting allowable destinations.
Deserialization
Injects malicious data into serialized content; when the app converts it back into an object, embedded code runs, leading to remote code execution, privilege escalation, or corruption. Avoid deserializing untrusted data; restrict allowed object types.
EXAM TIPS

what to know for the exam

  • Command injection = OS commands (highest impact); code injection = runs in the app's language, affects the app.
  • XSS types: reflected (URL, one user), stored (server, all visitors), DOM-based (client-side, document.*).
  • CSRF abuses server-to-client trust (tricks the user); SSRF abuses front-end-to-back-end trust (tricks the server).
  • Deserialization runs embedded code when serialized data is rebuilt into an object.
  • Root cause/fix for all: missing input validation and sanitization.
decision rules
  • Scenario runs OS commands via a ping field → command injection
  • Scenario stores a script that runs for every page visitor → stored XSS
  • Scenario forces the server to fetch cloud metadata → SSRF
  • Scenario tricks a logged-in user into an unwanted transfer → CSRF
  • Scenario sees document.cookie in a payload → DOM-based XSS

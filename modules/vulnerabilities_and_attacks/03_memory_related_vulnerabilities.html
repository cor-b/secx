Memory-related Vulnerabilities

Flaws in how an app handles memory: outdated functions, sloppy allocation, overflowing buffers, and timing races, all of which can let attackers run code or crash a system.

overview

Memory-related vulnerabilities are flaws in how an application handles memory, letting attackers execute code, crash systems, or leak data. Five concepts: deprecated functions, unsafe memory utilization, overflows, race conditions, and time-of-check to time-of-use (TOCTOU).

deprecated functions and unsafe memory

Deprecated functions
Older, insecure functions still in use that lack modern safety features. In C, gets() has no buffer-size check (overflow risk); the modern fgets() lets you specify a buffer size.
Unsafe memory utilization
Failing to manage memory correctly, causing buffer overflows or memory leaks (continuously allocating without releasing, eventually exhausting memory). Defend with memory checks, input validation, and protections like ASLR (randomizes memory addresses) and DEP (blocks code execution in data-only regions).

overflows

Buffer overflow
Storing more data than a buffer holds, spilling into adjacent memory (like overfilling a cup). Attackers overwrite the stack's return address to point to malicious code, stack smashing, often using a NOP slide (no-operation instructions that "slide" execution into their code).
Integer overflow
A math result exceeds a variable's storage capacity, wrapping around (e.g. a positive value becoming negative). Can alter buffer boundaries and enable buffer overflows. Prevent with boundary checks and suitable variable sizes.

race conditions and TOCTOU

A race condition occurs when the outcome depends on the timing/order of events that don't execute as intended, often when multiple threads access shared resources without synchronization. The classic example is Dirty COW (2016), a Linux privilege-escalation flaw in the copy-on-write mechanism that turned read-only access into write access and left no log traces.

Time-of-check to time-of-use (TOCTOU) is a specific race condition: an attacker changes a resource's state in the gap between when the system checks a condition (e.g. permissions) and when it uses the result, swapping in a sensitive file after the permission check but before access. Mitigate race conditions and TOCTOU with synchronization, locks and mutexes (mutual exclusion, so only one thread accesses a code section at a time), and database resource locking (e.g. SharePoint locking checked-out files).

EXAM TIPS

what to know for the exam

  • Deprecated functions (C gets()) lack safety checks; use modern equivalents (fgets()).
  • Buffer overflow = spill into adjacent memory (stack smashing, NOP slide, overwrite return address); integer overflow = value wraps around.
  • ASLR randomizes memory addresses; DEP blocks execution in data regions, both defend against memory exploits.
  • Race condition = timing-dependent outcome (Dirty COW); TOCTOU = state changed between check and use.
  • Fix races/TOCTOU with locks and mutexes (mutual exclusion).
decision rules
  • Scenario overwrites a return address via too much input → buffer overflow (stack smashing)
  • Scenario's value wraps from positive to negative → integer overflow
  • Scenario swaps a file between permission check and access → TOCTOU
  • Scenario fixes concurrent access with mutual exclusion → mutex/lock

Memory-based Attacks

Exploiting how a program manages memory: overflowing buffers and dereferencing null pointers, or slipping a malicious layer between an app and the OS to intercept everything.

overview

Memory-based attacks exploit a system's memory management to execute code, manipulate data, or crash systems. Two concepts: memory attacks (exploiting flaws like buffer overflows and null pointer dereferences) and shimming (placing a malicious layer between an app and the OS to intercept and manipulate system calls).

attack 01

Memory Attacks

Exploit memory management: buffer overflow (inject code into adjacent memory) and null pointer dereference (crash/DoS, sometimes escalation).

attack 02

Shimming

A malicious code layer between app and OS intercepting system calls. Low-level, stealthy; enables privilege escalation, data theft.

memory attacks

Memory attacks target memory management to exploit vulnerabilities and run harmful code, hard to detect because they leverage the system's own memory processes. Two examples:

Buffer overflow
Inputting data beyond a buffer's capacity, overwriting adjacent memory; an attacker replaces legitimate code with malicious instructions that run with the vulnerable app's permissions.
Null pointer dereference
Accessing/modifying data through a pointer with a null (zero) value pointing to no valid memory. Causes a crash or abnormal termination, can be a denial of service, and combined with other flaws can escalate privileges or run code.

Mitigate with secure coding (input validation, bounds checking), memory-safe languages/libraries, and ASLR + DEP to make addresses unpredictable and block execution in data regions.

shimming attacks

A shimming attack places a malicious shim (a layer of code) between an application and the operating system to intercept and manipulate system calls, controlling inputs/outputs to achieve unauthorized actions. Shims are favored for persistence and stealth because they operate at a low level below standard applications, hard for antivirus to detect. Example: an attacker injects a shim that intercepts authentication system calls to capture credentials or redirect commands, using methods like API hooking or reflective DLL injection, escalating from standard user to administrator unnoticed. Prevent with code-integrity practices (only trusted/signed code loads), application allow-listing, Secure Boot, regular patching, and monitoring tools (EDR, SIEM) that detect unusual system-call or privilege activity.

EXAM TIPS

what to know for the exam

  • Memory attacks exploit memory management: buffer overflow (overwrite adjacent memory) and null pointer dereference (crash/DoS).
  • Shimming inserts a malicious layer between an app and the OS to intercept system calls (API hooking, reflective DLL injection).
  • Shims are stealthy and persistent because they operate below normal applications.
  • Defenses: ASLR + DEP for memory attacks; code integrity, allow-listing, Secure Boot, EDR for shimming.
  • This lesson is the OBJ 3.4 attack counterpart to the OBJ 4.2 memory-related vulnerabilities.
decision rules
  • Scenario overflows a buffer to inject code → memory attack (buffer overflow)
  • Scenario crashes an app via a null pointer → null pointer dereference
  • Scenario intercepts system calls via an injected layer → shimming
  • Scenario uses API hooking to capture credentials → shimming attack

Hardware and Firmware Attacks

Attacks below the operating system, where antivirus can't see. Tampering with device firmware, hijacking the boot process, or smuggling code in through a USB port.

overview

Hardware and firmware attacks exploit vulnerabilities in physical components or embedded software, granting deep, hard-to-detect access at a fundamental machine level. Three TTPs: firmware tampering (modifying embedded device software), BIOS/UEFI attacks (targeting boot firmware), and USB-based attacks (malicious USB devices delivering code).

attack 01

Firmware Tampering

Modify embedded firmware (hard drives, NICs) stored in EEPROM/flash. Persistent backdoor antivirus can't scan. Fix: signed updates, integrity checks.

attack 02

BIOS/UEFI Attacks

Target boot firmware, control the system before the OS loads, persists through reboots/reinstalls. Fix: Secure Boot, firmware patches.

attack 03

USB-based Attacks

Malicious USB (Rubber Ducky, BadUSB) delivers code via direct physical connection, bypassing antivirus/firewalls. Fix: USB policy, disable autorun.

firmware tampering

Firmware is the low-level software running on hardware (hard drives, network cards, smart devices). Firmware tampering modifies it to control or alter device behavior, granting persistent control that's hard to detect because antivirus doesn't scan the hardware memory (EEPROM, flash) where firmware lives. Example: modifying a network card's firmware to intercept data packets. Tampered firmware acts as a low-level backdoor. Prevent with firmware updates from verified sources only, limited physical access, firmware integrity checks, and cryptographic signatures on updates.

BIOS and UEFI attacks

The BIOS (basic input/output system) or UEFI (unified extensible firmware interface) controls the boot process, initializing and loading the OS. Attacks here are especially dangerous because they take control before the OS loads and persist through reboots and OS reinstalls. Injecting malicious code into BIOS/UEFI firmware gives stealthy control to intercept data or disable security as soon as the device powers on. Prevent with Secure Boot (only signed/trusted firmware loads), regular BIOS/UEFI patches from manufacturers, and tamper-evident hardware/physical security.

USB-based attacks

USB-based attacks exploit the physical USB connection to introduce malicious code directly into hardware, bypassing antivirus, firewalls, and network IDS because they leverage direct physical access. Examples: a Rubber Ducky or BadUSB, devices that look like normal USB drives but contain hidden scripts executing on connection, used to install keyloggers, modify settings, or download malware. Prevent by prohibiting unauthorized USB devices, disabling autorun, using USB security software to authenticate devices, and in high-security environments using data-only USB cables and physically blocking unused ports.

EXAM TIPS

what to know for the exam

  • These attacks operate below the OS, antivirus often can't detect them.
  • Firmware tampering modifies device firmware (EEPROM/flash) for a persistent backdoor; fix with signed updates + integrity checks.
  • BIOS/UEFI attacks run before the OS loads and survive reinstalls; fix with Secure Boot and firmware patches.
  • USB-based attacks (Rubber Ducky, BadUSB) use direct physical access to bypass software defenses; disable autorun, restrict USB.
  • Secure Boot ensures only signed/trusted firmware loads at boot.
decision rules
  • Scenario modifies a NIC's firmware to sniff packets → firmware tampering
  • Scenario controls a system before the OS loads → BIOS/UEFI attack
  • Scenario uses a disguised USB to run hidden scripts → USB-based attack (BadUSB)
  • Scenario enables Secure Boot to block untrusted firmware → BIOS/UEFI defense

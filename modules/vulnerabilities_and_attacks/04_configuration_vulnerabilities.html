Configuration Vulnerabilities

Weaknesses you create yourself: over-permissioned directories, insecure defaults, secrets baked into code, and software too old to be safe.

overview

Configuration vulnerabilities occur when systems, apps, or networks are improperly configured. Five types: directory service misconfiguration, unsecure configuration, embedded secrets, outdated/unpatched software and libraries, and end-of-life software.

directory service misconfiguration

Improper setup of directory services like Active Directory, typically over-broad permissions or poor oversight, lets lower-privileged users reach sensitive/admin resources, enabling privilege escalation. Attackers use tools like BloodHound to map nested group memberships, trust paths, and permission inheritance, revealing hidden escalation paths. Mitigate with least-privilege access, routine directory audits, and monitoring for unusual access.

unsecure configuration

Improperly set parameters, often from default settings prioritizing usability over security (open services, management ports, weak default SSH credentials). The 2019 Capital One breach is the example: misconfigured AWS settings and insufficiently defined firewall rules (a web application firewall) let an attacker access ~100 million customer records. Mitigate with configuration hardening, regular cloud-security audits, network isolation, and layered access controls.

embedded secrets

Hard-coded passwords, API keys, or credentials stored directly in source code or config files, easily accessible once the codebase is reached. The 2021 Twilio incident is the example: developers pushed API keys to a public GitHub repo, and attackers used them to access infrastructure. Mitigate with secrets-management tools (HashiCorp Vault, AWS Secrets Manager) and automated repository scanning for secrets.

outdated/unpatched and end-of-life software

Outdated/unpatched software and libraries
Contain known vulnerabilities attackers scan for with automated tools. Example: a 2021 file-transfer-appliance breach (Accellion-style) exposed millions of records at multiple clients. Mitigate with vulnerability management and automated patch management.
End-of-life (EOL) software
No longer receives vendor security updates, so new vulnerabilities stay unpatched (e.g. Windows 10 reaching EOL Oct 2025). Can also breach compliance (HIPAA, PCI DSS). Mitigate by planning timely migration and, for legacy systems that must remain, isolation, restricted access, and compensating controls.
EXAM TIPS

what to know for the exam

  • Directory service misconfig (AD over-permissioning) enables privilege escalation; attackers map it with BloodHound.
  • Unsecure configuration = insecure defaults/open ports/weak credentials; Capital One 2019 is the cloud-misconfig example.
  • Embedded secrets = hard-coded credentials/API keys (Twilio 2021); fix with Vault / AWS Secrets Manager.
  • Outdated/unpatched software has known CVEs; EOL software gets no patches at all (Windows 10 EOL Oct 2025).
  • For EOL systems that can't be replaced: isolate, restrict access, apply compensating controls.
decision rules
  • Scenario maps AD escalation paths with a tool → directory service misconfiguration (BloodHound)
  • Scenario leaves default/open settings exposed → unsecure configuration
  • Scenario finds an API key in source code → embedded secrets
  • Scenario runs software the vendor no longer patches → end-of-life software

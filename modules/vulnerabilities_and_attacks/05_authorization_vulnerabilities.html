Authorization Vulnerabilities

When access controls fail: a privileged program tricked into acting for an attacker, encryption too weak to protect data, and trusted partners turned into back doors.

overview

Authorization vulnerabilities occur when a system improperly manages access controls, letting users or processes act beyond their permissions. Three types: confused deputy (a privileged program tricked into acting for an attacker), weak ciphers (outdated/insecure cryptographic algorithms), and vulnerable third parties (compromised external services becoming a back door).

vuln 01

Confused Deputy

A higher-privileged program (the "deputy") tricked into performing actions for an attacker, bypassing authorization. E.g. a proxy honoring a crafted request.

vuln 02

Weak Ciphers

Outdated algorithms: MD5/SHA1 (collisions), DES (short key/brute force), RC4 (predictable). Use AES-256, SHA-256, TLS.

vuln 03

Vulnerable Third Parties

A compromised external service/API/partner becomes a back door via the trust relationship. Vet vendors, limit permissions, audit.

confused deputy

The confused deputy vulnerability arises when an attacker exploits a higher-privileged program (the deputy) to perform actions they couldn't do themselves. The deputy has legitimate access to restricted resources; when manipulated, it executes the attacker's request without validating its origin, bypassing authorization. Example: a proxy server with high privileges that honors a crafted request for a restricted file, effectively bypassing the attacker's access checks. Prevent by validating every request against the user's original permissions at each stage (not relying on intermediaries), using tokens/certificates to verify permissions.

weak ciphers

Weak ciphers are outdated or insecure cryptographic algorithms that fail to protect data. Weak hashing: MD5 and SHA-1 (susceptible to collision attacks). Weak encryption: DES (short 56-bit key, brute-forceable) and RC4 (structural flaws, predictable patterns). Example: a legacy banking app using DES, an attacker with the encrypted data uses cloud power and tools like Hashcat/John the Ripper to brute-force the key in hours. Fix: use strong algorithms (AES-256 encryption, SHA-256 hashing), secure transport (up-to-date TLS), and regular crypto audits.

vulnerable third parties

When an app relies on external services, APIs, or partners with granted permissions, a compromised third party becomes a back door via the trust relationship. Example: a compromised third-party payment processor lets attackers intercept/manipulate transaction data, redirect payments, steal card info, or inject malicious code. Mitigate by rigorously evaluating partners' security before integration, regular security audits and vendor due diligence, limiting access permissions per integration, using secure APIs with restricted permissions, and continuous compliance monitoring.

EXAM TIPS

what to know for the exam

  • Confused deputy = a privileged program tricked into acting for an attacker (bypasses authorization); validate original permissions at each stage.
  • Weak hashing: MD5, SHA-1 (collisions). Weak encryption: DES (short key), RC4 (predictable). Replace with SHA-256, AES-256.
  • Brute-force tools: Hashcat, John the Ripper.
  • Vulnerable third parties become back doors via trust; vet vendors and limit permissions.
  • The common thread: trust extended without sufficient validation.
decision rules
  • Scenario tricks a privileged proxy into fetching a restricted file → confused deputy
  • Scenario brute-forces DES-encrypted data → weak ciphers
  • Scenario breaches via a compromised payment processor → vulnerable third party
  • Scenario finds MD5 used for hashing → weak cipher (collision risk)

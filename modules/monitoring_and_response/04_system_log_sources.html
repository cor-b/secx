System Log Sources

Where the records live: the network gear, the endpoints (Windows event logs and Linux /var/log), the applications, and the cloud posture tools, each a window into a different layer.

overview

System log sources are logs from network infrastructure and devices. Four types: infrastructure device logs (routers, firewalls, switches), endpoint logs (Windows event logs, Linux /var/log), application logs (software events), and CSPM tools (cloud posture monitoring). A log policy ensures all necessary data is captured (defaults often omit detail).

source 01

Infrastructure Device Logs

Routers, firewalls, switches: network anomalies, malicious traffic. Defaults may omit detail, so set a log policy (on-prem + cloud).

source 02

Endpoint Logs

Windows: Security, Application, System, Setup, Forwarded Events. Linux: /var/log (auth.log, messages, cron.log, dpkg/yum).

source 03

Application Logs

Software events by severity (info/warning/error/critical). Web/database logs detect SQLi attempts, crashes, suspicious behavior.

source 04

CSPM Tools

Monitor cloud for misconfigs/compliance (AWS Security Hub). Catch open S3 buckets, overly permissive IAM.

infrastructure and endpoint logs

Infrastructure device logs record activity from routers, firewalls, and switches, revealing traffic patterns and security issues (e.g. firewall logs showing repeated access attempts from suspicious IPs). Defaults may omit detail (dropped packets, certain access logs), so a log policy spanning both on-prem and cloud (AWS, Azure) ensures comprehensive coverage.

Endpoint logs differ by OS. Windows has five notable logs: Security (authentication events, logins), Application (app errors/warnings), System (hardware/software/driver events), Setup (OS installation), and Forwarded Events (aggregated remote logs). Linux stores logs in /var/log: auth.log (authentication, like Windows Security), /var/log/messages (system-wide events), app logs (e.g. /var/log/httpd for Apache), cron.log/daemon.log (background tasks), and package-manager logs (dpkg.log for Debian, yum.log for Red Hat).

application logs and CSPM tools

Application logs capture software events, organized by severity (information, warning, error, critical) on Windows (Event Viewer) and stored in /var/log on Linux. They track HTTP requests, detect SQL injection attempts, identify slow page loads, and surface failed logins or suspicious behavior, with alerts set by severity. CSPM (Cloud Security Posture Management) tools monitor cloud environments for misconfigurations, vulnerabilities, and compliance violations; AWS Security Hub is a prevalent example, detecting issues like open S3 buckets or overly permissive IAM roles and supporting compliance reporting (GDPR, HIPAA).

EXAM TIPS

what to know for the exam

  • Four log sources: infrastructure device, endpoint, application, CSPM tools.
  • Windows endpoint logs: Security, Application, System, Setup, Forwarded Events. The Security log = authentication events.
  • Linux logs live in /var/log: auth.log (authentication), messages (system), cron.log (tasks), dpkg.log/yum.log (packages).
  • CSPM (AWS Security Hub) monitors cloud for misconfigs (open S3 buckets, permissive IAM) and compliance.
  • Default configs often omit detail, set a log policy covering on-prem and cloud.
decision rules
  • Scenario reviews firewall/router records for odd traffic → infrastructure device logs
  • Scenario checks Windows logon events → Security log (endpoint)
  • Scenario inspects Linux authentication attempts → /var/log/auth.log
  • Scenario detects an open S3 bucket misconfiguration → CSPM tool

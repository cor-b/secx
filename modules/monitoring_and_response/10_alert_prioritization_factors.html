Alert Prioritization Factors

With thousands of alerts a day, which do you handle first? Five factors decide: how severe, how damaging, how valuable the asset, how much risk remains, and how sensitive the data.

overview

Alert prioritization factors determine the urgency of alerts so the most critical threats are handled first. Five factors: criticality (severity level), impact (potential damage), asset type (value of the affected system), residual risk (risk remaining after controls), and data classification (sensitivity of the data).

factor 01

Criticality

Severity level (warning → low → medium → high → critical). SIEMs rank by predefined rules/thresholds.

factor 02

Impact

Potential damage if unmitigated (data exfiltration > minor misconfig). Assessed via IDS/DLP.

factor 03

Asset Type

Value of the affected system (production server/critical DB > minor internal system).

factor 04

Residual Risk & Data Classification

Risk left after controls (bypassed MFA = high); sensitivity of data (PHI/financial = higher priority).

the five factors

Alert criticality
The severity level (warning, low, medium, high, critical). A low alert (failed login) vs a critical one (ransomware encrypting a database). SIEMs (Splunk, QRadar) rank alerts using predefined rules/thresholds.
Alert impact
The potential damage if not addressed. An attempted data exfiltration from a customer database has far higher impact than a noncritical misconfiguration. Assessed via IDS and DLP monitoring scope/consequences.
Asset type
The value of the affected system. Alerts on high-value assets (production servers, critical databases, cloud infrastructure) outrank those on less important systems.
Residual risk
The risk remaining after controls are applied. If an alert shows an attacker bypassed MFA via phishing, residual risk is high because a key control was defeated. Assess with frameworks like NIST or ISO 27001.
Data classification
The sensitivity of the data involved. Alerts touching classified/confidential data (PHI, financial records) get higher priority because a breach carries legal/financial/reputational damage. SIEMs correlate alerts with data sensitivity to raise priority.

These combine: an alert on a critical financial server storing classified data is prioritized over one on a low-impact system, because criticality, impact, and data sensitivity all stack.

EXAM TIPS

what to know for the exam

  • Five factors: criticality (severity), impact (damage), asset type (value), residual risk (after controls), data classification (sensitivity).
  • Criticality = severity level; impact = potential damage, distinct but related.
  • Residual risk = risk left after controls; a bypassed control (MFA) raises it.
  • Data classification raises priority for PHI/financial/classified data.
  • Factors stack: a critical, high-impact alert on a high-value asset with sensitive data is top priority.
decision rules
  • Scenario ranks alerts by severity level → criticality
  • Scenario weighs potential damage of the threat → impact
  • Scenario prioritizes an alert on a production database → asset type
  • Scenario notes MFA was bypassed, leaving exposure → residual risk
  • Scenario prioritizes an alert touching PHI → data classification

Operating System Security

Enforcing security policy from inside the kernel. SELinux applies mandatory access control through labeled security contexts, deciding every access by policy, not by file permissions.

overview

Operating system security protects the OS from unauthorized access, malware, and other threats. The focus here is SELinux (Security-Enhanced Linux), a module built into the Linux kernel that enforces mandatory access control. It intercepts every access attempt at the kernel level and checks it against predefined policies using security contexts (labels of user, role, type, and level), so access is decided by policy rather than standard file permissions.

how SELinux works

SELinux is integrated directly into the Linux kernel and enforces strict policies that control access to files, processes, and other resources. Operating at the kernel level, it intercepts every access attempt and checks it against policy before allowing or denying the action. When a web server process tries to read a configuration file, SELinux compares the security context of the process and the file; if policy does not explicitly allow it, the attempt is denied, blocked, and logged. Only actions permitted by policy are allowed, which greatly reduces the risk of unauthorized access.

SELinux uses mandatory access control (MAC): rules dictate what users, applications, and processes may do regardless of their traditional permissions. This is implemented through labels and tags assigned as part of each security context.

security contexts

A security context is the set of security attributes assigned to every file, process, and resource. It has four parts:

user
The SELinux user identity (e.g. unconfined_u). unconfined_u represents a user not restricted by SELinux policies, with fewer constraints.
role
The role of the object (e.g. object_r). object_r is the common role for files, directories, and devices: a regular object with no elevated role.
type
The type label that drives access decisions (e.g. user_tmp_t, httpd_sys_content_t). A file's type determines which processes can touch it. httpd_sys_content_t restricts a file to web-server (HTTP daemon) access only.
level
The sensitivity level (e.g. s0). s0 is the default, lowest sensitivity level.

Because access decisions come from these labels rather than user permissions, even if an attacker exploits a vulnerability in (say) a database server, SELinux can prevent the process from reaching files outside its own data directory.

key commands

  • sestatus: shows whether SELinux is enabled and its current mode (e.g. enforcing)
  • ls -Z: displays the security context of files
  • chcon -t <type>: changes a file's type context (e.g. to httpd_sys_content_t)
  • setsebool: enables or disables policy settings by toggling Boolean options
  • semanage: manages policies, such as allowing a new application access to a directory
  • audit2allow: builds custom rules by analyzing logged denials, generating policy to permit previously blocked actions

In enforcing mode, SELinux actively blocks any access not permitted by policy. For example, after a file is relabeled httpd_sys_content_t, a standard user trying to cat it would be denied, because that type is reserved for the HTTP daemon.

EXAM TIPS

what to know for the exam

  • SELinux is a kernel-integrated module enforcing mandatory access control (MAC) on Linux.
  • It decides access by security context (labels), not by traditional file permissions, every access is checked against policy.
  • Security context = user, role, type, level. The type label drives most access decisions.
  • Enforcing mode blocks unpermitted access; SELinux logs the denial.
  • Key tools: sestatus (status), chcon (change context), setsebool (toggle booleans), semanage (manage policy), audit2allow (build rules from denials).
decision rules
  • Scenario involves kernel-level mandatory access control on Linux → SELinux
  • Scenario involves access decided by labels (user/role/type/level) not file permissions → SELinux security context
  • Scenario involves confining a compromised web server to its own files → SELinux policy (type enforcement)
  • Scenario involves generating an SELinux rule from logged denials → audit2allow
  • Scenario involves changing a file's SELinux type → chcon

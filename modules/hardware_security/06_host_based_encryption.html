Host-Based Encryption

Encryption managed by the operating system to protect data at rest. Each major OS ships its own full-disk tool, all built on AES.

overview

Host-based encryption is encryption managed and controlled by the local operating system, used to secure data at rest: the contents of a storage device when the OS is not running. It prevents unauthorized access if someone gains physical access to a device. Each OS provides its own full-disk tool: BitLocker on Windows, LUKS/Cryptsetup on Linux, and FileVault on macOS, all using AES encryption.

windows

BitLocker

Built-in full-disk encryption using AES. Often relies on a TPM to manage keys securely within the device.

linux

LUKS / Cryptsetup

Cryptsetup with the Linux Unified Key Setup (LUKS) manages keys and encrypts disk partitions with AES.

macos

FileVault

Apple's built-in full-disk encryption using AES, protecting data when the Mac is powered off or unattended.

data at rest

Host-based encryption protects data at rest, the data stored on a device when the operating system is not running. This is essential for preventing unauthorized access to sensitive information, especially if a device is lost, stolen, or physically accessed. Because the OS manages the encryption, decryption only happens once the correct credentials (passphrase, key, or TPM-backed key) are supplied.

the tools

BitLocker (Windows)

BitLocker is built into Windows and uses AES to encrypt the entire disk. It often relies on a TPM to manage encryption keys securely within the device, so even if the computer is stolen, the data stays inaccessible without the correct credentials.

LUKS / Cryptsetup (Linux)

On Linux, Cryptsetup with the Linux Unified Key Setup (LUKS) provides full-disk encryption comparable to BitLocker, using AES. LUKS acts as the layer that manages encryption keys and ensures data is encrypted before it is written and decrypted when read, keeping data at rest unreadable without the correct passphrase or key.

FileVault (macOS)

FileVault offers full-disk encryption built into macOS, using the same AES standard to protect data on Apple devices when they are shut down or left unattended.

setting up LUKS on Linux

The typical Cryptsetup + LUKS workflow to encrypt a disk (for example /dev/sdb):

  1. Identify the target disk with lsblk and unmount it (umount /dev/sdb).
  2. Optionally wipe it with random data (dd if=/dev/urandom of=/dev/sdb).
  3. Initialize encryption headers: cryptsetup luksFormat /dev/sdb (set a passphrase).
  4. Open the encrypted volume to a mapped device: cryptsetup open /dev/sdb encrypted_disk.
  5. Create a file system on it: mkfs.ext4 /dev/mapper/encrypted_disk.
  6. Mount it (mount /dev/mapper/encrypted_disk /mnt); when done, unmount and cryptsetup close encrypted_disk to take it offline while keeping it encrypted.

EXAM TIPS

what to know for the exam

  • Host-based encryption = OS-managed encryption protecting data at rest (device powered off / OS not running).
  • Match the tool to the OS: BitLocker (Windows), LUKS/Cryptsetup (Linux), FileVault (macOS). All use AES.
  • BitLocker often relies on a TPM to store and manage encryption keys, tying together two hardware-security topics.
  • LUKS is the key-management standard; Cryptsetup is the tool that uses it on Linux.
  • Host-based encryption protects against physical theft/access; it does not protect data while the system is running and unlocked.
decision rules
  • Scenario involves full-disk encryption on Windows with TPM-managed keys → BitLocker
  • Scenario involves encrypting a Linux disk partition with a passphrase → LUKS / Cryptsetup
  • Scenario involves built-in full-disk encryption on a Mac → FileVault
  • Scenario involves protecting a stolen laptop's stored data → host-based (full-disk) encryption
  • Scenario names cryptsetup luksFormat → LUKS on Linux

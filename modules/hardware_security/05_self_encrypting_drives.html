Self-Encrypting Drives

Storage that encrypts itself. An onboard engine protects data transparently, with keys that never leave the drive, no software, no CPU overhead, no user effort.

overview

Self-encrypting drives (SEDs) are storage devices (HDD or SSD) with built-in encryption that automatically encrypt data on write and decrypt on read, with no user interaction or additional software. An onboard encryption engine handles everything transparently, and the keys are managed inside the drive and never leave the hardware, protecting them from attacks on the OS. SEDs are easy to deploy and offer consistent performance, but enterprise-grade models that meet strict standards can be costly.

how SEDs work

An SED integrates encryption algorithms and secure key management directly into the drive's hardware. Because the onboard engine operates transparently, users notice no difference in how the drive behaves. The encryption keys are managed within the drive itself and never leave the hardware, so even if the computer's operating system is compromised, the keys (and the drive's contents) stay protected. Deployment is simple: no additional configuration or software is required to enable the encryption.

advantages

  • Ease of use: encryption happens automatically, with no special knowledge required from the user
  • Consistent performance: the drive's own hardware handles encryption/decryption, not the CPU, avoiding the slowdowns of software-based encryption
  • Strong breach protection: keys are stored within the drive and never exposed to the rest of the system, making the encrypted data very hard to access

drawbacks

  • Cost: consumer SEDs are affordable, but enterprise-grade models that meet strict standards like FIPS 140-2 and IEEE 1667 can be much more expensive
  • Hard to upgrade: because security is hardware-based, adopting new encryption standards may require replacing the drive rather than updating it

EXAM TIPS

what to know for the exam

  • SEDs encrypt/decrypt transparently with an onboard engine, no user interaction, no extra software.
  • Keys are managed inside the drive and never leave the hardware, so an OS compromise does not expose them.
  • Performance stays consistent because the drive's hardware (not the CPU) does the work, an advantage over software encryption.
  • Enterprise SEDs meet FIPS 140-2 and IEEE 1667; these standards raise security and cost.
  • Being hardware-based, SEDs can require replacement (not just an update) to adopt new encryption standards.
decision rules
  • Scenario needs transparent disk encryption with no performance hit and no software → self-encrypting drive
  • Scenario stresses keys never leaving the storage hardware → SED
  • Scenario names FIPS 140-2 or IEEE 1667 drive requirements → enterprise SED
  • Scenario needs encryption that offloads work from the CPU → SED (vs software encryption)

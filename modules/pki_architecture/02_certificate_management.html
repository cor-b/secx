Certificate Management

Managing certificates across their lifecycle: the types you'll choose between, the extension fields that define how they're used, and the file formats they ship in.

overview

Certificate management covers issuing, renewing, revoking, and managing digital certificates through their lifecycle. Two concepts anchor it: certificate types (TLS, client, code-signing, wildcard, SAN, each for a different job) and certificate extensions (X.509 fields defining intended usage, marked critical or non-critical). Knowing the validation tiers (DV/OV/EV) and the wildcard-vs-SAN distinction is the exam core.

certificate types

A digital certificate is a signed document binding a public key to an identity (person, server, device, software), issued by a CA after verification. The main types:

SSL/TLS certificate
Secures data between browser and web server (HTTPS) and authenticates the server. Three validation tiers: DV (domain validation, basic, just proves domain control), OV (organization validation, verifies the org), EV (extended validation, most rigorous, highest trust).
Client certificate
Authenticates users and devices, replacing password-based auth with cryptographic verification. Common in enterprise access control and secure email (validating the sender).
Code-signing certificate
Verifies software authenticity and integrity, proving code is from a trusted source and untampered.
Wildcard certificate
Secures unlimited subdomains of one domain, e.g. *.securityx.com covers www, mail, etc. Simplifies management, but if compromised, every subdomain is at risk.
SAN (Subject Alternative Name) certificate
Secures multiple distinct domains under one certificate (e.g. example.com and example.org). Unlike a wildcard (one base domain, many subdomains), a SAN cert covers different domain names. Also called a multi-domain certificate.

certificate extensions

Extensions are extra X.509 fields describing a certificate's purpose and restrictions. Each is marked critical (an application must understand and process it or reject the certificate) or non-critical (can be ignored if unrecognized). Common ones:

  • Key Usage — primary functions (digital signature, key encipherment, certificate signing)
  • Extended Key Usage (EKU) — specific roles (client auth, server auth, code signing, email protection)
  • Subject Alternative Name — multiple domains/IPs/emails on one cert
  • Basic Constraints — whether the cert is an end-entity or a CA cert
  • Authority Key Identifier / Subject Key Identifier — link a cert to its issuer for chain validation
  • Certificate Policies — policies the cert was issued under
  • CRL Distribution Points — where to find the revocation list

encoding and file formats

Certificates follow the X.509 standard but are encoded (X.690 rules) as BER (Basic), CER (Canonical), or DER (Distinguished, strict and predictable, ideal for crypto). Common file formats:

.pemcerts, CA certs, keysmost common; flexible, often DER-based
.p12 / PKCS#12certs + private key (encrypted)bundles credentials for secure transport

EXAM TIPS

what to know for the exam

  • Wildcard cert = many subdomains of ONE domain (*.example.com). SAN cert = multiple DIFFERENT domains. This distinction is commonly tested.
  • TLS validation tiers: DV (domain only) < OV (organization verified) < EV (most rigorous, highest trust).
  • A critical extension MUST be understood or the certificate is rejected; non-critical can be ignored.
  • Certificates use the X.509 standard. .p12/.pfx can hold private keys; .p7b holds only the chain (no private key).
  • Don't memorize every file format; recognize they're all PKI-related and pick accordingly.
decision rules
  • Scenario secures www, mail, and shop under one domain → wildcard certificate
  • Scenario secures example.com and example.org together → SAN (multi-domain) certificate
  • Scenario needs the highest browser trust with org name shown → EV certificate
  • Scenario needs to move a cert plus its private key between systems → .p12 / .pfx
  • Scenario rejects a cert because an extension can't be processed → critical extension

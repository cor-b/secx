VPN Architecture

Encrypted tunnels that extend access controls beyond the network edge: connecting a single device, two whole networks, or maintaining an always-active secure link.

overview

VPN architecture extends access controls beyond traditional network boundaries by encrypting traffic between endpoints. Three models matter: client-server connects an individual device to a central server for remote access, site-to-site connects entire networks at the router level so offices act as one logical network, and always-on maintains a continuous, automatically established connection that enforces policy regardless of location. Encryption relies on IPsec or TLS.

model 01

Client-Server

Encrypted tunnel between one device and a central server. Ideal for remote access. Uses IPsec or TLS; server authenticates the client by credentials or certificate.

model 02

Site-to-Site

Connects entire networks at the router level, not individual devices. Branch offices act as one logical LAN. Typically IPsec.

model 03

Always-On

Continuous connection that auto-establishes on device startup. Used in government/healthcare. Often pairs with MFA and NAC.

client-server VPN

A client-server VPN connects a single client device to a central server through an encrypted tunnel, ideal for remote access from any location. Encryption uses IPsec or TLS. The server authenticates the client with username/password or digital certificates, and once authenticated the client reaches internal resources (files, applications, databases) as if physically on the corporate network.

site-to-site VPN

A site-to-site VPN connects two or more entire networks, such as a headquarters and its branch offices. It operates at the router level, so the tunnel is established between routers rather than individual devices. The standard protocol is IPsec, which encrypts and authenticates the packets traveling between sites. All devices in one network can then communicate securely with the other, making the two appear as one local network, useful for sharing file servers, intranet apps, and internal services across offices.

always-on VPN

An always-on VPN provides a continuous, uninterrupted connection that establishes automatically whenever the device powers on, regardless of where the user is or what network they join. Common in security-critical environments like government and healthcare, it keeps all device-to-network traffic encrypted using IPsec or TLS. Always-on VPNs frequently integrate MFA to verify identity on each connection and NAC to check the device's security posture before granting full access.

reading an OpenVPN config

An always-on OpenVPN client configuration for an organization shows the key directives that keep the tunnel secure and persistent:

client                  # this is a VPN client, not a server
dev tun                # create a tun (network-layer) tunnel interface
remote vpn.host 1194   # server address, protocol, and port
resolve-retry infinite # keep retrying DNS resolution on failure
nobind                 # do not bind to a fixed local port
persist-key            # keep encryption keys across reconnects
persist-tun            # keep the tunnel interface active across drops
ping-restart 60        # auto-reconnect if the link drops
user nobody            # drop privileges after connecting
group nogroup          # run with a low-privilege group
  • dev tun creates a network-layer tunnel interface that routes IP packets through the VPN
  • persist-key and persist-tun keep the keys and interface intact through network disruptions
  • ping-restart automatically reconnects if the connection drops, supporting the always-on model
  • user nobody and group nogroup drop privileges after connecting, applying least privilege to shrink the attack surface if the client is compromised

EXAM TIPS

what to know for the exam

  • Client-server VPN = one device to a central server. Remote access. IPsec or TLS.
  • Site-to-site VPN = network to network at the router level. Connects offices. Almost always IPsec.
  • Always-on VPN = continuous, auto-connecting. Government/healthcare. Pairs with MFA and NAC.
  • Site-to-site operates router-to-router; client-server and always-on operate at the individual device.
  • Dropping privileges (user nobody, group nogroup) after connecting applies least privilege and reduces the attack surface.
decision rules
  • Scenario involves a remote employee securely reaching internal apps from a laptop → client-server VPN
  • Scenario involves linking two branch office networks over the internet → site-to-site VPN
  • Scenario involves a device that stays connected and re-secures automatically as the user moves → always-on VPN
  • Question asks which VPN operates at the router level → site-to-site
  • Config drops to user nobody after connecting → least privilege to limit compromise

Asymmetric Cryptography

A public/private key pair removes the need to share a secret in advance, solving symmetric's key-distribution problem and enabling signatures and trust.

overview

Asymmetric (public-key) cryptography uses a key pair: a widely-shared public key and a secret private key. One key encrypts, the other decrypts, and the same key can't do both. It powers secure communication, digital signatures, and key exchange. It is computationally heavier and suited only to small data, so in practice it is used to securely exchange a symmetric key, which then encrypts the bulk traffic (the basis of SSL/TLS). Code signing is a key application.

concept 01

Key Pair

Public key shared freely, private key kept secret. One encrypts, the other decrypts. No pre-shared secret needed.

concept 02

Hybrid Use

Asymmetric exchanges a symmetric key, then symmetric encrypts the bulk data. Combines asymmetric's trust with symmetric's speed (SSL/TLS).

concept 03

Code Signing

Developer signs code with their private key; users verify with the public key, confirming the code is genuine and unaltered.

how it works

Two separate keys: a public key for encryption and a private key for decryption. Even if the public key is shared widely, data stays secure because only the matching private key can decrypt it. Common algorithms: RSA, DSA, Diffie-Hellman, ElGamal, and ECC (covered next lesson).

Trade-offs: asymmetric encryption is far more computationally intensive than symmetric, making it slower and suited only to small data like keys and signatures, and the security of the private key is everything, compromise it and the whole system falls.

the hybrid approach

The strengths of both families combine in real systems: asymmetric encryption securely delivers a symmetric session key, then symmetric encryption handles the bulk data efficiently. For example, a sender encrypts a freshly-created symmetric key with the recipient's public key; only the recipient's private key can unwrap it, so even if the exchange is intercepted, the session key stays safe. This is exactly the SSL/TLS key-exchange model.

code signing

Code signing uses asymmetric crypto to guarantee software authenticity and integrity. The developer signs the code with their private key, creating a digital signature, a seal showing the code is from a trusted source and unaltered. When a user downloads it, their system verifies the signature with the developer's public key: if it checks out, the software is genuine. Like a wax seal on a letter, it proves the code hasn't been tampered with, protecting both the developer's reputation and the user's security.

EXAM TIPS

what to know for the exam

  • Asymmetric = public/private key pair. One key encrypts, the other decrypts; the same key cannot do both.
  • It solves symmetric's key-distribution problem (no pre-shared secret) but is slow and only for small data.
  • Hybrid model: asymmetric exchanges a symmetric key, symmetric encrypts the bulk. This is how SSL/TLS works.
  • To send a confidential message, encrypt with the recipient's PUBLIC key (only their private key decrypts).
  • Code signing: sign with the developer's PRIVATE key, verify with the PUBLIC key, confirms authenticity and integrity.
decision rules
  • Scenario needs to exchange a key securely with someone you've never met → asymmetric cryptography
  • Scenario describes wrapping a session key in the recipient's public key, then bulk-encrypting → hybrid (SSL/TLS)
  • Scenario verifies downloaded software is genuine and unaltered → code signing
  • Scenario stresses slow performance limiting use to small data → asymmetric encryption

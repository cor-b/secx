Digital Signatures

Sign a hash of your message with your private key. Anyone with your public key can then prove it's really from you and that nothing changed, but the message itself stays readable.

overview

A digital signature proves the authenticity and integrity of a message, and provides non-repudiation. The sender hashes the message, then encrypts that hash digest with their private key, that encrypted hash is the signature. The recipient decrypts it with the sender's public key and compares hashes. Signatures do not provide confidentiality; for that, encrypt the message with the recipient's public key.

provides

Integrity

Hashing the message and comparing digests proves the content was not altered in transit.

provides

Authenticity

Only the sender's private key could have produced a signature their public key decrypts, proving who signed it.

provides

Non-Repudiation

Because only the sender holds their private key, they cannot later deny having signed.

how a signature works

Walk through Alice sending Bob a signed email (both have key pairs):

  1. Alice's client hashes her message (e.g. with SHA-256), producing a unique hash digest.
  2. It encrypts that digest with Alice's private key; the result, appended to the email, is the digital signature. (The message body itself is not encrypted, signatures are about authenticity/integrity, not confidentiality.)
  3. Bob's client decrypts the signature with Alice's public key, recovering the original hash digest.
  4. Bob's client independently hashes the received message with the same algorithm and compares. If the two digests match, the message is unaltered (integrity) and could only have come from Alice (authenticity + non-repudiation).

non-repudiation

Non-repudiation means the signer cannot later deny signing. Only Alice could have encrypted the digest with her private key, and Bob proves it by successfully decrypting with her public key, since in asymmetric crypto a public key only decrypts what its matching private key encrypted.

adding confidentiality

Signatures alone leave the message readable. To also keep it private, Alice encrypts the message with Bob's public key, so only Bob's private key can open it. Combine all three and you have encrypting and digitally signing:

  • Confidentiality → encrypt the message with the recipient's public key
  • Integrity → hash the message
  • Authenticity / non-repudiation → encrypt the hash with the sender's private key
EXAM TIPS

what to know for the exam

  • A digital signature = hash the message, then encrypt the hash with the SENDER'S PRIVATE key. Verify with the sender's public key.
  • Signatures provide integrity, authenticity, and non-repudiation, NOT confidentiality.
  • For confidentiality, encrypt the message with the RECIPIENT'S PUBLIC key.
  • Key-direction cheat: sign = sender's private key; confidentiality = recipient's public key.
  • Non-repudiation works because only the signer holds their private key.
decision rules
  • Scenario needs to prove who sent a message and that it's unchanged → digital signature
  • Scenario asks which key signs → sender's private key
  • Scenario asks which key makes a message confidential → recipient's public key
  • Scenario stresses the sender can't deny sending → non-repudiation
  • Scenario needs confidentiality AND proof of origin → encrypt and digitally sign

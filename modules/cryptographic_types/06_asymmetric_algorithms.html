Asymmetric Algorithms

Four algorithms, four jobs: RSA for encryption and signatures, DSA for signatures, Diffie-Hellman for key exchange, and ECC for the same strength with far smaller keys.

overview

The asymmetric algorithms differ by the hard math they rely on and the job they do. RSA (factoring large primes) handles encryption, key exchange, and signatures. DSA is signatures only. Diffie-Hellman is key exchange only. ECC uses elliptic curves to match RSA's security with much smaller keys, ideal for mobile and IoT.

algorithm 01

RSA

Based on factoring large primes. Encryption, key exchange, and signatures. Keys 1024–4096-bit. The workhorse of SSL/TLS.

algorithm 02

DSA

Digital Signature Algorithm. Signatures only (no encryption). Fast to sign, slower to verify than RSA.

algorithm 03

Diffie-Hellman

Key exchange only. Two parties derive a shared secret over an insecure channel. No authentication on its own.

algorithm 04

ECC

Elliptic Curve Cryptography. RSA-level security with much smaller keys. Efficient for mobile/IoT. Variants: ECDSA, ECDH.

the algorithms

RSA (Rivest-Shamir-Adleman)
Security rests on the difficulty of factoring the product of two large primes. Multiplying primes is easy; factoring the result is infeasible at large sizes. Keys 1024–4096-bit (bigger = stronger but slower). Used for SSL/TLS, key exchange, and digital signatures.
DSA (Digital Signature Algorithm)
Built solely for digital signatures, proving authenticity and integrity, not encryption. Relies on the discrete logarithm problem. Faster signature generation than RSA but slower verification, a classic crypto trade-off.
Diffie-Hellman (DH)
Lets two parties derive the same shared secret over an insecure channel without ever sending the secret. Think of mixing paint colors: a shared public color plus each party's private color produces a common mix that an eavesdropper can't un-mix. But DH has no authentication, so it's vulnerable to on-path (man-in-the-middle) and logjam attacks; use 2048-bit+ keys or Elliptic-Curve Diffie-Hellman, and authenticate with digital certificates.
ECC (Elliptic Curve Cryptography)
Security from the elliptic curve discrete logarithm problem; easy one way, infeasible to reverse. Matches RSA security with far smaller keys: a 256-bit ECC key ≈ a 3072-bit RSA key. Low overhead suits mobile/IoT. ECDSA signs data; ECDH does key exchange.

EXAM TIPS

what to know for the exam

  • RSA = factoring large primes; does encryption, key exchange, and signatures. DSA = signatures only. DH = key exchange only. ECC = small keys, big security.
  • ECC's headline fact: a 256-bit ECC key gives roughly the same security as a 3072-bit RSA key, hence its use on mobile/IoT.
  • Diffie-Hellman provides NO authentication by itself, so it's vulnerable to man-in-the-middle and logjam; pair it with certificates and use 2048-bit+ or ECDH.
  • DSA: faster to sign, slower to verify than RSA.
  • ECDSA = ECC signatures; ECDH = ECC key exchange.
decision rules
  • Scenario needs strong security with tiny keys on a mobile/IoT device → ECC
  • Scenario is two parties agreeing a shared secret over an open channel → Diffie-Hellman
  • Scenario needs encryption AND signatures from one widely-used algorithm → RSA
  • Scenario is signatures only / authenticity and integrity → DSA
  • Scenario warns of a man-in-the-middle during key agreement → unauthenticated Diffie-Hellman

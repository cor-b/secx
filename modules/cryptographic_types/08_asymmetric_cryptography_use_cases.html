Asymmetric Cryptography Use Cases

Where public/private key pairs show up in practice: proving identity with certificates, logging in without passwords, and locking down email end to end.

overview

Three common asymmetric use cases: certificate-based authentication (digital certificates from a trusted CA validate identity), passwordless authentication (cryptographic keys and biometrics replace passwords), and secure email (asymmetric encryption keeps messages confidential end to end, via S/MIME).

use case 01

Certificate-Based Auth

Digital certificates from a trusted CA prove identity. Client auth (often via NAC), server auth (HTTPS), and SSO with smart cards.

use case 02

Passwordless Auth

Cryptographic keys plus ownership (token/phone) or biometric factors replace passwords. Strongest when combined with MFA.

use case 03

Secure Email

S/MIME signs and encrypts each message for confidentiality, integrity, authentication, and non-repudiation.

certificate-based authentication

Identities are verified using digital certificates issued by a trusted certificate authority (CA) within a PKI. In client authentication, a server confirms a connecting device/user is pre-authorized via its certificate, commonly managed through Network Access Control (NAC). In server authentication, a client validates that a server is genuine, every HTTPS connection uses the server's certificate to confirm you're on the real site. In the enterprise, this is often woven into single sign-on (SSO): a user authenticates once (e.g. a smart card holding a certificate) and reaches many resources without re-entering credentials.

passwordless authentication

Passwordless auth lets users in without a knowledge-based secret, using public-key cryptography (a private key on the user's device) plus two factor types: ownership (something you have, smartphone, smart card, hardware token) and biometric (something you are, fingerprint, face). Benefits: stronger security (no passwords to phish or brute-force), better UX, lower IT cost. Risks: higher setup cost, training, and single points of failure, including biometric impersonation. Apple's figures illustrate the strength gradient: a 4-digit PIN is ~1 in 10,000, Touch ID ~1 in 50,000, Face ID ~1 in 1,000,000, and combining factors with MFA is stronger than any single one.

secure email

Secure email delivers confidentiality, integrity, authentication, and non-repudiation: encrypt the message with the recipient's public key (confidentiality), hash it (integrity), and sign the hash with the sender's private key (authentication + non-repudiation). The dominant standard is S/MIME (Secure/Multipurpose Internet Mail Extensions), built on PKCS, which secures individual messages rather than just the connection. Each user has a CA-signed certificate (public key) and a private key; clients like Outlook, Apple Mail, and Gmail support it. A caveat: because contents are encrypted, email security filters can't inspect them, so S/MIME can be abused to smuggle malware past boundary defenses, requiring extra controls in corporate use.

EXAM TIPS

what to know for the exam

  • Certificate-based auth uses CA-issued digital certificates. Server auth = HTTPS proving the site is real; client auth often via NAC; ties into SSO/smart cards.
  • Passwordless auth uses cryptographic keys plus ownership or biometric factors. Watch for biometric impersonation; MFA beats any single factor.
  • Apple strength gradient: PIN 1/10,000 → Touch ID 1/50,000 → Face ID 1/1,000,000.
  • S/MIME secures individual email messages (not just the connection) and provides confidentiality, integrity, authentication, and non-repudiation.
  • S/MIME downside: encrypted messages can hide malware from security filters.
decision rules
  • Scenario validates a user/device by a CA-issued certificate → certificate-based authentication
  • Scenario logs users in with a hardware token or fingerprint, no password → passwordless authentication
  • Scenario encrypts and signs individual emails → secure email (S/MIME)
  • Scenario uses a smart-card certificate to reach many apps after one login → certificate-based auth with SSO

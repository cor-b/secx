Attack Surface Management

Identify, reduce, and continuously monitor all potential attack paths into a system. The fewer entry points an attacker can reach, the smaller the risk.

overview

Attack surface management uses four techniques in combination: hardening (eliminate unnecessary exposure), defense-in-depth (layer controls so one failure does not compromise everything), vulnerability management (scan and remediate weaknesses continuously), and legacy component management (plan for systems that no longer receive security updates).

device hardening

Hardening reduces a system's attack surface by removing or disabling everything that is not strictly required for the system to perform its function. It applies to servers, endpoints, mobile devices, and network infrastructure alike.

hardening actions

  • Disable unused services and close unused ports — every running service is a potential entry point
  • Remove unnecessary applications and packages
  • Rename or disable default accounts (default credentials are widely known and frequently targeted)
  • Install and configure required security software: antivirus, host-based firewall, log collection agent
  • Apply encryption to protect data on the device
  • Apply group policies or configuration baselines to enforce strict, consistent security settings
  • Patch all software regularly to close known vulnerabilities

hardware security features

  • UEFI (Unified Extensible Firmware Interface): modern firmware standard that enables Secure Boot, preventing unauthorized code from loading during startup
  • TPM (Trusted Platform Module): dedicated hardware chip that stores cryptographic keys and supports device attestation and disk encryption (e.g., BitLocker)
  • HSM (Hardware Security Module): physical device for generating, storing, and managing cryptographic keys at hardware speed, used in high-assurance environments

service hardening reference

The principle: stop the service, prevent it from restarting, then mask it to block both automatic and manual restarts. Each step is more restrictive than the last.

CommandEffect
sudo systemctl stop <service> Stops the running service immediately
sudo systemctl disable <service> Prevents the service from starting automatically at boot (can still be started manually)
sudo systemctl mask <service> Links the service to /dev/null, blocking both automatic and manual starts — the most restrictive option

defense-in-depth

Defense-in-depth layers independent security controls so that if one control fails or is bypassed, others continue to protect the system. No single control is relied upon as the sole line of defense.

LayerControlWhat it protects against
1 Firewall Filters inbound and outbound traffic using security rules; blocks unauthorized external access at the network perimeter
2 IDS Monitors network activity for unusual behavior; alerts administrators to threats that bypass the firewall
3 Encryption in transit Secures data moving across the network; intercepted traffic is unreadable without the decryption key
4 Endpoint protection Antivirus and anti-malware tools protect individual devices from threats that enter through non-network vectors
5 Encryption at rest Secures stored data; unauthorized access to the physical or logical storage does not yield readable data

Physical security (locking server rooms, controlling physical access) complements every software layer. Each control addresses a different threat vector; together they form a resilient posture.

vulnerability management

Vulnerability management is the ongoing process of identifying, prioritizing, and remediating security weaknesses before they can be exploited. It is continuous — not a one-time activity.

  • Scan regularly with tools like Nessus, OpenVAS, or Qualys to discover open ports, outdated software, misconfigured firewalls, and weak protocols
  • Apply patches promptly — if a vulnerability is discovered in a widely used service (e.g., Apache HTTP Server), a patch should be deployed before attackers can exploit it
  • Use compliance benchmarks as scanning targets: CIS (Center for Internet Security) benchmarks and DoD STIGs (Security Technical Implementation Guides) define hardened configuration baselines
  • Use SCAP compliance checkers to automatically monitor system configurations against those benchmarks
  • Beyond patching: remove unnecessary services, close unused ports, disable default accounts — the same actions as hardening, applied continuously as the environment changes

legacy components

Legacy systems are those at or past their End of Life (EOL) or End of Support (EOS) dates. Once a vendor stops issuing security updates, any newly discovered vulnerability in that system will remain unpatched indefinitely.

End of Life (EOL)
The vendor no longer sells or develops new features for the product. At EOL, the product is no longer commercially available, but the vendor may still issue security patches. The primary concern is that development resources are no longer invested in the product.
End of Support (EOS)
The vendor no longer issues any patches, security updates, or technical support. EOS is the critical security milestone. Any vulnerability discovered after the EOS date will go unpatched. Attackers actively target known EOL/EOS systems because exploitation carries no risk of being closed by a vendor patch. Example: Windows Server 2019 reaches EOS in 2029 — organizations still running it after that date face permanent unpatched vulnerabilities.

Organizations must track the lifecycle of every component in their environment and plan upgrades or replacements before EOS is reached. Legacy systems left in place after EOS create persistent weak points regardless of how well the rest of the environment is secured.

exam tips

key takeaways

  • Hardening = reduce attack surface by removing what is not needed. Disabling unused services, closing unused ports, removing unnecessary apps, renaming default accounts. The goal is minimum required functionality.
  • Defense-in-depth = multiple independent layers. If the firewall fails, the IDS still detects; if the IDS misses something, endpoint protection still acts. No single failure breaks all defenses.
  • EOS (End of Support) is the critical risk date — not EOL. EOL means no more development. EOS means no more patches. After EOS, any new vulnerability stays open forever. Prioritize EOS in exam scenarios about legacy risk.
  • Masking is more restrictive than disabling. Disabling prevents automatic start at boot but allows manual start. Masking (linking to /dev/null) prevents both automatic and manual start. Know this distinction — it may appear in a scenario about hardening Linux services.
  • UEFI enables Secure Boot; TPM stores cryptographic keys. These are hardware-level hardening features. Know them by name and function.
  • CIS benchmarks and DoD STIGs are hardened configuration standards. SCAP compliance checkers automate verification against these benchmarks. These may appear in vulnerability management scenarios.
quick decision rules
  • Disabling unused services, ports, and accounts → Hardening
  • Multiple independent security controls that each address a different threat → Defense-in-depth
  • Regular scanning plus patching plus closing unused ports → Vulnerability management
  • System with no more vendor security patches available → End of Support (EOS) / legacy component
  • Prevent service from starting automatically at boot → systemctl disable
  • Prevent service from starting manually OR automatically → systemctl mask (links to /dev/null)
  • Hardware chip for secure boot and disk encryption key storage → TPM (Trusted Platform Module)
  • Hardened configuration baselines from CIS or DoD → CIS benchmarks / DoD STIGs

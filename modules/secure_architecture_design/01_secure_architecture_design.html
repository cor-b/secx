Secure Architecture Design

Given a scenario, integrate controls into the design of a secure architecture by protecting data across its lifecycle, managing complex environments, and continuously reducing exposure to threats.

overview

OBJ 2.3 covers how security controls are built into architecture from the start, not bolted on afterward. Eight topic areas anchor this objective: data states, data classification, data labeling and tagging, data loss prevention, hybrid infrastructure, third-party integrations, attack surface management, and control effectiveness.

The common thread is that protection must follow the data wherever it lives, moves, or gets processed, and architecture decisions directly determine how consistently that protection can be enforced.

eight topic areas

  1. Data States Data exists in three states: at rest (stored), in transit (moving), and in use (being processed). Each state requires a distinct protection method.
  2. Data Classification Organizing data into sensitivity tiers (public, internal, confidential, restricted) determines what security controls and handling procedures apply to it.
  3. Data Labeling and Tagging Labels indicate classification and sensitivity. Tags add machine-readable metadata such as project names or regulatory associations, enabling automated enforcement.
  4. Data Loss Prevention DLP discovers where sensitive data lives, monitors it in all states, and enforces policies to block unauthorized transfer, quarantine files, or alert administrators.
  5. Hybrid Infrastructures Combining on-premises systems with cloud resources creates environments where security controls must be extended and enforced consistently across both.
  6. Third-party Integrations Connecting external services via APIs requires encrypted transmission, extended access management, and monitoring to ensure security policies apply beyond the organization's own perimeter.
  7. Attack Surface Management Identifying, reducing, and continuously monitoring all potential attack paths into a system minimizes exposure. Hardening, defense-in-depth, and vulnerability management are the core techniques.
  8. Control Effectiveness Security controls must be measured to confirm they actually work. Metrics, scanning, and assessments provide quantitative and qualitative evidence of how well controls mitigate risk.

data states

Data can exist in three distinct conditions, and the protection method must match the state. The same data may pass through all three states during normal operations.

StateDefinitionProtection method
At rest Stored on a hard drive, database, or any persistent medium Encryption of stored data, access control policies
In transit Actively moving between systems across a network Encryption protocols such as TLS (Transport Layer Security)
In use Actively being processed by an application or system Secure processing environments (e.g. trusted execution environments)

data classification

Data classification is the process of organizing data into categories based on its sensitivity, value, or regulatory requirements. The assigned classification determines the level of security controls and handling procedures applied to that data.

classification models

Classification models use tiered categories to describe sensitivity. A common four-tier model uses:

  • Public: no restrictions, safe to share externally
  • Internal: for employees only, limited risk if exposed
  • Confidential: restricted access, significant risk if exposed
  • Restricted: highest sensitivity, strictest controls, often regulatory-driven

Example: a healthcare organization's patient records are classified as Restricted, requiring encryption and strict access controls to meet HIPAA requirements. Internal training documents are classified as Internal Use Only, with lower controls and broader employee access.

data labeling and tagging

Labeling and tagging assign visible or machine-readable markers to data so that classification decisions are enforced consistently, and automated systems know how to handle the data.

Labels
Human-readable classification markers applied to data or documents. Labels such as "Confidential" or "Public" communicate the data's sensitivity tier and instruct people how to handle and protect it. They make classification visible at the point of access.
Tags
Machine-readable metadata attached to data, such as project names, department associations, or regulatory identifiers. Tags enable automated processes. A DLP system can trigger a policy when it detects data tagged with "Finance" and "SOX Compliance," restricting who can access it and blocking unauthorized transfers automatically.

data loss prevention

Data loss prevention (DLP) detects, prevents, and responds to unauthorized access, transmission, or use of sensitive data. DLP depends on data being correctly classified, labeled, and tagged first.

data discovery

Before policies can be enforced, sensitive data must be located. Data discovery identifies where sensitive data exists across the organization's infrastructure, whether stored in databases and file systems (at rest) or flowing across the network (in transit).

policy enforcement

Once data is discovered and classified, DLP policies define automated responses to policy violations:

  • Alert administrators when a sensitive file is accessed or moved unexpectedly
  • Block unauthorized transfer attempts in real time
  • Quarantine files that have violated policy
  • Apply tombstone markers in place of the original file, notifying the user that a policy violation occurred and the file has been restricted

hybrid infrastructures

A hybrid infrastructure combines on-premises systems with cloud-based resources. Security must span both environments consistently, because data and workloads move between them.

  • Traditional controls (firewalls, encryption, access management) must be extended from the local data center to cloud environments
  • Security challenges arise from differences in management interfaces and control planes across on-premises and cloud systems
  • Policies and tools must enforce security consistently regardless of where data is processed or stored
  • Example: a company stores sensitive data on-premises for regulatory reasons while using cloud applications for processing. Secure communication channels and coordinated policies are required to protect data in both locations.

third-party integrations

Third-party integrations connect external services to an organization's internal systems, typically through APIs. Every integration extends the trust boundary and introduces new risk that must be explicitly managed.

  • Encrypted transmission and storage: API communications must encrypt sensitive data during transit and at rest using protocols like TLS
  • Extended access management: access controls and authentication must cover third-party services, not just internal systems
  • Monitoring and logging: interactions with third-party systems must be logged and monitored to detect anomalies and ensure policy compliance
  • Example: integrating a third-party payment processor via API requires TLS encryption, strict access controls, and real-time monitoring to keep financial transactions secure and compliant

attack surface management

Attack surface management identifies, reduces, and continuously monitors all potential paths of attack into a system. The goal is to minimize the number of ways an attacker can reach protected assets.

Hardening
Securing systems by reducing their attack surface. Hardening includes disabling unnecessary services and open ports, removing unused software, and applying up-to-date security patches. Fewer running components means fewer potential entry points.
Defense-in-depth
Implementing multiple, layered security controls so that a failure in one layer does not compromise the entire system. Combining firewalls, intrusion detection systems, and access controls means an attacker must bypass multiple independent barriers. No single control is relied upon as the sole line of defense.
Vulnerability management
Regular scanning and remediation of security weaknesses across the network and assets. This includes legacy components that no longer receive vendor updates but remain critical to the infrastructure. Identified vulnerabilities are prioritized and addressed before they can be exploited.

control effectiveness

Security controls are only valuable if they actually mitigate the risks they are designed to address. Control effectiveness refers to the degree to which controls achieve that goal, verified through ongoing measurement.

Metrics
Quantitative measures of security control performance. Examples include incident response times, the number of vulnerabilities patched within an SLA window, or the percentage of systems with up-to-date signatures. Metrics reveal whether controls are improving over time.
Scanning
Automated tool-based detection of vulnerabilities or misconfigurations within the system. Scanning produces a prioritized list of weaknesses that need remediation. Running scans regularly catches new vulnerabilities introduced by changes to systems or software.
Assessments
Structured evaluations of security controls through audits, penetration testing, or compliance checks. Assessments verify that controls are functioning as intended, not just that they are deployed. Penetration testing actively probes for gaps; audits verify policy compliance; compliance checks confirm alignment with regulatory requirements.
exam tips

key takeaways

  • Know all three data states and their protection method. At rest: encryption and access control. In transit: TLS. In use: secure processing environments. The exam will swap the state and ask for the control, or give the control and ask what state it applies to.
  • Data classification drives everything downstream. Labeling, tagging, and DLP policies all depend on data being classified first. If the scenario describes a failure in DLP, the root cause is often missing or incorrect classification.
  • Labels are for people; tags are for machines. Labels communicate sensitivity to users. Tags attach metadata that automated systems (like DLP engines) act on.
  • Tombstone markers are a DLP enforcement action. When a DLP policy is violated, a tombstone marker is placed where the file was, notifying the user that the file has been restricted due to a policy violation.
  • Hybrid infrastructure requires consistent policy across both environments. The challenge is not just connecting on-premises and cloud but enforcing the same security controls in both places simultaneously.
  • Attack surface management: three techniques. Hardening (reduce), defense-in-depth (protect with layers), vulnerability management (monitor and remediate). Know each by name and function.
  • Control effectiveness: three mechanisms. Metrics (measure), scanning (detect), assessments (validate). These are distinct: scanning finds vulnerabilities; assessments verify whether controls actually work.
quick decision rules
  • Data stored in a database or on disk → At rest: encryption + access control
  • Data moving across a network → In transit: TLS
  • Data actively being processed → In use: secure processing environment
  • Sensitivity tiers (public/internal/confidential/restricted) → Data classification model
  • Human-readable marker on a document → Data label
  • Machine-readable metadata enabling automated DLP → Data tag
  • User notified of a policy violation in place of their file → Tombstone marker (DLP enforcement)
  • On-premises plus cloud environment → Hybrid infrastructure
  • Disabling unused services and applying patches → Hardening
  • Firewalls plus IDS plus access controls working together → Defense-in-depth
  • Quantitative measure of how well controls perform → Metrics (control effectiveness)
  • Audit, penetration test, or compliance check to verify controls → Assessment (control effectiveness)

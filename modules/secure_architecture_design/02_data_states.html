Data States

Data exists in three states: at rest, in transit, and in use. The risks differ in each state, so the protection methods must too.

overview

The same piece of data can pass through all three states during normal operations: stored in a database (at rest), sent across a network (in transit), and loaded into an application for processing (in use). Choosing the wrong protection method for a given state leaves the data exposed even if other controls are in place.

quick reference

StateDefinitionPrimary protection tools
At rest Stored on a server, hard drive, or cloud storage; not actively accessed AES-256, BitLocker, FileVault, TDE, access control, MFA
In transit Moving over a network between systems or to the cloud TLS (HTTPS), IPsec (VPN), SSH / SFTP
In use Actively being accessed or processed by a user or application Memory encryption, RBAC / ABAC, data masking, secure enclaves

data at rest

Data at rest is stored and not currently being accessed. Even though it is not moving, it is vulnerable to unauthorized access if physical media is stolen or storage systems are compromised. The primary defense is encryption: encrypted data on a stolen drive is unreadable without the key.

encryption tools

BitLocker
Full-disk encryption built into Windows. Encrypts the entire drive so that data is unreadable if the device is lost or stolen. Requires authentication (PIN, password, or TPM chip) to decrypt at boot.
FileVault
Full-disk encryption built into macOS. Operates the same way as BitLocker but for Apple devices. The disk is encrypted at rest and requires user credentials to unlock.
AES-256
Advanced Encryption Standard with a 256-bit key length. The symmetric encryption standard most widely used to protect stored data. AES-256 is considered extremely difficult to break with current computing power and is the baseline for most data-at-rest encryption requirements.
TDE (Transparent Data Encryption)
Encrypts data automatically at the storage level within databases. TDE is transparent to applications because encryption and decryption happen at the database engine layer. No application code changes are required. Commonly used in SQL Server, Oracle, and PostgreSQL to protect database files and backups.

access controls

Encryption alone is not sufficient. Strict access control policies and multifactor authentication (MFA) ensure that only authorized users can unlock and read the data, even when they have physical or logical access to the storage system.

data in transit

Data in transit is moving across a network and is particularly vulnerable to interception or tampering by anyone with access to the network path. Encryption protocols protect data by making it unreadable to anyone who intercepts it in transit.

TLS (Transport Layer Security)
Encrypts data as it moves over the internet. TLS is the protocol behind HTTPS. When a browser connects to a site using HTTPS (HTTP over TLS), TLS negotiates a secure session and encrypts all data exchanged. This prevents eavesdropping even on public networks.
IPsec (Internet Protocol Security)
Encrypts and authenticates IP traffic at the network layer. IPsec is commonly used to secure VPN tunnels, encrypting all traffic between two endpoints over an untrusted network. Unlike TLS, which operates at the application layer, IPsec protects all traffic at the network layer regardless of the protocol above it.
SSH / SFTP
Secure Shell for encrypted remote access; SFTP (File Transfer Protocol over SSH) for encrypted file transfers. SSH encrypts the connection between a client and a remote system, protecting commands and responses from interception. SFTP extends this to file transfer operations, replacing unencrypted FTP.

data in use

Data in use is actively being processed by an application or user. This is the most exposed state because data must be decrypted and loaded into memory to be worked with. Protection focuses on limiting who can interact with the data and how, and isolating it from the rest of the system during processing.

memory encryption

Data loaded into RAM for processing can be encrypted while it sits in memory. This protects against attacks that target a system's memory directly, such as cold-boot attacks or memory scraping.

access controls

  • RBAC (Role-Based Access Control): users can only access data their role permits. A payroll clerk can access salary records; a developer cannot.
  • ABAC (Attribute-Based Access Control): access decisions use additional context attributes beyond just role, such as time of day, device type, or location.

data masking

Data masking shows users a partial or obfuscated version of sensitive data rather than the full value. A database administrator viewing customer records might see only the last four digits of a Social Security number. The user can perform their task without seeing the full sensitive value, limiting exposure without blocking access entirely.

secure enclaves

Secure enclaves create isolated processing environments within a processor where sensitive data can be handled without being exposed to the broader operating system or other applications. Even if the host system is compromised, data inside the enclave remains protected.

  • Intel SGX (Software Guard Extensions) is the most widely known implementation, creating isolated memory regions that the OS cannot read
  • Used in scenarios where data must be processed by third parties or in untrusted environments
exam tips

key takeaways

  • Know each state, its definition, and its primary controls. At rest: encryption (AES-256, BitLocker, TDE) + access control. In transit: TLS, IPsec, SSH/SFTP. In use: memory encryption, RBAC/ABAC, data masking, secure enclaves. The exam gives you a scenario and asks which control applies.
  • TDE is a database-level at-rest encryption tool. It encrypts data at the storage layer automatically without requiring changes to applications. Know it by name and function.
  • TLS protects web traffic; IPsec protects VPN traffic. Both are in-transit encryption protocols but operate at different layers. TLS is application-layer; IPsec is network-layer.
  • Data masking is a data-in-use protection technique. Showing partial data (last 4 SSN digits, masked credit card number) limits exposure while the data is actively being used.
  • Secure enclaves (Intel SGX) isolate sensitive data during processing. Even a compromised OS cannot read data inside an enclave. This is the primary tool for protecting data in use in untrusted environments.
  • RBAC restricts by role; ABAC restricts by attributes. ABAC is more flexible and context-aware (can factor in location, time, device) but more complex to configure.
quick decision rules
  • Encrypting a hard drive or database file → At rest (AES-256, BitLocker, FileVault, TDE)
  • Encrypting a database automatically at the storage layer → TDE
  • Securing web traffic (HTTPS) → TLS (in transit)
  • Encrypting VPN traffic between sites → IPsec (in transit)
  • Encrypted remote login or file transfer → SSH / SFTP (in transit)
  • Showing only last 4 digits of an SSN → Data masking (in use)
  • Isolated processing environment safe from a compromised OS → Secure enclave / Intel SGX (in use)
  • Access based on job function → RBAC (in use)
  • Access based on role plus context (time, location, device) → ABAC (in use)

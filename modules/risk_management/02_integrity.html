Integrity Risk Considerations

Protecting data and systems from unauthorized modifications that could compromise accuracy and trustworthiness. Detection matters as much as prevention.

overview

Integrity is the assurance that data remains accurate, unaltered, and that any unauthorized changes are detectable. Four considerations work together: detecting and preventing interference, using hashing to fingerprint and verify data, remotely journaling transaction logs for forensic reconstruction, and applying anti-tampering measures across hardware and software.

the four considerations

01 Interference Unauthorized modification or disruption of data or systems. Malicious, accidental, or system errors.
02 Hashing One-way cryptographic fingerprint. Any change to the input produces a different hash.
03 Remote Journaling Continuous transmission of transaction logs to a remote location for forensic reconstruction.
04 Anti-Tampering Physical and software measures to detect or prevent unauthorized modifications.

Interference

Any unauthorized modification or disruption of data or systems that compromises accuracy and integrity. Interference is a broad category covering three sources:

  • Malicious attacks: an attacker deliberately altering records, configurations, or system state
  • Accidental changes: a user or admin mistake that corrupts data or breaks a system
  • System errors: bugs, hardware failures, or race conditions that produce incorrect state

Mitigations

  • Strict access controls so only authorized users can modify data
  • User authentication protocols (MFA, strong identity verification)
  • System monitoring to detect unauthorized changes as they happen
  • Backup systems and failover strategies to recover and restore to an unaltered state

Hashing

A cryptographic technique that creates a fixed-size fingerprint (hash value) from an input of any size. Hashing is the primary tool for verifying data integrity: store the hash of a known-good file, recompute later, compare. Any difference means the file changed.

Trap-door (one-way) functions

Hashing algorithms are one-way: easy to compute the hash from input, computationally infeasible to reverse from hash back to input. The "trap door" metaphor: once you fall through, you can't climb back up.

Common algorithms

MD5
128-bit hash. Deprecated for integrity. Once widely used, but practical collisions have been demonstrated since 2004. Still appears in legacy systems and non-security uses (file checksums, deduplication) but should not be used to verify integrity against an attacker.
SHA-1
160-bit hash. Deprecated for integrity. Considered broken since the SHAttered attack (2017) demonstrated a practical collision. Most major browsers and CAs no longer accept SHA-1 certificates.
SHA-256
Part of the SHA-2 family. Current standard for integrity. Produces a 256-bit hash. No practical collisions known. Used widely in TLS certificates, blockchain (including Bitcoin), code signing, and file integrity monitoring.
SHA-3
Newer family, different internal design than SHA-2. Also current. Standardized by NIST in 2015. Offers a backup option if SHA-2 is ever broken. Same security goals as SHA-256, different mathematical construction (Keccak-based sponge function).

Hash collisions

A collision occurs when two different inputs produce the same hash output. When collisions are practical, the integrity guarantee breaks: an attacker can substitute a malicious file that hashes to the same value as a trusted one.

  • MD5 and SHA-1 have practical collisions: do not use them for integrity
  • SHA-256 and SHA-3 are collision-resistant in practice
  • A collision-resistant hash makes any unauthorized change detectable, no matter how small

Hashing in Practice

A short walkthrough on Kali Linux: detecting a single-character change across a folder of files using SHA-256 hashing.

The demo

  1. Set up An Integrity_Monitoring folder contains four files (SimpleFile_1 through SimpleFile_4), each holding five paragraphs of Lorem Ipsum text.
  2. Generate Monday's baseline hash Run SHA-256 over each file, combine the per-file hashes, then hash the combination to produce a single fingerprint for the whole folder. Save to Hash_Monday.txt.
  3. Verify on Tuesday (no changes) Re-run the same hash pipeline, save to Hash_Tuesday.txt. Compare with diff Hash_Monday.txt Hash_Tuesday.txt. Output is empty: files are identical.
  4. Simulate a Wednesday compromise Open SimpleFile_2 and delete the first character. Save. The change is one byte out of thousands.
  5. Re-hash and compare Run hash + diff in one command, chained with &&. The diff now reports a difference: the Wednesday hash does not match Tuesday's.
  6. Result A single-character change anywhere in any monitored file produces a completely different hash. The compromise is detected, even though we never opened the files to inspect them.
why this matters

File integrity monitoring (FIM) is the production version of this demo. Tools like Tripwire, AIDE, and OS-level FIM in EDR products run this loop continuously, alerting when a monitored file's hash drifts from baseline.

The technique is also how software supply chain protections work: signed package hashes catch tampered binaries, Subresource Integrity (SRI) catches tampered JavaScript loaded from a CDN, and Git commits chain hashes to detect history rewrites.

Remote Journaling

Continuous transmission of transaction logs to a remote location, ensuring a secure off-site record of system activity in real time.

Remote journaling vs backups

  • Backups copy entire data sets at scheduled intervals. Goal: restore lost data.
  • Remote journaling copies individual logs of system changes, transactions, or critical events as they happen. Goal: preserve the record of what happened, not the data itself.

Why it matters for integrity

  • After a system failure, security breach, or disaster, remotely stored logs allow reconstruction of recent system activities
  • If an attacker tampers with local logs to hide their tracks, the remote copy preserves the unaltered record
  • Provides forensic evidence and supports root-cause analysis
  • Does not restore lost data; backups still required for that

Anti-Tampering

Measures designed to detect or prevent unauthorized modifications to hardware or software. Covers both physical and digital surfaces.

Physical anti-tampering

  • Tamper-evident seals on hardware enclosures (breaking the seal is visible evidence)
  • Locks on server cabinets, USB port blockers, secure boot chips
  • Tamper-resistant hardware modules (HSMs) that zeroize keys if physical intrusion is detected

Software anti-tampering

  • Checksums and hashes on critical files (firmware, configurations, binaries)
  • Tamper-evident logs: append-only logging that records and alerts on attempts to alter system files or configurations
  • Code signing to detect modified executables
  • Secure boot to verify the OS hasn't been tampered with at startup

exam tips

Key Takeaways

  • Hashing is one-way (trap-door). Easy to compute, infeasible to reverse from hash back to input.
  • MD5 and SHA-1 are deprecated for integrity due to practical collisions. Use SHA-256 or SHA-3.
  • A hash collision is when two different inputs produce the same hash, breaking the integrity guarantee.
  • Remote journaling is not a backup. Journaling preserves logs of changes; backups preserve the data itself.
  • Interference can be malicious, accidental, or a system error. Don't assume malicious.
  • Anti-tampering covers both physical and software: seals and locks on the hardware side, checksums and tamper-evident logs on the software side.
  • Even a single-byte change produces a completely different hash. That's the whole point.
quick decision rules
  • Verify a file hasn't been changed → Hashing (SHA-256 or SHA-3)
  • Reconstruct recent system activity after a crash or breach → Remote journaling
  • Detect a hardware enclosure being opened → Anti-tampering (physical seals)
  • Detect configuration file changes → Anti-tampering (tamper-evident logs) or file hashing
  • Two different files produce the same hash → Collision (integrity guarantee broken)
  • Recommended hash for integrity today → SHA-256 or SHA-3 (never MD5 or SHA-1)
  • Restore lost data after a failure → Backup (NOT remote journaling)
  • Attacker tries to hide tracks by editing local logs → Remote journaling preserves the original

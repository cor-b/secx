Threat Actor Methods

How threats get analyzed and mapped: misusing legitimate features, the deceptively-bad design choices that create risk, and the diagrams that lay out every path to a goal.

overview

Threat actor methods are strategies attackers use to compromise systems. Three concepts: abuse cases (misusing legitimate features), anti-patterns (poor practices that appear effective but create risk), and attack trees/graphs (visual maps of the paths an attacker might take to a goal).

method 01

Abuse Cases

Misusing legitimate features to cause harm (fake reviews, bot upvotes, password-reset abuse). Fix: verification, CAPTCHAs, pattern monitoring.

method 02

Anti-patterns

Deceptively-bad practices that create risk (Base64 "hiding" passwords, hard-coded secrets). Fix: env variables, secure frameworks, code reviews.

method 03

Attack Trees/Graphs

Visual maps of attack paths from entry to goal. Help teams see where to defend. Mitigate with vuln assessments, patching, least privilege.

abuse cases

Abuse cases are scenarios where attackers manipulate legitimate features to cause harm or gain unauthorized access. Example: flooding a platform with fake reviews or using bots to upvote/downvote to manipulate ratings and rankings, damaging reputation and trust; password-reset misuse is another. Mitigate with review verification (verified purchases), CAPTCHAs to limit bots, monitoring for patterns (rapid submissions, repeated IPs), usage limits, and audits.

anti-patterns

Anti-patterns are poor practices or design choices that seem functional but create security risk, often from misconceptions about obscurity. Example: encoding a password in Base64 to "hide" it, Base64 only changes the format and is trivially reversible (e.g. with CyberChef). Another is hard-coding passwords or API keys in source code, exposing them if the repo is cloned, shared, or breached. Mitigate by storing secrets in environment variables (outside the codebase), using secure frameworks (Django, Spring Security), and regular code reviews with encryption/hashing.

attack trees/graphs

Attack trees/graphs are visual diagrams mapping the different paths an attacker might take to a goal, breaking a main objective (e.g. unauthorized access) into components like entry points, privilege escalation, and lateral movement. Example: a tree starting from entry points (phishing, weak passwords, unpatched software) branching into escalation and lateral movement to reach a database. Each node is a potential attack phase, helping teams visualize where defenses must be strongest. Mitigate with regular vulnerability assessments, patching, and least-privilege policies along the mapped paths.

EXAM TIPS

what to know for the exam

  • Abuse cases = misusing legitimate features (fake reviews, bot manipulation, password-reset abuse).
  • Anti-patterns = deceptively-bad practices: Base64 to "hide" passwords (reversible), hard-coded secrets.
  • Base64 is encoding, not encryption, it provides no protection.
  • Attack trees/graphs visually map attack paths from entry point to goal so defenders prioritize.
  • Fix anti-patterns with environment variables and secure frameworks; fix abuse cases with verification/monitoring.
decision rules
  • Scenario floods a site with fake reviews → abuse case
  • Scenario "secures" a password with Base64 → anti-pattern
  • Scenario hard-codes an API key in source → anti-pattern
  • Scenario diagrams every path to a target → attack tree/graph

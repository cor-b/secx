Secrets Management

Securely storing and controlling the sensitive things that grant access: tokens, certificates, passwords, and keys, plus rotating and deleting them so a leak has a short shelf life.

overview

Secrets management securely stores, manages, and controls access to sensitive credentials. The secrets: tokens (temporary digital credentials), certificates (identity-verifying documents), passwords (character strings), and keys (cryptographic elements). The lifecycle practices: rotation (regularly update/replace) and deletion (securely destroy when no longer needed, e.g. cryptographic shredding).

tokens

Tokens are temporary digital credentials for authentication or authorization, acting like time-limited digital keys, widely used in web apps and APIs. A session token keeps you logged in without re-entering credentials. Types include JWT (JSON Web Tokens, carry encoded claims like roles/permissions), OAuth access tokens (delegate access between apps), and API tokens. Tokens are signed and often encrypted to prevent tampering. Because a stolen token grants access until it expires, they're designed to expire quickly (e.g. 15 minutes of inactivity) and can be revoked.

certificates

Certificates are digital documents that verify the identity of users, devices, or websites using public/private key pairs. Types: SSL/TLS (secure websites), code-signing (verify software is untampered and from a trusted source), client (authenticate users to servers), email/S-MIME (encrypt and sign email), and device (authenticate hardware like printers/servers). They must be renewed before expiration, revoked if compromised, and configured correctly, an expired SSL/TLS cert triggers "not secure" warnings.

passwords

Passwords are secret character strings used for authentication, vulnerable to guessing, phishing, brute force, and credential stuffing. They should never be stored in plain text, instead stored as a hash (a fixed-length value); at login the entered password is hashed and compared, so the original is never exposed. Rotate periodically, and in high-security environments add salting, domain-wide policies, 2FA, and compromised-credential monitoring.

keys

Keys are cryptographic elements (public/private pairs in PKI, symmetric keys) that encrypt and decrypt data. Improper handling, insecure storage or embedding in code, leads to exposure. Common key issues:

Mismatched keys
Wrong public/private pair used to decrypt; produces errors like "key mismatch" or X509_check_private_key.
Improper key handling
Failing to protect private/symmetric keys; insecurely stored keys must be assumed compromised, revoked, and replaced.
Embedded keys
Stored on read-only crypto chips, can't be easily revoked/replaced, so the device must be protected from tampering.
Rekeying
Renegotiating session keys during a session (often by traffic volume); WPA3 SAE rekeys frequently to keep data in transit secure.
Compromised/exposed keys
An exposed private key (and its certificates) must be immediately revoked, replaced, and destroyed; protected data should be remediated.

rotation and deletion

Rotation regularly updates or replaces keys, passwords, tokens, and credentials, limiting how long a compromised secret is useful. Active keys are replaced, old ones revoked and destroyed, and data is re-encrypted with the new keys while access controls and systems stay synchronized. Deletion securely destroys secrets no longer needed. A key method is cryptographic shredding: destroy the decryption key and the encrypted data becomes permanently unreadable, useful in cloud storage where simply deleting files isn't enough.

EXAM TIPS

what to know for the exam

  • Four secret types: tokens (temporary), certificates (identity), passwords (strings), keys (crypto).
  • Tokens expire fast and can be revoked; JWT carries claims, OAuth tokens delegate access.
  • Passwords are stored hashed (never plain text); add salting and 2FA in high-security settings.
  • Exposed/compromised keys must be revoked, replaced, and destroyed; embedded keys can't easily be revoked.
  • Rotation limits the lifespan of a compromised secret; cryptographic shredding deletes data by destroying its key.
decision rules
  • Scenario makes data unrecoverable by destroying its key → cryptographic shredding
  • Scenario regularly replaces keys to limit exposure → rotation
  • Scenario throws an X509_check_private_key error → mismatched keys
  • Scenario stores passwords so the original is never exposed → hashing

Authentication and Authorization

The protocols that let you log in once and reach many services: SAML and OpenID prove who you are, OAuth controls what an app can do, and federation extends trust across domains.

overview

Authentication verifies identity; authorization determines access rights. Five concepts: attestation (verify an entity meets security policy), SAML (XML-based auth data exchange for SSO), OpenID (authentication via a single identity provider), OAuth (authorization framework using tokens), and federation (asserting identities across domains). Remember: the "auth" in OAuth is authorization, not authentication.

attestation

Attestation verifies that a user, system, or device meets specific security standards and policies, proving it's legitimate and uncompromised before granting access. It often generates a unique digital signature or hash proving credentials are genuine and unaltered, then sends it to a verifying party (an access control system) that checks compliance. Used in cloud services, remote access, and secure boot.

SAML (security assertion markup language)

SAML is a standard for exchanging authentication and authorization data between an identity provider and a service provider, using XML-based tokens. It enables single sign-on: an employee logs in once to the company identity provider, and SAML passes their authentication to email, file storage, and HR systems, reducing password fatigue. It's popular for secure cross-domain SSO.

OpenID

OpenID is an authentication protocol that lets users log into multiple services with a single identity managed by an identity provider, logging into many sites with your Google account, where Google is the identity provider verifying your credentials. It standardizes authentication, improves user experience, and reduces reused/weak passwords by outsourcing authentication to a trusted provider.

OAuth (open authorization)

OAuth is an authorization framework letting third-party apps access a user's data without handling their credentials. It determines what data the third party can reach and what it can do. When you connect a third-party app to share photos, OAuth issues a token granting only the data you agreed to (revocable anytime), instead of revealing your password. By issuing tokens instead of credentials, it reduces the risk of sharing passwords. For authentication, OAuth may employ OpenID.

federation

Federation shares and manages user identities across multiple domains or organizations, so a user authenticated in one trusted domain can access resources in another without separate logins. A student authenticated by their home campus can access a second campus's resources. Federation often uses SAML to exchange auth data, and is common in businesses, universities, and government.

how they work together

In a federated university SSO: SAML authenticates the student via the university identity provider (or OpenID lets them log in with Google). Once authenticated, when they use a third-party tool, OAuth issues a scoped token granting only the data the app needs, no university credentials shared. SAML/OpenID handle authentication; OAuth handles authorization.

EXAM TIPS

what to know for the exam

  • SAML = XML-based, exchanges auth/authz data between IdP and SP, the classic enterprise SSO protocol.
  • OpenID = authentication (log in to many services via one identity provider).
  • OAuth = authorization (token-based access for third-party apps; never shares the password). The "auth" is authorization.
  • Attestation verifies an entity complies with security policy before access.
  • Federation asserts identities across domains/organizations, often using SAML.
decision rules
  • Scenario grants a third-party app scoped access via a revocable token → OAuth
  • Scenario logs into many sites with a Google identity → OpenID
  • Scenario exchanges XML SSO assertions between IdP and SP → SAML
  • Scenario lets users in one org access another org's resources → federation

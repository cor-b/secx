Subject Access Control

A subject isn't only a person. Users, processes, devices, and services all request access, and each must be controlled so it touches only what it needs.

overview

Subject access control determines and enforces which subjects can reach specific resources. Subjects are four things: users (individuals), processes (running programs acting for a user/function), devices (physical or virtual hardware on the network), and services (background applications/functions). Each must be authenticated and scoped to least privilege.

subject 01

Users

Individuals needing access. Permissions match the role (accountant sees finance, developer doesn't), preventing over-access.

subject 02

Processes

Running program instances acting for a user/function. Granted only safe, necessary permissions; blocked if they exceed them (malware defense).

subject 03

Devices

Physical/virtual hardware on the network. Must authenticate and be compliant; unauthorized or compromised devices are blocked.

subject 04

Services

Background apps/functions interacting without a user. Access is scoped (email can send/receive, not read financial records).

the four subject types

Subject access control regulates each type so it can act only within its intended scope:

  • Users: individuals or entities needing resources. Access control gives each the correct permissions for their role, so an accountant can reach financial records and a developer cannot, keeping sensitive data on a need-to-know basis.
  • Processes: instances of programs running for a user or system function (a web browser checking email). They need specific permissions (open a file, reach the internet); if a process tries an unauthorized action, access control blocks it, protecting against threats like malware.
  • Devices: any physical or virtual hardware connecting to the network (laptops, phones, VMs). They must authenticate and may need to prove they're patched and compliant, so only approved devices connect and compromised ones are blocked.
  • Services: system functions running continuously in the background (email, file sharing, databases), often interacting without direct user involvement. Access control defines what each can access and do, so an email service can send/receive but not reach financial records.
EXAM TIPS

what to know for the exam

  • The four subjects are users, processes, devices, and services, not just people.
  • Users get role-based permissions (least privilege, need-to-know).
  • Processes are running programs granted only necessary, safe permissions; blocking unauthorized actions defends against malware.
  • Devices must authenticate and prove compliance before connecting.
  • Services run in the background and must be scoped so they only access what their function requires.
decision rules
  • Scenario controls what a running program can touch → process subject
  • Scenario blocks a non-compliant laptop from the network → device subject
  • Scenario limits a background app to only its needed data → service subject
  • Scenario gives an accountant finance access but not dev tools → user subject (role-based)

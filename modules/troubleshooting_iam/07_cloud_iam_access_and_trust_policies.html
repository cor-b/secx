Cloud IAM Access and Trust Policies

Two policy types working together: access policies say what a user or service can do, trust policies say which accounts and services are allowed to assume a role across boundaries.

overview

Cloud IAM manages who can access cloud resources and under what conditions. Access policies assign roles and permissions to users, groups, and services, defining what actions they can perform. Trust policies define relationships and trust boundaries between cloud accounts/services, specifying which entities may assume a role. Together: access policies define permissions, trust policies govern how those permissions are shared.

policy 01

Access Policies

Define permissions for users/groups/services: what they can do (e.g. read a bucket but not delete). AWS uses JSON, Azure uses RBAC, GCP assigns to users/groups.

policy 02

Trust Policies

Define trust boundaries between accounts/services: which entities may assume a role (e.g. one AWS account assuming a role in another).

cloud IAM access policies

Access policies manage who can access cloud resources and what actions they can perform, granting a user permission to read a storage bucket but not delete or modify it. They enforce least privilege across all major platforms:

  • AWS: policies defined as JSON documents attached to users, groups, or roles.
  • Azure: uses role-based access control (RBAC).
  • GCP: permissions assigned directly to users or groups.

A typical example: a data analyst is granted read access to a data warehouse but not the ability to alter configurations or delete resources, letting them do their job without risking stability. Platforms also let admins audit access activity for oversight.

cloud IAM trust policies

Trust policies define relationships and trust boundaries between accounts, services, or external entities, specifying which are trusted to assume roles. In a multi-account AWS setup, a trust policy can let an admin in one account assume a role in another for cross-account operations. Separate dev/test/prod accounts can use trust policies to allow controlled interaction while keeping security boundaries intact. Platform mechanisms: AWS IAM roles use trust policies for cross-account role assumption; Azure uses service principals and managed identities; GCP configures trust between projects and services. Example: a Lambda function in one account assumes a role in a second account to access its resources.

EXAM TIPS

what to know for the exam

  • Access policies = what an identity can do (permissions on resources); trust policies = which entities can assume a role (boundaries between accounts).
  • AWS access policies are JSON; Azure uses RBAC; GCP assigns permissions to users/groups.
  • Trust policies enable secure cross-account/cross-service operations (e.g. assuming a role in another account).
  • The two work together: access defines permissions, trust controls how they're shared.
decision rules
  • Scenario lets a user read but not delete a storage bucket → access policy
  • Scenario lets account A assume a role in account B → trust policy
  • Scenario uses JSON documents for AWS permissions → access policy
  • Scenario isolates dev/test/prod but allows controlled interaction → trust policy

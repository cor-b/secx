Access Control

Enforcing who reaches what: one login for many apps (SSO), ticket-based proof of identity (Kerberos), and tightly governed elevated access (PAM).

overview

Access control manages and enforces who can access resources based on policies and roles. Three concepts: single sign-on (SSO) (one set of credentials for many services), Kerberos (ticket-based network authentication, common in Active Directory and SSO), and privileged access management (PAM) (controls and monitors elevated access, granting it just-in-time).

concept 01

SSO

One login for multiple apps within a domain. Fewer passwords to remember, easier to enforce policy, fewer password-related breaches.

concept 02

Kerberos

Ticket-based authentication over non-secure networks (AD). TGT + service tickets, time-bound, supports mutual authentication.

concept 03

PAM

Controls/monitors elevated access; just-in-time privileges, password rotation, session management, auditing.

single sign-on (SSO)

SSO lets users access multiple applications with a single set of credentials within a domain or organization. It simplifies the user experience (one username/password) and improves security: users are less likely to write down or forget credentials, fewer password-related breaches occur, and the organization can enforce password policy more effectively with just one credential set to manage.

Kerberos

Kerberos is a network authentication protocol providing secure identity verification over non-secure networks, widely used in Microsoft Active Directory. It uses a ticket-based system: the password is used locally to encrypt the request (never sent in plain text). After authenticating, the user gets a ticket-granting ticket (TGT), then receives service tickets to access specific resources. Each ticket is time-stamped with a limited lifespan, so access is time-bound and an intercepted ticket has limited use. Kerberos supports mutual authentication (both user and service verify each other) and is commonly integrated into SSO.

privileged access management (PAM)

PAM controls and monitors access to critical systems by managing user privileges, ensuring only authorized users gain elevated access, on a just-in-time basis (privileges given only when needed for a task). It enforces strict access policies and uses tools that automatically rotate passwords, manage sessions, and enforce MFA, while continuously monitoring privileged activity to detect suspicious behavior. PAM also provides detailed auditing and reporting of all privileged actions for accountability and compliance.

PAM in practice: the Linux sudoers file

Configuring the sudoers file is a concrete example of PAM. It controls which users can run commands with elevated privileges, enforcing least privilege. In the demo, a new user (Jeremiah) initially can't run sudo apt update or install packages ("not in the sudoers file"). After adding him to sudoers with permission for apt update and apt install htop, but deliberately not apt remove, he can run only the allowed commands; apt remove htop still fails. This shows elevated access restricted to exactly what the role requires.

EXAM TIPS

what to know for the exam

  • SSO = one credential set for many apps; improves UX and lets you enforce policy centrally.
  • Kerberos = ticket-based auth (TGT + service tickets), time-stamped/time-bound, never sends plain-text passwords, supports mutual auth, used in AD.
  • PAM grants elevated access just-in-time and audits/monitors privileged use; rotates passwords, manages sessions, enforces MFA.
  • The Linux sudoers file is a classic PAM example enforcing least privilege.
decision rules
  • Scenario uses time-stamped tickets in an AD environment → Kerberos
  • Scenario grants admins elevated rights only when needed and audits them → PAM
  • Scenario lets a user reach many apps with one login → SSO
  • Scenario restricts which sudo commands a user may run → PAM (sudoers)

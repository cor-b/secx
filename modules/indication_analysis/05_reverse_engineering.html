Reverse Engineering

Taking compiled software apart to understand it: the two forms code takes (byte code, binary), and the two ways to make it readable (disassembly to assembly, decompilation to high-level).

overview

Reverse engineering breaks down software or hardware to understand its structure, functionality, and vulnerabilities. Four concepts: byte code (intermediate code for virtual machines), binary code (machine-level ones and zeros), disassembly (binary → assembly language), and decompilation (executable → high-level language).

concept 01

Byte Code

Intermediate, platform-independent code run by a virtual machine (.class, .pyc, .dll). Architecture-neutral.

concept 02

Binary Code

Machine-level instructions (1s and 0s) executed directly by the CPU. Tools: binwalk, hexdump, strace, ldd.

concept 03

Disassembly / Decompilation

Disassembly: binary → assembly (IDA Pro, Ghidra, OllyDbg). Decompilation: → high-level language (closer to source).

byte code and binary code

Byte code is a low-level, platform-independent intermediate representation executed by a virtual machine, more abstract and portable than assembly, which is tied to a CPU architecture. In forensics it lets analysts examine software/malware without source code (e.g. analyzing Java byte code), found in .class (Java), .pyc (Python), or .dll (.NET) files. Binary code is machine-level instructions (ones and zeros) the CPU executes directly, key for analyzing executables and firmware. Tools: binwalk (extract embedded firmware components), hexdump (display binary in hex), strace (trace system calls of a running binary), and ldd (identify shared library dependencies).

disassembly and decompilation

Disassembly
Converts binary code into assembly language, a low-level human-readable form, to see how software operates. Tools: IDA Pro, Ghidra, OllyDbg (and debuggers like GDB). Debuggers add breakpoints to pause execution and inspect memory/variables/registers at critical moments.
Decompilation
Goes further, translating executable/byte code back into a high-level language (C, Python, Java) that resembles the original source, easier to interpret. Languages that compile to byte code (Java, Python, JavaScript) decompile most successfully (tools: Java Decompiler, Uncompyle6, Ghidra).

The distinction: disassembly produces assembly (low-level), decompilation produces high-level source-like code.

EXAM TIPS

what to know for the exam

  • Byte code = platform-independent intermediate code run by a VM (.class, .pyc, .dll); binary = machine code (1s and 0s) the CPU runs directly.
  • Disassembly converts binary → assembly language; decompilation converts executable → high-level language.
  • Disassembly/debugging tools: IDA Pro, Ghidra, OllyDbg, GDB; breakpoints pause execution to inspect state.
  • Binary analysis tools: binwalk (firmware), hexdump (hex view), strace (system calls), ldd (dependencies).
  • Byte-code languages (Java, Python) decompile most successfully.
decision rules
  • Scenario analyzes a .class or .pyc file without source → byte code
  • Scenario converts a binary to assembly to study it → disassembly
  • Scenario recovers high-level source-like code → decompilation
  • Scenario extracts components from a firmware image → binary analysis (binwalk)

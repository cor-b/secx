Volatile and Non-volatile Storage Analysis

Collect the most fleeting evidence first. The order of volatility decides what to capture before it vanishes, and forensic imaging makes a court-admissible copy of what persists.

overview

Volatile and non-volatile storage analysis examines temporary and permanent data for evidence of compromise. Two concepts: the order of volatility (the sequence to collect evidence, most ephemeral first, per RFC 3227) and forensic imaging (creating an exact, court-admissible bit-by-bit copy of non-volatile storage).

order of volatility

The order of volatility is the sequence in which to collect digital evidence, since data types are lost at different rates. Volatile data (CPU cache, RAM) disappears at power-off; non-volatile data (hard drives, USBs) persists. Collecting in the right order preserves the most transient evidence first. RFC 3227 ("Guidelines for Evidence Collection and Archiving") defines the order:

  1. Registers and CPU cache
  2. Routing tables, ARP cache, process tables, kernel statistics, and RAM
  3. Temporary file systems / swap space
  4. Disks
  5. Remote logging and monitoring data
  6. Physical configuration and network topology
  7. Archival media (e.g. backup tapes)

So a forensic analyst at a powered-on machine captures CPU registers/cache and RAM (active processes, network connections, encryption keys) before disks, because that memory data is permanently lost if the system is turned off.

forensic imaging

Forensic imaging creates an exact, bit-by-bit copy of a storage device while preserving integrity to a court-admissible standard, so the original stays untouched. It captures not just visible files but slack space (unused space between the end of a file and the end of its cluster/block), unallocated space, and deleted files. Three common tools:

dd
Native Unix/Linux command for a bit-by-bit copy (e.g. if= input device, of= output file, bs=64K block size, conv to continue past errors).
dcfldd
A forensic version of dd (US DoD Computer Forensics Lab) adding on-the-fly hashing (MD5, SHA-256) and progress indicators to verify integrity.
FTK Imager
A graphical tool that creates copies, automates hash generation, and supports chain of custody.

Best practices: use a write blocker to prevent modifying the original, hash the image to verify it matches the original, and image the original once, then analyze a copy-of-the-copy to preserve integrity.

EXAM TIPS

what to know for the exam

  • Order of volatility (RFC 3227): registers/cache → routing/ARP/process tables/RAM → swap → disks → remote logs → physical config/topology → archival media.
  • Capture the most volatile data (CPU cache, RAM) first, it's lost at power-off.
  • Forensic imaging = exact bit-by-bit copy preserving integrity; captures slack space, unallocated space, deleted files.
  • Tools: dd (basic), dcfldd (adds hashing), FTK Imager (GUI, chain of custody).
  • Use a write blocker and hash the image to keep evidence admissible.
decision rules
  • Scenario decides what evidence to grab first on a live system → order of volatility (RAM before disk)
  • Scenario makes a court-admissible exact copy of a drive → forensic imaging
  • Scenario prevents writing to the original during imaging → write blocker
  • Scenario needs on-the-fly hashing during imaging → dcfldd

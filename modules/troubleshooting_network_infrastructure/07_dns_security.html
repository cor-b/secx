DNS Security

DNS was built in the 1980s with no security. These four concepts add it back: signing responses, securing replication, understanding the poisoning attack, and redirecting malicious traffic.

overview

DNS security protects name resolution from attacks and misconfigurations. Four concepts: DNSSEC (digitally signs DNS data for integrity/authenticity), zone transfers (legitimate replication between servers that must be secured), DNS poisoning (cache poisoning, injecting false records), and sinkholing (redirecting malicious traffic to a safe server).

concept 01

DNSSEC

Digitally signs DNS data with public/private keys so resolvers verify authenticity. Prevents cache poisoning. Both zone and resolver must support it.

concept 02

Zone Transfers

Normal replication of zone data between servers (redundancy). If unsecured, attackers grab all records. Restrict + use TSIG.

concept 03

DNS Poisoning

Attack: inject false records into a resolver's cache, redirecting users to malicious sites. Defend with DNSSEC + patching.

concept 04

Sinkholing

Defense: reroute malicious traffic (e.g. malware C2) to a controlled sinkhole server to drop or analyze it.

DNSSEC (DNS Security Extensions)

DNS translates domain names to IP addresses, but it was built in the 1980s without security, there's no built-in way to verify a response came from the correct source, which enables cache poisoning. DNSSEC adds digital signatures and public-key cryptography: each zone gets a public/private key pair, the owner signs DNS data with the private key, and resolvers verify it with the public key before accepting a response. This prevents attackers from poisoning the cache. Both the zone owner and the resolvers must be configured to support DNSSEC.

zone transfers

A DNS zone is a portion of the namespace a server manages; zone transfers are the normal process of synchronizing that data between servers for reliability and redundancy (if one authoritative server fails, another has the data). But if unsecured, an attacker can initiate an unauthorized zone transfer and obtain all records in a zone, revealing system names and IPs that make targeting easier. Secure transfers by limiting which servers may request them and using TSIG (transaction signatures) to authenticate. The lesson's demo: in Windows Server DNS, switching the Zone Transfers setting from "only to servers in the Name Servers tab" to "to any server" let a Parrot box pull every record via dig axfr, one radio button from a serious vulnerability.

DNS poisoning

DNS (cache) poisoning is when an attacker sends fake DNS responses that the resolver accepts and caches; anyone then querying that domain is sent to the attacker's IP, often a convincing fake site. Techniques include DNS response spoofing (flooding the server with fake responses hoping one matches a real request) and exploiting outdated/poorly-configured servers that don't verify authenticity. It's hard to detect because the fake site looks normal. Defenses: DNSSEC (verifies responses via signatures) and keeping DNS servers patched.

sinkholing

DNS sinkholing diverts malicious traffic to a controlled sinkhole server instead of letting it reach its destination (like an external command-and-control server). When the DNS server identifies a malicious domain/IP, it reroutes traffic there; infected-device requests to C2 are intercepted and sent to the sinkhole, which either drops the traffic or lets analysts safely observe the malware. It's effective for cutting off botnet communication after a threat is detected. The sinkhole needs current threat intelligence to know which domains are malicious, and must itself be secured.

EXAM TIPS

what to know for the exam

  • DNSSEC signs DNS data (public/private keys) so resolvers verify authenticity, prevents cache poisoning. Both zone and resolver must support it.
  • Zone transfers are legitimate replication; unsecured ones leak all records. Restrict requesters and use TSIG.
  • DNS poisoning = injecting false records into a resolver's cache to redirect users; defend with DNSSEC + patching.
  • Sinkholing is a defense, reroute malicious traffic (e.g. malware C2) to a safe sinkhole to drop or analyze.
  • DNS was designed in the 1980s with no security, hence these add-ons.
decision rules
  • Scenario verifies DNS responses with digital signatures → DNSSEC
  • Scenario lets an unauthorized host pull all DNS records → insecure zone transfer
  • Scenario redirects users to a fake site via cached false records → DNS poisoning
  • Scenario redirects malware C2 traffic to a safe server → sinkholing

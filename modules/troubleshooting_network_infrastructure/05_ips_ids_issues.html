IPS/IDS Issues

A detection system is only as good as where you put it and how you tune it. The three failure modes: bad placement, missing rules, and misconfigured rules.

overview

IPS/IDS issues are challenges in deploying, configuring, and tuning these systems. Three problems: poor placement (wrong network location → missed threats or noise), lack of rules (no policies/signatures to identify threats), and rule misconfiguration (rules set wrong → too many or too few alerts). Key distinction: an IDS passively monitors (out of band, on a mirror/SPAN port); an IPS actively blocks (must be in-line).

issue 01

Placement

IDS = out of band (mirror/SPAN port, copies traffic). IPS = in-line (drops packets, ends connections). Wrong spot misses threats or floods alerts.

issue 02

Lack of Rules

No/incomplete signatures and policies → threats go undetected, especially sophisticated APT tactics.

issue 03

Rule Misconfiguration

Wrong thresholds, too broad/narrow, outdated signatures → false positive floods or missed threats; IPS may block legit apps.

placement

Placement determines effectiveness, and it differs by system type. An IDS passively monitors and only generates alerts, so it doesn't need to be in-line: it can sit on a switch mirror port (SPAN port) receiving a copy of traffic without affecting flow. An IPS actively blocks malicious traffic in real time, so it must be in-line, dropping packets or terminating connections; place it on high-traffic or sensitive paths (like the internal-to-external partner link). Wrong placement causes inefficiency or gaps: a misplaced IDS detects irrelevant traffic and floods the team with false positives, while a poorly placed IPS lets harmful traffic bypass it.

lack of rules

Both systems rely on predefined policies/signatures to identify malicious behavior (known malware signatures, suspicious flows, unusual logins). Without comprehensive, updated rules, an IDS may miss threats, and one with an incomplete rule set might catch common attacks but miss sophisticated ones (e.g. APT tactics, techniques, and procedures). An IPS lacking sufficient rules may let malicious traffic pass unblocked, allowing data exfiltration or exploitation. Keeping rule sets current ensures the latest threats are recognized.

rule misconfiguration

Misconfigured rules, wrong thresholds, overly broad or narrow rules, outdated signatures, cause two problems. Too aggressive → false positive floods (legitimate traffic flagged, overwhelming the team); too lenient → false negatives (real threats like exfiltration go unnoticed). For an IPS the stakes are higher because it actively blocks: overly aggressive rules can block business-critical apps (e.g. unusual-looking encrypted financial traffic), causing disruption. Signature updates matter, too: outdated signatures miss new malware variants. Regular audits, automated rule updates, and tuning to the network's traffic prevent these issues.

EXAM TIPS

what to know for the exam

  • IDS = passive, out of band (mirror/SPAN port), only alerts. IPS = active, in-line, can block/drop.
  • The "must be in-line" requirement is the IPS; IDS doesn't affect traffic so it can be off to the side.
  • Lack of rules → missed threats (especially sophisticated/APT). Keep signatures updated.
  • Misconfigured rules cause false positive floods (too aggressive) or false negatives (too lenient).
  • An overly aggressive IPS can block legitimate business-critical traffic.
decision rules
  • Scenario needs to block malicious traffic in real time → IPS, placed in-line
  • Scenario monitors a copy of traffic without affecting it → IDS on a mirror/SPAN port
  • Scenario floods the team with alerts on benign traffic → rule misconfiguration (too aggressive)
  • Scenario misses a new malware variant → outdated signatures / lack of rules

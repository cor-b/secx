Data State Protection

Data needs different protection depending on what it is doing: sitting in storage, moving across a network, or being processed in memory.

overview

Data state protection safeguards data based on its current condition. Data at rest is in storage, protected with encryption (AES) and access controls. Data in transit moves across networks, protected by encrypted tunnels (TLS, IPsec). Data in use is being processed in RAM or CPU registers, protected by in-memory encryption and secure enclaves. Each state has a different threat model and a different defense.

state 01

At Rest

Stored on disk, SSD, or cloud. Protected with AES and access controls. Encryption granularity: disk, block, file, or record level.

state 02

In Transit

Moving across networks. Protected by encrypted tunnels: TLS (web, email, file transfer) and IPsec (VPNs).

state 03

In Use

Active in RAM or CPU registers. Vulnerable to memory scraping. Protected by in-memory encryption (AMD SME) and secure enclaves (Intel SGX).

data at rest

Data at rest is stored on devices (thumb drives, hard drives, SSDs) or cloud storage, not moving or being processed, files, databases, backups. The primary protection is encryption: AES turns data into ciphertext only readable with the correct key, backed by access controls like passwords, MFA, and permissions. Encryption can be applied at four granularities:

Disk-level
Encrypts an entire disk or partition with a symmetric algorithm like AES. One key for the whole volume; can slow boot and login. BitLocker (Windows) and FileVault (macOS) are examples.
Block-level
Secures individual fixed-size blocks of data, mainly for virtual partitions and storage area networks. Protects specific data segments rather than the whole disk, useful in complex storage environments.
File-level
Encrypts individual files, each with the same or a different key. Lets users protect their own files on a shared drive so others cannot read them without the key.
Record-level
Encrypts each record in a database individually for the most granular control. Useful for high-security databases. Like file-level but finer; can slow access since data is decrypted on each open.

data in transit

Data in transit moves across corporate networks, the internet, or wireless links, making it vulnerable to interception and tampering. The fix is encrypted tunnels.

TLS

Transport Layer Security creates an encrypted tunnel between a client and server, ensuring confidentiality and integrity. SSL was its 1990s predecessor and is now outdated; TLS 1.2 and 1.3 are the recommended standards. TLS operates at the application layer (OSI Layer 7). To set up, the client encrypts a random value with the server's certificate and public key; the server decrypts it to derive a symmetric key both sides use, defeating on-path attacks. TLS secures more than web traffic, it also protects email, file transfers (FTP secured with TLS becomes FTPS), and remote authentication. A cipher suite like ECDHE_RSA_AES128_GCM_SHA256 specifies key exchange (ECDHE), signatures (RSA), encryption (AES-128-GCM), and message authentication (SHA-256).

IPsec

IPsec secures data by creating encrypted tunnels between devices, often for VPNs, providing confidentiality, integrity, and authentication. Its protocols are Authentication Header (AH) (authentication and integrity) and Encapsulating Security Payload (ESP) (adds encryption). It runs in transport mode or tunnel mode; tunnel mode encrypts both data and headers and is preferred for site-to-site VPNs. Internet Key Exchange (IKE) sets up the tunnel: Phase 1 authenticates peers and exchanges keys, Phase 2 establishes the security associations used to encrypt and transmit data.

data in use

Data in use is being actively accessed or processed, residing temporarily in RAM or CPU registers. This exposes it to memory scraping, where attackers read data directly from memory, since traditional encryption does not protect data during active processing. Protections include in-memory encryption (for example AMD's Secure Memory Encryption, which encrypts data in RAM) and secure enclaves (Intel SGX isolates sensitive data within an application, resisting memory scraping).

EXAM TIPS

what to know for the exam

  • Three states: at rest (storage), in transit (networks), in use (RAM/CPU registers).
  • At rest → AES. In transit → TLS and IPsec. In use → in-memory encryption and secure enclaves.
  • At-rest encryption granularity, broad to fine: disk → block → file → record. Disk-level = BitLocker/FileVault.
  • TLS (1.2/1.3) replaced SSL; operates at OSI Layer 7. FTP + TLS = FTPS. Cipher suites name the algorithms used.
  • IPsec: AH = authentication/integrity, ESP = encryption. Tunnel mode encrypts headers too (site-to-site VPNs). IKE has Phase 1 and Phase 2.
  • Data in use is exposed to memory scraping; Intel SGX secure enclaves and AMD SME defend it.
decision rules
  • Scenario involves encrypting stored database files → data at rest (AES)
  • Scenario involves securing a credit card number traveling to a payment processor → data in transit (TLS)
  • Scenario involves a site-to-site VPN encrypting packets and headers → IPsec tunnel mode
  • Scenario involves protecting data being processed in memory → data in use (secure enclave / in-memory encryption)
  • Scenario involves encrypting each database record with its own key → record-level encryption

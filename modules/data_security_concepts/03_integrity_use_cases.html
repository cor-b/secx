Integrity Use Cases

Trusting software, not just data. Knowing where code came from (provenance) and proving it has not been altered (code integrity) defends the supply chain.

overview

Integrity use cases apply integrity controls to software to keep it functional and secure. Software provenance answers "where did this come from and can I trust it?" using version control, an SBOM, and SCA tooling. Software and code integrity proves code has not been altered from its trusted state using cryptographic hashes and code signing. Together they defend against supply chain attacks.

use case 01

Software Provenance

Tracks origin and history of components. Version control (Git), SBOM inventory, and SCA tools verify sources and flag tampering.

use case 02

Software & Code Integrity

Proves code is unaltered from its trusted state via cryptographic hashes and code signing, automated in CI/CD pipelines.

software provenance

Software provenance is about understanding the origin and history of every software component: who developed it, where it came from, and how it changed over time. It answers the trust question for code the way a chain of custody does for evidence.

version control

Version control systems like Git record every change to the codebase, what changed, when, and by whom, creating an auditable history of development. Beyond tracking, version control lets teams roll back to a previous trusted version if a problem is found.

SBOM and SCA

A software bill of materials (SBOM) is a detailed recipe listing every component, library, and dependency in an application, including third-party and open-source pieces. Software Composition Analysis (SCA) tools such as Snyk, Black Duck, or WhiteSource scan the SBOM and codebase to verify components come from trusted sources, checking for vulnerabilities, licensing issues, and tampering.

SCA tools identify each component and version, compare them against a database of trusted components and known vulnerabilities, and verify integrity by comparing cryptographic hashes against known-good sources. A hash mismatch flags a component that may have been swapped for a compromised version, the core defense against supply chain attacks.

software and code integrity

Code integrity ensures software stays secure, reliable, and unchanged from its trusted state. Two mechanisms work together.

cryptographic hashes

A hash creates a unique fingerprint of each version of the code. The hash is stored securely and recalculated whenever the code is accessed, modified, or deployed; if the new hash matches the original, the code is unaltered. This catches unauthorized modifications early.

code signing

Code signing lets a developer sign code with a digital certificate from a trusted certificate authority. The code itself is not encrypted; instead a hash of the code is encrypted with the developer's private key, forming a digital signature attached to the code. On download, the system uses the developer's public key to decrypt the signature (revealing the original hash), independently hashes the downloaded code, and compares. A match confirms the code is authentic and unaltered; a mismatch flags it as untrusted. This is automated in CI/CD pipelines with tools like Jenkins or GitLab CI.

EXAM TIPS

what to know for the exam

  • Provenance = origin and history of software. Code integrity = proof it has not been altered.
  • Version control (Git) provides the auditable change history and rollback for provenance.
  • SBOM lists all components/dependencies; SCA tools (Snyk, Black Duck, WhiteSource) scan it for bad sources, vulnerabilities, and tampering.
  • Code signing = encrypt a hash of the code with the developer's private key; verify with the public key. Confirms authenticity and integrity.
  • Together these defend against supply chain attacks (a compromised component swapped in during development).
decision rules
  • Scenario involves tracking who wrote code and how it changed → software provenance (version control)
  • Scenario involves an inventory of all components and dependencies → SBOM
  • Scenario involves scanning dependencies for compromised or vulnerable components → SCA
  • Scenario involves signing software so users can verify it is authentic and unaltered → code signing
  • Scenario involves preventing a malicious component being inserted during development → integrity use cases / supply chain defense

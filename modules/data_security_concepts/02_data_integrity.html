Data Integrity

Proving data has not changed. Hashing turns any input into a fixed-length fingerprint, so even a one-character change is instantly detectable.

overview

Data integrity ensures the accuracy, consistency, and reliability of data throughout its lifecycle, confirming it has not been altered without permission. Hashing is the primary technique: a one-way cryptographic function turns any-size input into a fixed-size hash digest that uniquely fingerprints the data and cannot be reverse-engineered. Key algorithms are MD5, the SHA family, and RIPEMD. Hashing underpins file verification, file integrity monitoring, and digital signatures.

how hashing works

A hashing function takes input of any length and transforms it into a fixed-length output called a hash digest, a digital fingerprint of the original data. The slightest change to the input produces a dramatically different output, and the function cannot be reversed to recover the input. Three properties make a hashing algorithm reliable:

  • Fixed-length output regardless of input size (MD5 always produces a 128-bit digest, whether you hash a word or a whole book)
  • Deterministic: the same input always produces the same output, ensuring consistency across systems
  • One-way (trapdoor): the output cannot be used to recreate the input

A collision occurs when two different inputs produce the same hash output, which undermines integrity validation. Resistance to collisions is what separates strong algorithms from weak ones.

hashing algorithms

MD5
Message Digest 5, a 128-bit digest. Once popular but now considered insecure because its short digest makes it prone to collisions. Its weakness drove the development of SHA.
SHA family
Secure Hash Algorithm. SHA-1 (160-bit) improved on MD5 but hit similar collision issues. SHA-2 (digests from 224 to 512 bits) greatly reduces collision risk and is the current standard. SHA-3 is the latest, adding more rounds of computation for stronger security.
RIPEMD
RACE Integrity Primitives Evaluation Message Digest, with 128, 160, 256, and 320-bit variants. Developed independently of government influence, which makes it popular with privacy advocates. Used in PGP and Bitcoin.

practical applications

file and download verification

If a file is sent along with its hash digest, the recipient can hash the received file with the same algorithm and compare. Matching digests confirm the file was not altered in transit. Software vendors publish a hash next to downloads for exactly this reason: hash the download, compare to the vendor value, and a match confirms it was not tampered with.

file integrity monitoring (FIM)

FIM creates a baseline hash digest for critical files and continuously re-checks them. If a file changes, its new digest differs from the baseline, triggering an alert, useful for catching an attacker modifying a config file or installing unauthorized software. Tools include Tripwire, OSSEC, and AIDE.

digital signatures

Digital signatures combine integrity with non-repudiation. The sender hashes the message, then encrypts that digest with their private key to form the signature. The recipient decrypts the signature with the sender's public key to reveal the original digest, independently hashes the received message, and compares. A match proves the message is unaltered and that only the sender (who holds the private key) could have signed it.

EXAM TIPS

what to know for the exam

  • Hashing = one-way, fixed-length digest, deterministic, not reversible. It validates integrity, not confidentiality.
  • MD5 (128-bit) and SHA-1 (160-bit) are deprecated due to collisions. SHA-2 is the current standard; SHA-3 is newest.
  • A collision = two different inputs producing the same hash. Collision resistance is the measure of a hash's strength.
  • RIPEMD is the privacy-community algorithm, used in PGP and Bitcoin.
  • FIM tools (Tripwire, OSSEC, AIDE) baseline file hashes and alert on changes.
  • Digital signatures = hash the message, encrypt the hash with the sender's private key. Provides integrity AND non-repudiation.
decision rules
  • Scenario involves confirming a downloaded file was not modified → compare hash digests
  • Scenario involves two different files producing the same hash → collision
  • Scenario involves alerting when a critical system file changes → file integrity monitoring
  • Scenario involves proving both integrity and that the sender cannot deny sending → digital signature
  • Scenario asks for the current recommended hashing family → SHA-2 (or SHA-3)

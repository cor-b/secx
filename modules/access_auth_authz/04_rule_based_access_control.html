Rule-Based Access Control

Administrator-defined rules and conditions that determine who can access resources — from strict government clearance systems to flexible attribute-driven policies.

overview

Rule-based access control manages user access through predefined rules set by administrators. Three models operate under this umbrella: MAC enforces rigid clearance-level classifications (used in government/military systems), DAC lets resource owners grant or deny access at their own discretion (the default in Windows and Linux), and ABAC evaluates combinations of user, resource, and environment attributes dynamically for more granular, context-aware decisions.

model 01

MAC

Mandatory Access Control. Strict classifications; users cannot alter policies. Clearance level must match or exceed resource classification. Government/military use.

model 02

DAC

Discretionary Access Control. Resource owner controls permissions. Default in Windows and Linux. Flexible but potentially less secure if owners grant permissions carelessly.

model 03

ABAC

Attribute-Based Access Control. Access determined dynamically by user, resource, and environment attributes. Most flexible; ideal for complex, context-sensitive environments.

rule-based access control

Rule-based access control applies predefined rules configured by administrators. These rules determine who can access what and under what conditions, applying equally across all users who fall within the rule's scope. Access control lists (ACLs) on routers and firewalls are a classic example: when an administrator configures a firewall rule to block traffic from a particular subnet, that rule applies to all traffic matching that condition without individual user judgment.

  • Rules are set by administrators and cannot be overridden by individual users
  • Well-suited to environments where access must be tightly and consistently controlled
  • Example: allow access to financial records only during business hours, applied to all users

mandatory access control (MAC)

MAC is a strict rule-based model where access decisions are based on security classification labels. Users cannot modify or override these policies. Only administrators and the system itself control label assignments and access rules. MAC is reserved for high-security environments where unauthorized disclosure is unacceptable.

classification levels

Top Secret
Highest classification. Disclosure could cause exceptionally grave damage to national security. Users must hold Top Secret clearance to access. A user with Secret clearance is blocked from Top Secret resources.
Secret
Disclosure could cause serious damage. Accessible to users with Secret or Top Secret clearance. Most common clearance level in government settings.
Confidential
Disclosure could cause damage. Accessible to Confidential, Secret, or Top Secret clearance holders.
Unclassified
No formal classification. Accessible to any cleared individual or the public depending on the system. A user with any clearance level can access Unclassified resources.

SELinux

SELinux (Security-Enhanced Linux) is a MAC implementation developed by Red Hat in partnership with the NSA. It modifies the Linux kernel to enforce mandatory access controls and additional security policies, providing fine-grained control over what processes and users can do, even beyond standard Linux permissions.

discretionary access control (DAC)

DAC gives resource owners full control over their resources. The owner decides who can read, write, or execute a file or object. This is the default access model in both Windows and Linux. DAC is often described as a "need to know" model where access is granted at the owner's discretion.

DAC on Linux: chmod and chown

On Linux, file permissions are controlled via chmod (change mode) and chown (change owner). The file owner can grant or revoke permissions for themselves, their group, and all other users:

  • chmod 700 myfile: owner gets read/write/execute; all others get nothing
  • chown root myfile: ownership transferred to root. In DAC, the owner can give away ownership.

This flexibility is DAC's strength and weakness. Owners can configure permissions precisely, but if they grant access carelessly (for convenience or by mistake), the model provides no enforcement backstop.

attribute-based access control (ABAC)

ABAC is the most dynamic and flexible access control model. Rather than fixed roles or clearance levels, access is determined by evaluating a combination of attributes at the time of the request. This allows fine-grained, context-sensitive decisions that simpler models cannot express.

attribute categories

User attributes
Username, role, department, organization, security clearance, job title. Example: role = nurse, department = cardiology, clearance = standard.
Resource attributes
File name, data sensitivity level, resource owner, creation date, classification. Example: sensitivity = high, owner = HR department, type = patient record.
Environment attributes
Time of access, user location, current security alert level, device type. Example: time = 02:00, location = off-site, device = personal (unmanaged).

Example: a company's SharePoint site shows different content based on whether a user is accessing from within the corporate office or remotely via VPN — same user, same credentials, different access based on environment attributes. In healthcare, a nurse may access basic patient data during working hours; a doctor may access sensitive records at any time during their shift.

EXAM TIPS

what to know for the exam

  • MAC = mandatory. Strict. Clearance must meet or exceed resource classification. Only admins modify labels. Used in government/military. SELinux is the Linux implementation.
  • DAC = discretionary. Resource owner controls access. Default in Windows and Linux. chmod and chown on Linux.
  • ABAC = attribute-based. Most dynamic. Evaluates user + resource + environment attributes simultaneously. Use when context matters (time, location, device).
  • Rule-Based AC is the umbrella concept; MAC is an example of it. ACLs on firewalls are rule-based.
  • MAC clearance hierarchy (high to low): Top Secret → Secret → Confidential → Unclassified. A Secret-cleared user cannot access Top Secret resources.
  • DAC risk: owners may grant access too freely. No enforcement backstop if the owner makes a poor decision.
decision rules
  • Scenario involves a government system where users cannot change their own access permissions → MAC
  • Scenario involves a user's clearance level determining what classified documents they can view → MAC
  • Scenario involves a file owner granting a colleague read access to their document → DAC
  • Scenario involves chmod or chown on Linux → DAC
  • Scenario involves access based on role, time of day, and location simultaneously → ABAC
  • Scenario involves a firewall rule blocking all traffic from a specific subnet → rule-based access control (ACL)
  • Question asks which model SELinux implements → MAC

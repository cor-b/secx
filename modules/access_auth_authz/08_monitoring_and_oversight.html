Monitoring and Oversight

Continuously capturing what happens in a system and analyzing those records to enforce compliance, detect anomalies, and maintain an unalterable history.

overview

Monitoring and oversight ensures that access, authentication, and authorization activities are continuously tracked and reviewable. Logging collects detailed records of events — logins, access attempts, system changes — and centralizes them via Syslog or a SIEM for real-time analysis. Auditing analyzes those logs to verify regulatory compliance and detect violations, relying on tamper-proof storage mechanisms like WORM devices and blockchain to ensure the integrity of the audit trail.

function 01

Logging

Captures detailed records of access events: logins, failures, changes. Centralized via Syslog or SIEM. Normalized across heterogeneous sources for correlation and analysis.

function 02

Auditing

Analyzes logs to confirm regulatory compliance (HIPAA, PCI-DSS) and detect breaches or policy violations. Relies on tamper-proof audit trails that cannot be altered after the fact.

logging

Logging captures a continuous record of events across systems and devices. Effective logging programs centralize those records so they can be correlated, searched, and alerted on from a single location.

syslog

Syslog is the standard protocol for sending log messages from network devices, servers, and applications to a central logging server. Every Syslog message carries a PRI code (priority code) that encodes two values: the facility that generated the message and the severity of the event.

PRI code components

Facility
The type of system or process that generated the log. 24 defined facility codes. Facility 0 = kernel messages. Facility 4 = security/authentication messages. Facility 1 = user-level messages. Facility 2 = mail system.
Severity
The urgency of the event. Scale from 0 (highest urgency) to 7 (lowest). 0 = Emergency (system unusable). 1 = Alert (immediate action required). 2 = Critical. 3 = Error. 4 = Warning. 5 = Notice. 6 = Informational. 7 = Debug.
PRI calculation
PRI = (facility × 8) + severity. Example: facility 4 (security/auth) × 8 = 32, plus severity 5 (Notice) = PRI 37. A lower PRI value within a given facility means higher urgency.

SIEM (Security Information and Event Management)

A SIEM collects logs from across the entire environment — firewalls, servers, operating systems, applications, databases, and network devices — and provides centralized storage, correlation, and real-time alerting.

log normalization

Different systems generate logs in different formats: JSON, XML, proprietary Syslog variants, CSV. A core SIEM function is normalization — converting all incoming formats into a single, consistent structure so that events from a firewall, a Windows domain controller, and a cloud application can be correlated against each other. Without normalization, cross-system pattern detection would be impractical.

  • Sources include: firewalls, intrusion detection systems, servers, endpoints, applications, databases, cloud platforms
  • Normalized logs enable detection rules that span multiple systems (for example, a failed login on the VPN followed immediately by a successful login from a different country)
  • SIEM platforms also provide dashboards, alerting, and reporting for compliance

auditing

Auditing goes further than logging. It involves analyzing the collected log data to answer specific questions: are users accessing only what they are authorized to access? Are access patterns consistent with policy? Have any violations or anomalies occurred?

regulatory compliance

Many regulations mandate logging and auditing of specific events. HIPAA requires that all access to patient health information be logged and regularly audited to ensure only authorized personnel accessed it, and that improper access is detected. Regular audits of access logs are how organizations demonstrate compliance and catch violations before they become incidents.

tamper-proof audit trails

For an audit trail to be legally and forensically meaningful, it must be tamper-proof: no one should be able to modify or delete records after they are written. Two mechanisms provide this guarantee:

WORM storage (Write Once, Read Many)
Storage media that allows data to be written exactly once. After writing, the data is permanent and cannot be modified or overwritten. Example: a financial institution writes each transaction record to WORM storage. Once recorded, the entry is immutable — no admin, attacker, or insider can alter it. Critical for regulatory compliance and forensic investigations.
Blockchain
Each audit log entry is stored in a cryptographically linked block. Modifying any past block breaks the chain, generating an immediately detectable inconsistency. The cryptographic linkage makes retroactive alteration mathematically detectable. Blockchain-based logs are increasingly used in high-assurance environments where the integrity of the record is as important as its content.

EXAM TIPS

what to know for the exam

  • Syslog PRI = (facility × 8) + severity. Know the formula. Example: facility 4, severity 5 → PRI 37.
  • Severity scale: 0 = emergency (most urgent), 7 = debug (least urgent). Lower number = higher urgency.
  • Facility 4 = security/authentication logs. Facility 0 = kernel messages.
  • SIEM normalizes logs from disparate sources into a common format so cross-system correlation is possible.
  • Auditing = analyzing logs for compliance and anomaly detection. Logging = collecting the records in the first place.
  • WORM = write once, read many. Data written to WORM cannot be modified after writing. Used for tamper-proof audit trails.
  • Blockchain audit logs: each block cryptographically linked to the previous. Modification of any past record is detectable.
  • HIPAA requires logging and auditing of all access to patient health information.
decision rules
  • Scenario involves collecting records of login events and access attempts → logging
  • Scenario involves analyzing access records to verify policy compliance → auditing
  • Scenario involves sending log messages from multiple devices to a central server → Syslog
  • Question gives a facility and severity value and asks for the PRI code → (facility × 8) + severity
  • Scenario involves correlating events across firewalls, servers, and applications in real time → SIEM
  • Scenario involves converting logs from multiple formats into a common structure → SIEM normalization
  • Scenario involves storage that prevents records from being modified after they are written → WORM storage
  • Scenario involves an audit log where tampering with any past entry is cryptographically detectable → blockchain
  • Scenario involves a healthcare organization needing to prove all access to patient data was authorized → HIPAA audit / tamper-proof audit trail

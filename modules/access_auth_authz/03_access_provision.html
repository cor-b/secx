Access Provision

The full lifecycle of granting and revoking access rights — from identity verification at onboarding to credential removal at offboarding.

overview

Access provision manages user access to resources across their entire relationship with an organization. Four processes form the lifecycle: credential issuance creates and distributes authentication credentials after verifying identity, provisioning assigns access rights based on role, self-provisioning allows users to request access through automated systems, and deprovisioning removes access when it is no longer needed. Together they ensure the right people have access to the right resources at the right time.

step 01

Credential Issuance

Identity is verified (proofing), then credentials are generated and distributed: username, password, MFA token. The entry point for any new user.

step 02

Provisioning

Access rights are assigned based on the user's role. Marketing gets marketing tools; engineers get dev environments. Often automated via RBAC.

step 03

Self-Provisioning

Users request access to specific systems through a portal. Requests route to admin approval or are auto-approved by policy. Speeds up onboarding.

step 04

Deprovisioning

Access is revoked when no longer needed: role change, contract end, termination. Automated deprovisioning removes access across all systems immediately.

credential issuance

Credential issuance is the first step in the access lifecycle. It has two parts: identity proofing and credential generation.

identity proofing

Before credentials are issued, the user's identity must be confirmed. For a new employee, this typically involves providing identification documents or undergoing background checks. Proofing ensures that credentials are issued to the correct person before granting system access.

credential generation and distribution

Once identity is confirmed, credentials are created and delivered. Common patterns include:

  • Email with a temporary password the user must change at first login
  • One-time link that lets the user create their own password
  • MFA enrollment: linking an authenticator app, registering a hardware token, or adding biometrics

provisioning

Provisioning assigns access rights based on the user's role in the organization. A marketing employee receives access to marketing tools and databases; a software engineer gets access to development environments and repositories. Role-based access control (RBAC) systems handle this automatically by linking roles to predefined permissions, so assigning a user to a role immediately grants the right access without manual permission-by-permission configuration.

self-provisioning

Self-provisioning allows users to request additional access without directly involving IT administrators for each request. A new employee might use a self-service portal to request access to the HR system, a project management tool, or internal communication platforms. The request is then:

  • Reviewed and approved by an administrator, then access is granted automatically
  • Or auto-approved based on predefined policies (for example, RBAC rules that allow all marketing employees to access certain tools)

Self-provisioning speeds up onboarding and reduces IT burden while maintaining security through required approvals and policy-based gates.

deprovisioning

Deprovisioning removes access rights when they are no longer appropriate. Key triggers include an employee leaving the organization, changing to a different role, or a contract expiring. Deprovisioning is critical for security: former employees or users with changed responsibilities who retain system access represent a significant risk.

Automated deprovisioning immediately revokes access to all systems tied to the user's account the moment a trigger event is recorded in the HR or identity management system, ensuring no gap between the departure event and access removal.

EXAM TIPS

what to know for the exam

  • Identity proofing = verifying who the person is before issuing credentials. Happens before any access is granted.
  • Provisioning = assigning access rights based on role. RBAC automates this by linking roles to permissions.
  • Self-provisioning = user-initiated access requests via a portal. Admin approval or policy-based auto-approval grants access.
  • Deprovisioning = removing access when no longer needed. Automation ensures immediate revocation with no gap between the trigger and removal.
  • Deprovisioning is essential after role changes, not just terminations. A user who moves departments and retains old access is a security risk (privilege accumulation).
decision rules
  • Scenario involves verifying a new user's identity before issuing a username and password → credential issuance / identity proofing
  • Scenario involves automatically assigning system access based on a user's job role → provisioning
  • Scenario involves a user requesting access to additional tools through a portal → self-provisioning
  • Scenario involves an employee leaving and their access being removed → deprovisioning
  • Scenario involves a user who changed roles still having access to systems from their previous role → deprovisioning failure / privilege accumulation

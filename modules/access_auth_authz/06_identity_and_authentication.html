Identity and Authentication

How identity is verified once and trusted many times — across applications, organizations, and federated networks — through providers, single sign-on, and secure attestation.

overview

Identity and authentication systems verify who a user is and propagate that trust across services. Identity providers (IdPs) centralize authentication, managing credentials so individual services do not have to. SSO lets a single authentication event grant access to multiple services. Service providers delegate authentication to the IdP and trust its attestations. Federation extends this trust across organizational boundaries. Attestations (via SAML or JWT) carry proof of authentication from the IdP to the service provider.

concept 01

Identity Providers

Centralize authentication for multiple systems. Google, Microsoft Azure AD. Manage user identities so service providers delegate verification instead of handling it themselves.

concept 02

Single Sign-On

One authentication event grants access to multiple services. Kerberos handles SSO within enterprise domains; Shibboleth handles privacy-preserving cross-domain SSO.

concept 03

Federation

Organizations share authentication trust across institutional boundaries. Users access resources at other organizations using home credentials. Transitive trust extends this across multiple federations.

concept 04

Attestation

The IdP sends a signed message to the service provider confirming the user's identity and access rights. Delivered as SAML (XML) or JWT (JSON). The SP trusts the attestation without re-authenticating the user.

identity providers

An identity provider (IdP) is a system that creates, maintains, and manages user identities, and authenticates users on behalf of service providers. Rather than each application managing its own user database and authentication logic, they delegate that responsibility to the IdP.

Google is a widely recognized IdP: it allows users to log into many third-party services using their Google account. Google verifies the credentials, and the service trusts Google's confirmation. Microsoft Azure Active Directory is the dominant enterprise IdP, centralizing authentication for thousands of internal and cloud applications.

single sign-on (SSO)

SSO allows a user to authenticate once and then access multiple applications within the same domain or trust boundary without re-entering credentials. After the initial login, the user's authenticated session is reused across all participating services.

kerberos

Kerberos is the SSO protocol underlying Microsoft Active Directory. It uses a ticket-based system managed by the Key Distribution Center (KDC).

Step 1: TGT issuance
User logs in. The KDC verifies credentials and issues a Ticket Granting Ticket (TGT). The TGT is a time-limited encrypted token proving the user has been authenticated. It does not grant access to any specific service — it only proves identity to the KDC.
Step 2: Service ticket request
When the user wants to access a specific resource, the client presents the TGT to the KDC's Ticket Granting Service (TGS) and requests a service ticket for that resource. The user never re-enters credentials. The TGT does the talking.
Step 3: Resource access
The client presents the service ticket to the target application (service provider). The application validates it and grants access. The service ticket is specific to that one service. A separate service ticket is required for each different resource the user wants to access.

shibboleth

Shibboleth is an open-source SSO and federated identity system designed with privacy in mind. When a user authenticates with their origin (home) server, the target service provider only receives confirmation that the user has been authenticated — not the user's actual identity or credentials. This protects user privacy while enabling secure access across organizations. Common in research and academic environments.

service providers

Service providers (SPs) are the applications and systems that rely on an IdP for authentication. Instead of managing their own login systems, SPs delegate authentication to the IdP and grant access based on the IdP's attestation. Examples include Slack, Dropbox, and Salesforce, all of which support login via Google or other IdPs.

JWT (JSON Web Token)

JWTs (pronounced "jots") are a compact, signed token format used to transfer authentication information between an IdP and a service provider. A JWT contains claims about the user (identity, roles, expiry) in JSON format and is cryptographically signed so the SP can verify it has not been tampered with. The SP reads the JWT, validates the signature, and grants the appropriate level of access.

federation

Federation extends SSO trust across organizational boundaries. Rather than one IdP serving all users within a single organization, federated systems establish trust agreements between multiple organizations so that users from one can access resources at another using their home credentials.

eduGAIN is a real-world example: it connects national identity federations across the global research and education sector. A student at a university in one country can access library databases or research systems at a partner university in another country using only their home institution credentials.

transitive trust

Transitive trust allows trust relationships to chain across federations without requiring direct agreements between every pair. If Federation A trusts Federation B, and Federation B trusts Federation C, then Federation A automatically trusts Federation C. Users authenticated in Federation A can access resources in Federation C without any additional authentication steps, even though A and C have no direct trust relationship.

attestation

An attestation is a signed message from the IdP to the service provider that confirms the user has been authenticated and specifies what access they are entitled to. The SP receives the attestation, validates it, and grants access based on its contents — without performing its own authentication.

SAML (Security Assertion Markup Language)
XML-based attestation format. The IdP constructs an XML document (the assertion) containing the user's identity and permissions, signs it, and sends it to the SP. Widely used in enterprise SSO. The SP validates the signature and reads the claims from the XML structure.
JWT (JSON Web Token)
JSON-based token format. Compact and well-suited to web and API contexts. Signed for integrity. Contains identity claims (subject, issuer, expiry, roles). Used by OAuth 2.0 and OpenID Connect flows. More common in modern cloud and mobile applications than SAML.

EXAM TIPS

what to know for the exam

  • Identity provider (IdP) = centralized authentication authority. Authenticates the user so service providers do not have to.
  • SSO = one login, many services. User authenticates once; subsequent access to participating services requires no new credentials.
  • Kerberos SSO flow: login → TGT from KDC → service ticket from TGS → present service ticket to application. The TGT proves identity; the service ticket grants resource access.
  • Shibboleth = privacy-focused federated SSO. The SP only receives confirmation of authentication, not the user's identity details.
  • Federation = cross-organizational trust. Multiple organizations accept each other's IdP assertions.
  • Transitive trust: A trusts B, B trusts C, therefore A trusts C. No direct A-C agreement needed.
  • SAML = XML attestation format. JWT = JSON attestation format. Both carry signed identity claims from IdP to SP.
  • eduGAIN = the real-world example of a research/education federation using federated identity.
decision rules
  • Scenario involves a user logging in once and accessing email, file storage, and internal apps without re-entering credentials → SSO
  • Scenario involves Active Directory issuing tickets so users can access internal services → Kerberos
  • Scenario involves users at one university accessing another university's systems with home credentials → federation (Shibboleth / eduGAIN)
  • Scenario involves the SP not receiving the user's identity, only a confirmation that they are authenticated → Shibboleth
  • Scenario involves organizations trusting each other's authentication without a direct agreement → transitive trust
  • Scenario involves a signed XML message sent from the IdP to the SP confirming identity → SAML attestation
  • Scenario involves a compact JSON token carrying user claims between systems → JWT

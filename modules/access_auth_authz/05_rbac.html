Role-Based Access Control

Assigning permissions to roles rather than individuals — so access follows the job function, not the person, and changes automatically when roles change.

overview

RBAC simplifies access management at scale by replacing individual permission assignments with role-based inheritance. Administrators define roles that represent job functions (HR, Finance, Developer), attach specific permissions to those roles, and assign users to the matching role. The user inherits all permissions of that role automatically. When a user changes roles or leaves, updating their group membership is all that is needed — permissions follow.

how RBAC works

The core idea is separating permission management from user management. Instead of granting each user individual permissions to each resource, you define roles based on job function and link permissions to roles. Users are then members of roles, not direct holders of permissions.

  • Define roles based on job functions within the organization (HR, Finance, IT Support, Developer, Sales)
  • Assign permissions to roles — HR can access employee records, Finance can access financial systems, IT Support can access all systems
  • Assign users to roles — the user inherits all permissions of their role immediately
  • Manage by role — when a user changes jobs, update their role membership; permissions update automatically

This makes administration much simpler in large organizations. Adding a new employee means assigning them to their role — no manual per-resource permission configuration required. Changing roles means switching role membership rather than auditing and updating dozens of individual permissions.

RBAC on Windows: active directory groups

In Windows enterprise environments, RBAC is typically implemented using Active Directory groups. Groups represent roles in the organization. Permissions are assigned to groups, not to individual user accounts.

implementation steps

  • Create groups in Active Directory representing each role: HR, Finance, IT_Support, Developers
  • Assign resource permissions to those groups (the HR group gets read/write to the HR file share, Finance gets access to accounting software)
  • Add users to the appropriate group — they inherit all group permissions immediately
  • When an employee changes roles, remove them from the old group and add them to the new one; access updates automatically

RBAC on Linux: user groups

Linux implements RBAC through user groups. Administrators create groups representing roles and set directory and file permissions for each group. Users are added to the groups matching their job function.

implementation steps

  • Create groups: admins, developers, sales
  • Assign directory and file permissions to groups — developers get write access to /src, sales gets read access to /reports
  • Add users to their group with usermod -aG groupname username
  • When someone changes roles, update group membership; permissions follow automatically

EXAM TIPS

what to know for the exam

  • RBAC = permissions tied to roles, users inherit permissions from their role. No individual permission assignments.
  • Windows implementation: Active Directory groups. Linux implementation: user groups.
  • Key benefit: when a user changes roles, update group membership — all permissions update automatically. No manual per-resource audit needed.
  • RBAC is the mechanism behind provisioning automation: assigning a user to an AD group automatically grants access to all resources that group is permitted to use.
  • Contrast with DAC: in DAC, resource owners individually decide who can access their files. RBAC centralizes that decision in the role definition.
decision rules
  • Scenario involves assigning permissions based on a user's job function → RBAC
  • Scenario involves Active Directory groups controlling access to shared resources → RBAC (Windows)
  • Scenario involves updating a user's access by changing their group membership → RBAC
  • Scenario involves an organization wanting to simplify access management as it scales → RBAC
  • Question asks how access changes when a user moves from HR to Finance → remove from HR group, add to Finance group; permissions update automatically

Application Layer Security

API gateways and WAFs protect applications and their data at Layer 7 of the OSI model, where traffic has the most context and the most specific rule sets can be applied.

overview

Application layer security operates at the highest layer of the OSI model. Unlike lower-layer defenses that inspect packets by port or protocol, Layer 7 tools can understand what an application is actually doing and apply fine-grained policy to it.

The two key components are API gateways (which manage and secure traffic between clients and backend services) and WAFs (which filter HTTP traffic against specific web attack patterns). Together they create a layered defense at the application level.

API gateways

An API gateway sits between clients and backend services, accepting all incoming API calls and enforcing security policy before any request reaches the backend. It acts primarily as a reverse proxy, routing requests to the appropriate internal service based on the nature of the request.

Think of it like a toll booth: only requests that meet the gateway's conditions (authentication, rate limits, valid format) are allowed through. Everything else is stopped before it can reach or overload the backend.

key capabilities

Reverse proxy and request routing
Accepts all API calls entering the network and routes each to the appropriate backend service. A request for account details goes to the account service; a payment request goes to the payment service. This ensures the right resource handles each specific type of request.
Authentication and authorization
Verifies that the requester has valid credentials and the correct permissions before allowing access to protected data or services.
Rate limiting and throttling
Prevents any single client from overwhelming backend services by capping how many requests they can send in a given time window. This protects against both accidental overload and deliberate abuse such as credential stuffing or scraping attacks.
Traffic filtering and security
Blocks unauthorized requests and controls the flow of data, preventing unauthorized access and reducing the attack surface exposed to the internet.

WAF (web application firewall)

A WAF is a specialized firewall that inspects and filters HTTP traffic to protect web applications from common attack patterns. It applies rule sets that identify and block harmful requests before they reach the web server.

Because a WAF operates at the application layer it has context that a standard network firewall lacks. It can detect malicious SQL inside a form field or recognize a cross-site scripting payload in a URL parameter.

deployment modes

ModeHow it worksTrade-off
Inline Sits directly between the network firewall and the web server; actively blocks live attack traffic Can slow traffic slightly and may occasionally block legitimate requests (false positives)
Out-of-band Receives a copy of traffic and analyzes it passively; does not interrupt live connections Unintrusive and does not affect performance, but functions as detection only, not prevention

installation options

  • Separate appliance: the WAF operates as an independent device within the network infrastructure, outside the web server itself
  • Software plugin: the WAF integrates directly with the web server, running as a module alongside the application it protects
exam tips

key takeaways

  • API gateway = reverse proxy + authentication + rate limiting + routing. It is not just a firewall. Its job is to manage, secure, and intelligently route API traffic.
  • WAF targets web application traffic specifically. SQL injection and XSS are the canonical attack types a WAF defends against. Standard firewalls cannot do this at the application layer.
  • Inline WAF = active blocking; out-of-band WAF = detection only. A question describing a WAF that does not interfere with live traffic is describing out-of-band deployment.
  • Both components operate at OSI Layer 7. This is the distinguishing feature that separates them from network-layer tools like packet-filtering firewalls.
  • API gateways are always deployed inline since their purpose is to manage and route live traffic. WAFs have a choice of inline or out-of-band.
quick decision rules
  • Managing API traffic, auth, rate limiting, routing to backend services → API gateway
  • SQL injection, XSS, HTTP traffic filtering → WAF
  • WAF actively blocks live attacks → Inline deployment
  • WAF detects but does not interrupt traffic → Out-of-band deployment

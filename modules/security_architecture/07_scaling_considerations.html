Scaling Considerations

Vertical scaling adds capacity to a single machine; horizontal scaling adds more machines. Each approach has distinct limits, costs, and application design requirements.

overview

Scalability is the ability of a system to handle increasing demand. The limits come from processing capacity, memory, or available bandwidth. When a system hits those limits, it must either grow upward (more powerful machine) or grow outward (more machines). The choice shapes application architecture and long-term cost.

vertical scaling (scale up)

Vertical scaling means adding more resources to an existing machine: more CPU, more RAM, more storage, or more bandwidth. The machine gets bigger, not more numerous.

  • Simple to implement: often just a configuration or plan upgrade, no architectural change required
  • Works with any application, including stateful ones that store session data on the server
  • Popular in traditional on-premises systems and straightforward to do in cloud environments (e.g. upgrading an AWS Lightsail plan from 1 GB RAM to 4 GB RAM with a button click)
  • Has a ceiling: every machine or plan has a maximum. Once you hit it, vertical scaling is no longer an option.

horizontal scaling (scale out)

Horizontal scaling means adding more machines to share the workload rather than making one machine larger. Requests are distributed across many servers, each handling a portion of the total load.

  • Better for long-term growth: adding a machine is cheaper per unit than continuously upgrading a single machine
  • Virtually limitless: there is no hard ceiling — machines can be added indefinitely to meet demand
  • In cloud environments: resources become elastically scalable with per-use cost, built-in redundancy, and easy resizing
  • Requires up-front architectural work: workloads must be broken into smaller, independent pieces that can run in parallel across many servers

stateless application design

Horizontal scaling requires applications to be stateless: each request must be handled independently, with no reliance on data stored from previous requests on the same server.

  • No session data is stored on the server itself
  • Any necessary session data (user state, preferences) is stored externally in a shared database or cache that any server in the pool can access
  • This design means any server can handle any request, making it trivial to add or remove machines without disrupting users
  • Example: a blog's articles partitioned across multiple servers by year — a request for a 2022 article routes to the 2022 server; no single server needs to know everything

exam tips

key takeaways

  • Vertical = scale up = bigger machine; horizontal = scale out = more machines. Know both terms and both synonyms — the exam uses all four interchangeably.
  • Vertical scaling has a limit; horizontal does not. When a scenario describes a system that has hit the maximum resources for a single machine, the answer is to re-architect for horizontal scaling.
  • Horizontal scaling requires stateless application design. Stateless means no session data on the server; state is stored externally in a shared database or cache.
  • Horizontal scaling provides built-in redundancy because the workload is distributed across many machines. Vertical scaling leaves a single point of failure.
  • Horizontal is better long-term because adding inexpensive machines is cheaper per unit than repeatedly upgrading a single expensive machine.
quick decision rules
  • Add CPU, RAM, or storage to an existing server → Vertical scaling (scale up)
  • Add more servers to share the load → Horizontal scaling (scale out)
  • Hit the maximum plan or hardware limit → Re-architect for horizontal scaling
  • Application stores no session state on the server → Stateless design, required for horizontal scaling

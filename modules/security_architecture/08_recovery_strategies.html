Recovery Strategies

Disaster recovery planning, backup types, failover mechanisms, and DR testing methods that restore operations after disruption.

overview

Recoverability is the organization's ability to restore normal operations as quickly and effectively as possible after an incident. It is built on three pillars: a documented disaster recovery plan (with RTO and RPO objectives), a solid backup strategy (with the right backup type for each scenario), and failover mechanisms (so systems stay online when components fail).

Regular DR testing validates that the plan works before it is needed in a real incident.

disaster recovery planning

A disaster recovery plan documents the step-by-step procedures for restoring systems, applications, and data after an incident. It must include clear roles and responsibilities and two key objectives that define what "recovered" means.

RTO (Recovery Time Objective)
The maximum acceptable length of time that a system can be down after a disaster. RTO defines how quickly systems must be back online. A shorter RTO requires more investment in standby infrastructure and faster failover mechanisms.
RPO (Recovery Point Objective)
The maximum acceptable amount of data loss, measured in time. RPO defines how far back in time a restore is acceptable. An RPO of 4 hours means the organization can tolerate losing up to 4 hours of data. The RPO directly determines how frequently backups must be taken.

backup types

Different backup strategies balance storage cost, backup speed, and restore complexity. Choosing the right type depends on the RPO and how quickly data needs to be restored.

TypeWhat it capturesRestore process
Full Complete copy of all data Simple: restore from a single backup. Slowest to create and uses the most storage.
Incremental Changes since the last full or incremental backup Complex: must restore the last full backup plus every incremental in sequence. Fast to create, low storage per backup.
Differential All changes since the last full backup Simpler: restore the last full backup plus only the most recent differential. Grows larger over time between full backups.
Synthetic full Administratively merged full + incremental/differential Simple: produces a single complete backup without re-copying all data from source. Saves time and storage while retaining full-restore simplicity.

failover mechanisms

Failover mechanisms keep services available when a primary system fails by automatically transferring operations to a backup system.

Active-active
Both the primary and backup systems are running and handling traffic simultaneously. If one fails, the other continues without interruption. No switchover delay. More expensive since both systems must be fully provisioned and operational at all times.
Active-standby
One system handles all traffic while the standby sits idle, ready to activate if the primary fails. Less expensive than active-active since the standby does not need to handle production load until needed. There is a brief activation delay when the standby takes over.

DR testing methods

Regular testing validates that the disaster recovery plan actually works. Tests range from low-impact discussions to full production disruptions, in increasing order of realism and risk.

  1. Tabletop testing Key team members walk through the disaster recovery plan step-by-step in a discussion format. No systems are touched. Identifies gaps in the plan and clarifies roles before any real action is needed.
  2. Parallel testing Backup systems are activated and run alongside production systems simultaneously. Verifies that backup systems work correctly without risking production downtime.
  3. Simulation testing A mock disaster scenario is staged to test how the team responds in real time. More realistic than tabletop but still does not shut down the production environment.
  4. Full interruption testing Production systems are actually taken offline and backup systems must take over completely. The most realistic test, as it replicates an actual disaster. Carries real operational risk and requires careful planning and communication.
exam tips

key takeaways

  • RTO = how fast you restore; RPO = how much data loss is acceptable. RTO drives failover speed requirements. RPO drives backup frequency requirements.
  • Differential vs incremental restore complexity: incremental requires every backup in the chain; differential only needs the last full plus the most recent differential. Differential restores are simpler.
  • Active-active = both systems running simultaneously = no switchover delay. Active-standby = one idle backup = brief delay when activating, but cheaper.
  • Full interruption testing is the most realistic and most disruptive. Production goes offline. Tabletop is the least disruptive (discussion only, no systems affected).
  • Synthetic full backups combine full and incremental/differential data without re-copying from source, saving time and storage while keeping restore simple.
quick decision rules
  • How quickly systems must be restored → RTO
  • How much data loss is acceptable → RPO
  • Complete copy, simplest restore, most storage → Full backup
  • Changes since last backup, fast to create, complex restore → Incremental
  • Changes since last full, simpler restore than incremental → Differential
  • Both systems running, immediate failover → Active-active
  • One system idle until needed → Active-standby
  • Discussion only, no systems touched → Tabletop test
  • Production taken offline, backups take over → Full interruption test

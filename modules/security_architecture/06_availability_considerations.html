Availability Considerations

Load balancing, session persistence, and interoperability are the three techniques that keep systems accessible under high demand, failures, and disruption.

overview

Availability is about ensuring users can access services even when things go wrong. Load balancing distributes demand so no single server breaks under pressure. Persistence controls whether sessions stay tied to one server or float freely. Interoperability ensures that diverse systems can cooperate to handle requests rather than creating bottlenecks at integration points.

load balancing

A load balancer sits in front of a pool of backend servers and distributes incoming requests across them so that no single server becomes overwhelmed. If traffic spikes or a server fails, the remaining servers absorb the load and the service stays available.

load balancing algorithms

AlgorithmHow it worksKey note
Round Robin Rotates requests sequentially through all available servers Simplest method. No consideration for current server load — each server gets the next request in line regardless of how busy it is.
Least connections Routes each new request to the server with the fewest active connections Better suited for workloads with variable request duration where some requests take much longer than others.
IP hashing Hashes the client's IP address and maps it to a specific server Ensures the same user is always routed to the same server. Effectively creates session affinity at the algorithm level.

load balancing as a DDoS defense

A load balancer can mitigate DDoS attacks by distributing the flood of requests across multiple servers rather than letting them converge on one. If incoming traffic exceeds the capacity of all servers, the load balancer can drop additional requests, preventing total system collapse.

persistence and non-persistence

Persistence controls whether a user's session stays tied to a specific server or can be handled by any server in the pool.

Persistent (session affinity)
Once a user connects to a server, all subsequent requests in that session are routed to the same server. Required when the server stores session-specific state such as shopping cart contents or user preferences. Guarantees data integrity for the duration of the session, but limits load distribution flexibility.
Non-persistent
Each request may be handled by any available server in the pool, regardless of where previous requests went. More efficient for stateless workloads where requests are independent of each other. The load balancer can send each request to whichever server is currently least busy, maximizing throughput during high traffic.

interoperability

Interoperability is the ability of different systems, applications, and devices to work together seamlessly, exchanging and using information without barriers. In the context of availability, it means that the failure or limitation of one component does not create a hard wall that makes the whole service unavailable.

  • When systems are interoperable, they can share resources and hand off requests between each other, reducing individual bottlenecks
  • Interoperable systems recover more gracefully from partial failures: other components absorb the load rather than cascading into a full outage
  • Example: banking systems from different institutions that interoperate allow customers to transfer funds, use ATMs, and transact online regardless of which bank they belong to
exam tips

key takeaways

  • Round Robin ignores server load — requests rotate in order regardless of how busy each server is. Least connections routes to the least-loaded server.
  • IP hashing creates implicit session affinity because the same client IP always maps to the same server. Know this as an alternative to application-level persistence.
  • Persistent = session affinity = same server for the session. Required for stateful applications. Non-persistent = any server = better load distribution, requires stateless design.
  • Load balancers can defend against DDoS by distributing attack traffic and dropping excess requests when all servers are saturated.
  • Interoperability supports availability by allowing diverse systems to share load and continue functioning when one component is degraded.
quick decision rules
  • Distribute requests evenly with no load awareness → Round Robin
  • Route to the least-busy server → Least connections
  • Always send the same user to the same server → IP hashing (or persistent affinity)
  • Shopping cart, user preferences, stateful session → Persistent (session affinity)
  • Independent requests, no state needed, max throughput → Non-persistent

Configuration Files

The four formats that drive automated processes and security controls, each with a sweet spot: readable hierarchies, strict structure, lightweight data exchange, or simple readability.

overview

Configuration files define the settings and parameters guiding automated processes and security controls. Four formats: YAML (readable, indentation-based, complex configs), XML (strict tag-based hierarchy), JSON (lightweight key-value, data exchange/APIs), and TOML (simple, INI-like, small configs).

format 01

YAML

Yet Another Markup Language. Indentation-based, highly readable. DevOps and cloud infrastructure configs.

format 02

XML

Extensible Markup Language. Strict tag-based hierarchy, verbose. Web services, security policies, enterprise systems.

format 03

JSON

JavaScript Object Notation. Lightweight key-value pairs and arrays. API responses, data interchange.

format 04

TOML

Tom's Obvious Minimal Language. INI-like, simple, readable. Small configs like monitoring tools.

the four formats

YAML (Yet Another Markup Language)
Known for readability; uses indentation rather than brackets to define structure, ideal for complex, hierarchical configs. Common in cloud deployment and DevOps (resources, services, security groups), e.g. defining a web app's environments with nested database and security settings.
XML (Extensible Markup Language)
Highly structured, uses tags to denote hierarchy; verbose but strict, good where rigorous organization is needed. Used for web services and enterprise systems: security policies, authentication methods (e.g. OAuth2), access controls.
JSON (JavaScript Object Notation)
Lightweight, uses key-value pairs and arrays; compact and fast to parse. The default for API configs and data interchange (endpoints, encryption settings, session timeouts), readable by humans and machines.
TOML (Tom's Obvious Minimal Language)
Designed for readability and ease of writing; INI-like with bracketed sections and key-value pairs. Ideal for small/medium configs like monitoring tools (log file paths, alert thresholds, report intervals).
EXAM TIPS

what to know for the exam

  • YAML = indentation-based, readable, complex cloud/DevOps configs.
  • XML = strict tag-based hierarchy, verbose, web services and enterprise security policies.
  • JSON = lightweight key-value, the default for APIs and data interchange.
  • TOML = simple INI-like format for small configs (monitoring tools).
  • Match the format to the need: readability (YAML), strictness (XML), exchange (JSON), simplicity (TOML).
decision rules
  • Scenario uses indentation for a complex cloud deployment → YAML
  • Scenario needs strict tag hierarchy for security policies → XML
  • Scenario exchanges lightweight data between APIs → JSON
  • Scenario configures a small monitoring tool simply → TOML

Cloud Automation

Two pillars of automating the cloud: packaging apps into lightweight isolated containers, and defining infrastructure itself as version-controlled code.

overview

Cloud automation manages, deploys, and secures cloud resources efficiently. Two concepts: containerization (packaging apps with their dependencies into isolated containers for consistency across environments) and infrastructure as code (IaC) (defining infrastructure through version-controlled code for repeatable, consistent deployments).

concept 01

Containerization

OS-level virtualization: containers share the host OS, isolating only the app + its dependencies. Lightweight. Docker, Kubernetes.

concept 02

Infrastructure as Code

Define infrastructure in code (YAML/JSON) via Terraform, CloudFormation, Ansible. Consistent, repeatable, version-controlled.

containerization

Containerization is OS-level virtualization: the host OS creates separate environments, packaging each application with its dependencies into an isolated container that runs consistently across environments. Unlike traditional (Type 1/Type 2) virtualization, which isolates at the hardware level with each VM running a full guest OS, containers isolate at the operating-system level, sharing the host OS and isolating only the app, making them lightweight and resource-efficient. They use a containerization engine instead of a hypervisor (e.g. Docker, Kubernetes, Parallels Virtuozzo, OpenVZ). Containers are isolated by default (must use a virtual network to communicate) and bundle all their own dependencies, so they run the same on any host. The demo deployed an Nginx web server as a Docker container (docker pull nginx, docker run -d -p 8080:80).

infrastructure as code (IaC)

IaC configures and provisions infrastructure (networks, servers, storage) using code instead of manual setup, often written in JSON or YAML and implemented with tools like Terraform (multi-cloud), AWS CloudFormation (AWS), and Ansible (config management). It treats infrastructure as software: rather than logging into each server to configure it manually (slow, error-prone), you define the setup once in code and deploy it across many environments in seconds, consistently and repeatably. IaC also enables version control: every change is tracked and can be rolled back, so updates are as simple as modifying and reapplying the code, reducing human error and configuration drift.

EXAM TIPS

what to know for the exam

  • Containers isolate at the OS level (share the host OS); VMs isolate at the hardware level (full guest OS each).
  • Containers use a containerization engine (Docker), not a hypervisor; they're lightweight and bundle their dependencies.
  • IaC defines infrastructure in code (YAML/JSON) via Terraform, CloudFormation, or Ansible.
  • IaC gives consistent, repeatable deployments + version control + rollback, reducing drift and human error.
  • Terraform = multi-cloud, CloudFormation = AWS, Ansible = configuration management.
decision rules
  • Scenario packages an app with dependencies to run anywhere → containerization
  • Scenario provisions cloud infra from version-controlled code → infrastructure as code
  • Scenario needs lightweight isolation sharing the host OS → containers (vs VMs)
  • Scenario rolls back infrastructure to a prior config → IaC version control

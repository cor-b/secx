Cloud Management

Knowing exactly who secures what. The shared responsibility model draws the line between provider and customer, and it shifts as you move up the stack.

overview

Cloud management secures cloud resources while clearly dividing duties between provider and customer. The shared responsibility model sets that division: the provider handles the cloud itself (physical infrastructure, hardware), the customer handles what's in the cloud (data, access, configuration). This extends to encryption keys and licenses, which may be cloud-managed or customer-managed.

the shared responsibility model

The model defines which security tasks belong to the provider (AWS, Azure, GCP) and which to the customer. Providers own physical infrastructure, hardware security, and environmental controls; customers own data encryption, access management, and security permissions. Three control types:

Inherited controls
Fully managed by the provider (e.g. data-center environmental conditions).
Shared controls
Split between both, e.g. patch management: the provider patches the infrastructure, the customer patches the OS they run.
Customer-specific controls
Solely the customer's (e.g. configuring security zones, communication protocols).

The rule of thumb: the further up the stack you go (infrastructure → user-facing apps), the more responsibility shifts to the customer. The provider handles hardware, regions, availability zones, and core compute/storage; the customer protects data, patches and configures systems, and manages identity/access.

cloud vs customer-managed resources

For critical resources, responsibility often shifts to the customer even when the provider runs the infrastructure:

  • Encryption keys: the provider secures the key infrastructure (AWS KMS, Azure Key Vault), but the customer must generate, manage, and rotate keys properly, mismanage them (no rotation, loose access) and data is exposed despite secure infrastructure.
  • Licenses: in cloud environments, licenses are often bundled into the subscription and the provider handles renewals/compliance; in customer-managed environments, the organization tracks, renews, and maintains compliance itself.
EXAM TIPS

what to know for the exam

  • Shared responsibility: provider secures the cloud (hardware, physical, infrastructure); customer secures what's in the cloud (data, access, config).
  • Control types: inherited (provider-only), shared (e.g. patching, split), customer-specific (customer-only).
  • The higher up the stack, the more responsibility is the customer's.
  • Key management: the provider secures the key infrastructure (KMS/Key Vault), but the CUSTOMER must generate, manage, and rotate keys.
  • Licenses are often bundled/handled by the provider in cloud subscriptions; customer-managed environments put that burden on the org.
decision rules
  • Scenario asks who secures the physical data center → provider (inherited control)
  • Scenario splits patching between provider and customer → shared control
  • Scenario blames a breach on un-rotated customer keys → customer key-management responsibility
  • Scenario configures security zones for an app → customer-specific control

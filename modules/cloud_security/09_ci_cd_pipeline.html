CI/CD Pipeline

The automated assembly line from code commit to production. Security isn't a final gate, it's baked into every stage: integrate, test, build, deploy, monitor.

overview

A CI/CD pipeline automates building, testing, and deploying code: continuous integration (CI) merges and validates code changes frequently; continuous deployment (CD) automatically releases validated changes. Security is embedded across four stages, integrate, test, build and deploy, monitor, so vulnerabilities are caught early (shift left).

  1. integrate Developers merge code changes frequently into a shared repository. Automated checks, including security scans (SAST) and dependency checks, run on each commit so issues surface immediately.
  2. test Automated tests validate functionality and security: unit/integration tests plus security testing (SAST on source, DAST on running app) to catch vulnerabilities before release.
  3. build and deploy Validated code is packaged and released to environments automatically. Secure the build artifacts and use controlled, repeatable deployments (IaC) so nothing untrusted ships.
  4. monitor After deployment, continuously monitor the running application for performance issues, errors, and security threats, feeding findings back into the next cycle.

security baked into the pipeline

The point of a secure CI/CD pipeline is to shift security left, integrate it early and throughout rather than bolting it on at the end. Each stage carries its own controls:

  • Integrate: automated security scans and dependency/package checks on every commit.
  • Test: SAST (static analysis of source code) and DAST (dynamic testing of the running app).
  • Build and deploy: protect build artifacts, sign/verify, and deploy via controlled IaC.
  • Monitor: runtime monitoring and logging to detect threats in production and inform the next iteration.

Automating these checks means vulnerabilities are caught consistently and early, when they are cheapest to fix, instead of reaching production.

EXAM TIPS

what to know for the exam

  • CI = frequent integration + automated validation of changes; CD = automated deployment of validated changes.
  • Four stages: integrate, test, build and deploy, monitor, each with security controls.
  • Shift left: embed security early and throughout, not as a final gate.
  • SAST = static analysis of source code (early); DAST = dynamic testing of the running app.
  • Dependency/package checks in the pipeline defend against supply-chain risk.
decision rules
  • Scenario scans source code for flaws before it runs → SAST
  • Scenario tests a running application for vulnerabilities → DAST
  • Scenario automatically deploys every validated change → continuous deployment
  • Scenario wants vulnerabilities caught as early as possible → shift left

Cloud Security Considerations

The four ways cloud data goes wrong: storage left open, data accidentally exposed, data leaking out, and deleted data that lingers and gets recovered.

overview

Four key cloud data risks: unsecured storage resources (misconfigured buckets/blobs), data exposure (sensitive data accidentally made accessible), data leakage (unauthorized transmission out of the org), and data remanence (residual data recoverable after deletion).

risk 01

Unsecured Storage

Misconfigured buckets (AWS) / blobs (Azure): default open access, bad CORS/origin settings. Fix container policies, IAM, and object ACLs.

risk 02

Data Exposure

Sensitive data accidentally accessible via misconfig or weak access control. Mitigate with encryption and permission audits.

risk 03

Data Leakage

Unauthorized transmission out of the org (email, personal cloud, USB). Often unnoticed. Mitigate with DLP and policy/training.

risk 04

Data Remanence

Residual data recoverable after deletion (reused cloud hardware). Mitigate with cryptographic erasure or physical destruction.

unsecured storage resources

Cloud storage containers, buckets (AWS) and blobs (Azure), are the most commonly misconfigured resources (you can't nest containers; each holds its own objects). Access is controlled by three things: container policies, IAM authorizations, and object ACLs. Misconfigurations expose data: newly created containers may default to read/write access, and bad origin settings (CDN distribution) or CORS (cross-origin resource sharing) policies can leak data across domains or enable cross-site scripting. Always update defaults and double-check CORS/origin settings.

data exposure

Exposure is sensitive data accidentally made accessible, typically from misconfiguration (a bucket set to public, so anyone with the URL reads it) or weak access controls (a departed employee whose permissions weren't revoked). Mitigate with encryption (at rest and in transit, via KMS/Key Vault and TLS), regular permission audits, and tools like AWS Config or Azure Security Center to catch misconfigurations early.

data leakage

Leakage is the unauthorized transmission of data out of the org, through insecure email, unauthorized cloud apps, or USB drives (an employee emailing unencrypted financial reports externally, or syncing files to a personal Dropbox). It often goes unnoticed without monitoring. Prevent it with DLP tools plus clear policy and employee training on personal devices, cloud services, and email.

data remanence

Remanence is residual data that lingers on storage media after deletion and can be recovered, a risk when retiring cloud instances or when a provider reuses hardware that wasn't fully wiped (potentially exposing your data to the next tenant). Mitigate with secure deletion, cryptographic erasure or physical destruction, and confirm your provider follows industry-standard data destruction.

EXAM TIPS

what to know for the exam

  • Buckets (AWS) and blobs (Azure) are cloud storage containers; misconfigured access (default open, bad CORS/origin) is the top exposure risk.
  • Data exposure = accidental accessibility (misconfig/weak access); data leakage = unauthorized transmission OUT of the org.
  • Data remanence = recoverable leftover data after deletion; fix with cryptographic erasure or physical destruction.
  • Container access = container policies + IAM + object ACLs. CORS controls cross-domain resource sharing.
  • Encryption protects exposed data; DLP and training reduce leakage.
decision rules
  • Scenario has a public bucket exposing customer data → unsecured storage / data exposure
  • Scenario has an employee emailing sensitive files externally → data leakage
  • Scenario recovers deleted data from reused cloud hardware → data remanence
  • Scenario misconfigures cross-domain sharing enabling XSS → CORS misconfiguration

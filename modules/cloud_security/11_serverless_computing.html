Serverless Computing

You still run on servers, you just don't manage them. Code runs as short-lived functions triggered by events, and securing it shifts to the code, its permissions, and its triggers.

overview

In serverless computing the provider manages the underlying servers; you deploy code as functions that run on demand in response to events (FaaS, function as a service). You pay only for execution time. Security focus shifts from infrastructure to workloads (the function code), functions (permissions and triggers), and resources (what each function can access).

concept 01

Workloads

The application logic running serverlessly. Secure the code itself: input validation, no embedded secrets, scanned dependencies.

concept 02

Functions (FaaS)

Short-lived, event-triggered units of code (AWS Lambda, Azure Functions). Apply least-privilege roles and validate every trigger.

concept 03

Resources

What functions touch: storage, databases, APIs. Grant each function only the access it needs, nothing more.

what serverless is

Serverless does not mean there are no servers, it means the cloud provider provisions, scales, and maintains them so developers focus only on code. You deploy individual functions that execute on demand when triggered by an event (an HTTP request, a file upload, a queue message). The provider scales automatically and you pay only for actual execution time. Examples: AWS Lambda, Azure Functions.

securing serverless

Because the provider owns the infrastructure, your responsibility narrows to the function and what surrounds it:

  • Least-privilege permissions: give each function only the IAM permissions it needs; over-permissioned functions widen the blast radius if compromised.
  • Secure the code and dependencies: validate all input, avoid embedded secrets (use a secrets manager), and scan libraries, the same supply-chain concern as anywhere.
  • Protect the triggers: event sources (APIs, queues, storage events) are the attack surface; validate and authenticate them, and watch for injection through event data.
  • Monitor and log: functions are ephemeral, so centralized logging/monitoring is essential to detect abuse and investigate incidents.

An advantage worth noting: short-lived, stateless functions reduce the persistent attack surface, there is no long-running server to compromise, but misconfigured permissions and insecure event triggers remain real risks.

EXAM TIPS

what to know for the exam

  • Serverless = provider manages the servers; you deploy event-triggered functions and pay per execution (FaaS).
  • "Serverless" still uses servers, you just don't manage them. Examples: AWS Lambda, Azure Functions.
  • Security shifts to the code, function permissions, and triggers, apply least privilege to each function's IAM role.
  • Event triggers (APIs, queues, storage events) are the attack surface; validate and authenticate them.
  • Ephemeral functions shrink the persistent attack surface but require centralized logging to monitor.
decision rules
  • Scenario runs code on demand with no servers to manage → serverless / FaaS
  • Scenario gives a Lambda function broad permissions it doesn't need → least-privilege violation
  • Scenario abuses an unauthenticated event trigger → insecure trigger
  • Scenario pays only for actual execution time → serverless billing model

API Security

APIs are the front door to cloud services. Locking that door means controlling who gets in, capping how fast they can knock, and watching everything that passes through.

overview

APIs let applications communicate and exchange data, so they are a prime attack target. Three pillars secure them: authorization (verify identity and permissions), rate limiting (cap request volume to prevent abuse/DoS), and logging and monitoring (record and watch API activity to detect threats).

pillar 01

Authorization

Verify identity and grant only permitted actions. OAuth (delegated access tokens) and JWT (signed identity claims).

pillar 02

Rate Limiting

Cap how many requests a client can make in a window. Blocks brute force, scraping, and DoS abuse.

pillar 03

Logging & Monitoring

Record who called what and when; watch for anomalies (spikes, unusual sources) to detect attacks early.

authorization

Authorization ensures only authenticated, permitted users and applications access the API, the first defense against unauthorized access. Two common mechanisms:

OAuth
A delegated-access framework: issues access tokens so a third-party app can act on a user's behalf without sharing the password. Used widely for "sign in with" flows and granting scoped, revocable access to resources.
JWT (JSON Web Token)
A compact, signed token carrying identity claims, verified on each request. The signature lets the server trust the token's contents without a database lookup; commonly paired with OAuth.

rate limiting

Rate limiting caps how many requests a client can make in a given time window, protecting the API from abuse and overload. It blocks brute-force attempts, automated scraping, and denial-of-service attacks that try to overwhelm the service. Limits can be per user, per API key, or per IP; exceeding them returns a throttling response (HTTP 429) rather than processing the request.

logging and monitoring

Logging records API activity (who called which endpoint, when, and with what result); monitoring watches those logs in real time for suspicious patterns, a sudden spike in requests, calls from unusual locations, or repeated authorization failures. Together they enable fast detection and response, support forensic investigation after an incident, and help meet compliance and audit requirements.

EXAM TIPS

what to know for the exam

  • API security rests on three pillars: authorization, rate limiting, and logging/monitoring.
  • OAuth = delegated access via tokens (act on a user's behalf, no password sharing); JWT = compact signed token carrying identity claims.
  • Rate limiting caps request volume to stop brute force, scraping, and DoS; over-limit returns HTTP 429.
  • Logging records API activity; monitoring watches it in real time to detect anomalies and support audits.
decision rules
  • Scenario grants a third-party app scoped access without sharing the password → OAuth
  • Scenario passes signed identity claims verified on each request → JWT
  • Scenario throttles a client flooding the API → rate limiting
  • Scenario detects an unusual spike in API calls → logging and monitoring

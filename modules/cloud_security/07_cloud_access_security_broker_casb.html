Cloud Access Security Broker (CASB)

A security checkpoint sitting between your users and the cloud, enforcing policy on everything that passes. It works one of two ways: through the provider's API, or inline as a proxy.

overview

A CASB is a security enforcement point placed between cloud users and cloud providers, applying the organization's security policies to cloud traffic (visibility, compliance, data security, threat protection). It operates in two architectures: API-based (integrates with the provider's API, out of band) and proxy-based (sits inline), with proxy split into forward and reverse.

model 01

API-Based

Integrates directly with the cloud provider's API. Out of band, no traffic interception; good visibility and data inspection, not real-time inline.

model 02

Forward Proxy

Sits in front of users, inline on outbound traffic. Enforces policy in real time, usually needs an agent/config on the endpoint.

model 03

Reverse Proxy

Sits in front of the cloud service, inline on inbound traffic. Agentless, ideal for unmanaged/BYOD devices.

what a CASB does

A CASB gives the organization control and visibility over cloud usage by enforcing policy between users and providers. Core functions:

  • Visibility: discover cloud apps in use (including shadow IT) and who is using them.
  • Compliance: ensure cloud usage meets regulations (HIPAA, GDPR, PCI DSS).
  • Data security: enforce DLP, encryption, and access controls on cloud data.
  • Threat protection: detect and block malware, compromised accounts, and anomalous behavior.

api-based deployment

An API-based CASB integrates directly with the cloud provider's API rather than intercepting traffic, so it works out of band. It can inspect data already stored in the cloud, apply DLP, and scan for threats without sitting in the traffic path, so there is no latency impact, but it is not real-time and only works with providers that expose suitable APIs.

proxy-based deployment

Forward proxy
Sits between users and the cloud, inline on outbound traffic, enforcing policy in real time before requests reach the provider. Typically requires an agent or network configuration on the user's device, so it suits managed endpoints.
Reverse proxy
Sits in front of the cloud service, inline on inbound traffic to the provider. Agentless, so it works well for unmanaged or BYOD devices where you can't install software.
EXAM TIPS

what to know for the exam

  • CASB = policy enforcement point between cloud users and providers: visibility, compliance, data security, threat protection.
  • Two architectures: API-based (out of band, integrates with provider API) and proxy-based (inline).
  • Forward proxy = inline in front of users, real-time, usually needs an agent (managed devices).
  • Reverse proxy = inline in front of the cloud service, agentless (great for unmanaged/BYOD).
  • CASBs are a key tool for discovering and controlling shadow IT.
decision rules
  • Scenario inspects data at rest in the cloud without touching traffic → API-based CASB
  • Scenario enforces policy in real time on managed-device traffic → forward proxy
  • Scenario protects a cloud app for BYOD/unmanaged devices with no agent → reverse proxy
  • Scenario needs visibility into shadow IT and cloud compliance → CASB

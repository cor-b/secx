Container Management

Containers package an app with everything it needs to run anywhere. Securing them means trusting the image, isolating the runtime, and locking down the orchestrator.

overview

Containers bundle an application with its dependencies into a portable, isolated unit that runs consistently across environments. Securing them spans two areas: container security (trusted images, runtime isolation, least privilege) and orchestration (managing containers at scale with tools like Kubernetes, secured with RBAC, network policies, and secrets management).

area 01

Image Security

Use trusted base images, scan for vulnerabilities, keep them minimal and patched. A bad image ships its flaws everywhere it runs.

area 02

Runtime Isolation

Containers share the host kernel, so enforce isolation and least privilege: no unnecessary root, limited capabilities, resource limits.

area 03

Orchestration

Kubernetes manages containers at scale; secure it with RBAC, network policies, and proper secrets management.

container security

A container packages an app with its libraries, binaries, and config so it runs the same anywhere; it is more lightweight than a VM because containers share the host OS kernel instead of each carrying a full guest OS. That shared kernel is also the key security consideration: weak isolation can let a compromised container affect the host or its neighbors. Core practices:

  • Trusted images: pull from trusted registries, use minimal base images, and scan for known vulnerabilities before deployment.
  • Least privilege: don't run containers as root, drop unneeded Linux capabilities, and make filesystems read-only where possible.
  • Isolation and limits: enforce resource limits and segmentation so one container can't starve or reach others.

orchestration

At scale, organizations run many containers across many hosts, which is what an orchestrator manages, automating deployment, scaling, networking, and self-healing. Kubernetes is the dominant platform. Securing the orchestrator is essential because it controls the whole cluster:

RBAC
Role-based access control restricts who can perform which actions in the cluster, enforcing least privilege for users and service accounts.
Network policies
Control which pods/services can talk to each other, segmenting traffic so a compromise can't move freely.
Secrets management
Store credentials, keys, and tokens securely (encrypted, access-controlled) rather than hard-coding them into images or config.
EXAM TIPS

what to know for the exam

  • Containers share the host OS kernel (lighter than VMs, which each run a full guest OS); that shared kernel is the main isolation concern.
  • Container security = trusted/scanned minimal images + least privilege (no root) + runtime isolation and resource limits.
  • Kubernetes is the standard orchestrator; secure it with RBAC, network policies, and secrets management.
  • Never hard-code secrets into images, use proper secrets management.
decision rules
  • Scenario restricts who can act within a Kubernetes cluster → RBAC
  • Scenario limits which pods can communicate → network policies
  • Scenario flags a container running as root → least privilege violation
  • Scenario scans a base image before deployment → image security

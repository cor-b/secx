Modeling with an Existing System

Threat modeling what you already run: map the architecture, identify the threats relevant to it, and choose controls, with extra layers for legacy systems that can't be patched.

overview

Modeling with an existing system evaluates threats and vulnerabilities specific to the current environment to determine what can be exploited and which controls are needed. The process: understand the current architecture/operations, identify relevant threats, then select mitigating controls (MFA, encryption, IDS/IPS). Legacy systems often need extra controls.

the process

  1. Map the architecture and operations: understand how the current system works (software, hardware config, interactions). A web platform has a different risk profile than internal apps.
  2. Identify relevant threats: a database-heavy system worries about SQL injection; an external-facing e-commerce site worries about XSS and DDoS.
  3. Select mitigating controls: match controls to the threats.

mitigating controls

Multi-factor authentication (MFA)
Protects against unauthorized access, especially for remote access.
Encryption
Protects sensitive data at rest and in transit; even if stolen, it's unusable without the decryption keys.
Intrusion detection/prevention (IDS/IPS)
IDS monitors and alerts; IPS goes further, actively blocking malicious traffic in real time before it reaches the system.

legacy systems

Legacy systems often lack modern security features (unpatched vulnerabilities, weak authentication), so they need extra controls: deploy MFA to strengthen access, use network segmentation to mitigate the lack of available patches, and apply encryption for data at rest and in transit, layered defense around outdated systems. Regular assessments of architecture and operations keep controls current with emerging threats.

EXAM TIPS

what to know for the exam

  • Process: understand the existing architecture → identify relevant threats → select controls.
  • Match threats to the system: database-heavy → SQL injection; external-facing → XSS/DDoS.
  • Controls: MFA (access), encryption (data), IDS (alert) vs IPS (actively block).
  • Legacy systems need extra layers, especially network segmentation, when patches aren't available.
  • Regular reassessment keeps controls effective against emerging threats.
decision rules
  • Scenario models threats to a system already in production → modeling with an existing system
  • Scenario can't patch a legacy system → network segmentation + compensating controls
  • Scenario actively blocks malicious traffic (not just alerts) → IPS
  • Scenario protects a database-heavy app → defend against SQL injection

Technical Attack Surface

The entry points in your technology itself. Reviewing the design, tracing how data moves, guarding the boundaries between trust zones, and examining the source code.

overview

The technical attack surface is the sum of all potential entry points in an organization's technology infrastructure. Four concepts: architecture reviews (evaluate design for weaknesses), data flows (how information moves), trust boundaries (where data crosses between trust levels), and code reviews (examine source code for vulnerabilities).

concept 01

Architecture Reviews

Evaluate system design for weaknesses/misconfigs early. Tools: Microsoft Threat Modeling Tool, OWASP Threat Dragon.

concept 02

Data Flows

Map how data moves; ensure sensitive data isn't sent in plain text or over insecure channels. Tools: Visio, Lucidchart.

concept 03

Trust Boundaries

Points where data crosses between trust zones (internet ↔ internal). Secure with firewalls and segmentation.

concept 04

Code Reviews

Examine source for vulnerabilities: static, dynamic, side-channel, reverse engineering, SCA, fuzzing.

architecture reviews, data flows, and trust boundaries

Architecture reviews evaluate system design for weaknesses or misconfigurations, ideally early, catching issues like centrally stored sensitive data accessible from multiple networks or insecure third-party dependencies (tools: Microsoft Threat Modeling Tool, OWASP Threat Dragon). Data flow analysis maps how data moves to find exposure (e.g. sensitive data sent in plain text), prompting encryption; it also verifies access controls (tools: Visio, Lucidchart). Trust boundaries are points where data crosses between zones of different trust (internet ↔ internal, web server ↔ database); secure them with firewalls and network segmentation.

code reviews

Code reviews find vulnerabilities in source code before deployment using several techniques:

Static analysis
Inspects code without executing it, manually or with tools (SonarQube, Checkmarx). Use both.
Dynamic analysis
Tests code while executing in a controlled environment/sandbox (Burp Suite as a proxy). Side-channel analysis (a subset) observes indirect outputs like network traffic (Wireshark).
Reverse engineering
Pulls apart compiled/obfuscated code (decompiler IDA Pro → assembly; debugger OllyDbg → step-by-step).
Software composition analysis (SCA)
Identifies vulnerable third-party/open-source dependencies (Snyk, WhiteSource).
Fuzz testing (fuzzing)
Injects randomized data to find input-handling weaknesses (Peach Fuzzer, WSFuzzer).
EXAM TIPS

what to know for the exam

  • Four technical concepts: architecture reviews, data flows, trust boundaries, code reviews.
  • Trust boundaries = where data crosses between different trust levels; protect with firewalls/segmentation.
  • Static analysis = code not executed; dynamic analysis = code executed in a sandbox; side-channel observes indirect outputs.
  • SCA finds vulnerable third-party libraries; fuzzing injects random data to find input flaws.
  • Reverse engineering uses a decompiler (IDA Pro) and debugger (OllyDbg) for obfuscated code/malware.
decision rules
  • Scenario examines code without running it → static analysis
  • Scenario runs code in a sandbox to watch behavior → dynamic analysis
  • Scenario flags a vulnerable open-source library → software composition analysis
  • Scenario secures where data crosses from internet to internal → trust boundary

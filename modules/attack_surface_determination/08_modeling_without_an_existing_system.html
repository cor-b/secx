Modeling without an Existing System

Threat modeling something that doesn't exist yet: envision the design, anticipate the risks, and build security in from the ground up rather than retrofitting it later.

overview

Modeling without an existing system assesses threats for systems not yet developed or deployed, to identify risks and build in controls preemptively. It requires envisioning how the system will function, what data it handles, how users interact, and its integration points, then selecting controls (encryption, access management, network segmentation) before production. The advantage: building security in from the ground up rather than retrofitting.

what to anticipate

  1. Architecture: cloud, on-premise, or hybrid? Will it interact with third parties? A cloud-based system raises concerns like unauthorized cloud access or insecure APIs.
  2. Data types: what data will it process/store? Sensitive data (credit cards, medical records) needs prioritized protection, encryption at rest and in transit.
  3. User interactions: who needs access (employees, customers, third parties) and at what levels? Multiple user types call for role-based access control.
  4. Integration points: integrations expand the attack surface. Third-party API integration risks API abuse; plan strong API security (authentication, rate limits, monitoring).

selecting controls before launch

With risks identified, select controls and embed them in the design:

  • MFA: require two or more verification methods to block unauthorized access.
  • Encryption: protect sensitive data at rest and in transit, unreadable without keys.
  • Network segmentation: divide the network into isolated segments so a compromise in one part can't spread, planned early so it's built in rather than added later.

This proactive approach (e.g. modeling a new cloud-based CRM by analyzing common cloud threats, then adding MFA, encryption, and strict API access controls) prevents issues and ensures the system is secure at launch.

EXAM TIPS

what to know for the exam

  • Modeling without an existing system = anticipate risks before development/deployment and build security in from the start.
  • Anticipate: architecture (cloud/on-prem/hybrid), data types, user interactions, integration points.
  • Integrations (especially third-party APIs) expand the attack surface, risk of API abuse.
  • Controls to embed early: MFA, encryption (at rest + in transit), RBAC, network segmentation.
  • Building security in from the ground up beats retrofitting it after launch.
decision rules
  • Scenario models threats for a system not yet built → modeling without an existing system
  • Scenario plans API authentication and rate limits pre-launch → integration-point mitigation
  • Scenario designs network segmentation into a new system → build security in from the start
  • Scenario assigns access by role for a planned system → RBAC
